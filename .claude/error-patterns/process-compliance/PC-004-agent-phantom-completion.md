# PC-004: 代理人幻影完成（Agent Phantom Completion）

## 錯誤分類

| 項目 | 值 |
|------|---|
| 編號 | PC-004 |
| 類別 | process-compliance |
| 嚴重度 | 高 |
| 發現版本 | 0.1.0 |
| 發現日期 | 2026-03-05 |
| 相關 Ticket | 0.1.0-W1-001.1 |

## 症狀

- 背景代理人（run_in_background）回報「修改已成功完成」
- 但目標檔案實際未被修改（git diff 無變更、grep 找不到新增內容）
- PM 若信任代理人報告而不驗證，會導致缺陷流入後續階段

## 根因分析

代理人在背景執行時，可能因以下原因導致「報告完成但實際未修改」：

1. **Edit 工具被 Hook 靜默拒絕**：保護分支 Hook 或其他 PreToolUse Hook 阻止了編輯，但代理人未正確處理拒絕回應
2. **代理人認知偏差**：代理人在規劃修改內容時「認為」已完成，但實際的 Edit/Write 工具呼叫未成功執行
3. **工具呼叫被跳過**：代理人在思考過程中生成了修改計劃，但最終未實際呼叫 Edit/Write 工具

## 行為模式

這是一種「過度自信」的代理人行為模式：
- 代理人讀取了檔案、分析了問題、規劃了修改方案
- 在回報時使用了「已完成」的語氣
- 但中間的實際修改步驟被跳過或失敗

## 解決方案

### 立即修復
- 重新派發代理人（非背景模式或加強指令要求驗證）

### 防護措施

**PM 端強制驗證（並行派發後驗證）**：

```
所有並行代理人回報完成後：
1. [強制] git diff --stat 確認實際變更
2. 比對代理人報告 vs 實際變更檔案
3. 缺失 → 重新派發
```

**代理人 prompt 強化**：
在派發 prompt 中加入：
```
完成後用 Read 驗證修改成功。
```
或：
```
請先用 Read 工具讀取檔案，找到準確的插入位置，然後用 Edit 工具修改。完成後用 Read 驗證。
```

## 預防措施

| 措施 | 說明 |
|------|------|
| git diff --stat 強制驗證 | 並行派發後的標準檢查（已在 parallel-dispatch.md 定義） |
| 代理人 prompt 要求 Read 驗證 | 在 prompt 結尾加上「完成後用 Read 驗證修改成功」 |
| 不信任代理人的文字報告 | 以 git diff 為 Source of Truth |

## 本次案例

- **場景**：並行派發 3 個 saffron-system-analyst 代理人修改 UC 文件
- **問題代理人**：uc007-fix（修改 UC-007-jsonl-parsing.md）
- **報告內容**：「修改已經成功完成」+ 詳細的修改摘要
- **實際狀態**：檔案未被修改（git diff 無變更）
- **偵測方式**：PM 執行 `git diff --stat` 並行派發後驗證
- **修復**：重新派發代理人，prompt 中加入「完成後用 Read 驗證」

## 教訓

> 代理人的文字報告是「意圖」，git diff 是「事實」。永遠以事實為準。
