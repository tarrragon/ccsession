# Phase 2 測試設計：ID 解析包容性擴充和命名字典

## 1. 測試策略規劃

### 1.1 測試分層決策

根據程式碼層級和複雜度選擇測試策略：

| 層級 | 模組 | 複雜度 | 測試策略 | 理由 |
|------|------|--------|--------|------|
| **Layer 5 (Domain)** | `id_parser.py` 正則邏輯 | 高 | 單元測試 | 複雜的邊界條件判斷，需隔離測試 |
| **Layer 5 (Domain)** | `extract_core_ticket_id()` | 高 | 單元測試 | 字串解析邏輯，需完整邊界覆蓋 |
| **Layer 5 (Domain)** | `has_description_suffix()` | 中 | 單元測試 | 布林判斷邏輯 |
| **Layer 2 (Behavior)** | `ticket_loader.py` 去重邏輯 | 高 | 單元測試 | 集合操作和邏輯判斷 |
| **Layer 3 (UseCase)** | `ticket_loader.list_tickets()` 整合 | 高 | 整合測試 | 多檔案場景、去重驗證 |
| **Layer 1 (Hook)** | `ticket-id-validator-hook.py` | 中 | BDD 行為測試 | 提示輸出驗證，需行為確認 |
| **迴歸測試** | 22 個現存非標準檔案 | 中 | 整合測試 | 確保向後相容 |

### 1.2 Sociable Unit Tests 原則應用

**Core Principle**: 測試**行為**而非實作，測試應依賴**真實 Domain Objects**，只隔離**外部依賴**。

| 測試類型 | Mock 策略 | 真實物件 | 理由 |
|---------|---------|--------|------|
| `extract_core_ticket_id()` | 無 Mock | 字串（值物件）| 純邏輯函式，無外部依賴 |
| `TICKET_ID_PATTERN` 驗證 | 無 Mock | 正則物件 | 測試核心行為：正則匹配邏輯 |
| `ticket_loader.list_tickets()` | Mock `Path`、`load_ticket()` | 真實檔案結構 | 隔離檔案系統，但使用真實 Ticket dict |
| Hook 驗證 | Mock `sys.stderr` | 正則邏輯 | 只隔離輸出流，驗證邏輯執行 |

### 1.3 BDD 行為鏈式設計

測試應基於行為事件逐步推演（**前置條件 → 動作 → 結果**），先驗證前置條件再測試行為。

**範例**（場景 3：去重邏輯）:
```
前置條件 A: 版本目錄存在 "0.1.0-W11-004.md"
前置條件 B: 同一目錄存在 "0.1.0-W11-004-phase1-design.md"
  ↓
動作: list_tickets() 掃描目錄
  ↓
行為驗證 1: 兩個檔案的 extract_core_ticket_id() 都返回 "0.1.0-W11-004"
  ↓
行為驗證 2: loaded_core_ids 集合中只包含一次該 ID
  ↓
結果: 只載入一個 Ticket（去重成功）
```

        "---\nid: 0.1.0-W11-004\ntitle: Test Design\n---\nContent"
