---

## Problem Analysis

W27-001 完成了 WAVE_COMPLETION_REMINDER 常數的定義（ask_user_question_reminders.py L204-239），但 commit-handoff-hook.py 未使用此常數實作自動偵測邏輯。

目前 Hook 的行為：
- commit 成功後，根據 commit type 判斷是否跳過 #16
- 輸出 COMMIT_HANDOFF_REMINDER 或 COMMIT_HANDOFF_SKIP16_REMINDER
- **缺少**情境 C 的自動判斷：當同 Wave 無 pending ticket 時，應附加 WAVE_COMPLETION_REMINDER

需要實作的邏輯：
1. 讀取 docs/todolist.yaml 的 current_version
2. 掃描該版本的 ticket 檔案
3. 解析 frontmatter 取得 wave 和 status
4. 找到 in_progress ticket 確定當前 Wave
5. 統計同 wave 的 pending ticket 數量
6. 若 pending == 0，判斷為情境 C

---

## Solution

新增四個函式到 commit-handoff-hook.py：

### 1. `read_current_version(project_dir, logger) -> Optional[str]`
從 docs/todolist.yaml 讀取 current_version 欄位，使用正則表達式解析 YAML。

### 2. `scan_wave_tickets(project_dir, version, logger) -> List[Dict]`
掃描 `docs/work-logs/v{version}/tickets/` 目錄中的所有 .md 檔案，逐一解析 frontmatter，提取 wave 和 status 欄位。回傳清單 [{wave, status}]。

### 3. `detect_wave_completion(logger) -> bool`
主邏輯函式（5 步驟）：
- Step 1: 讀取 current_version
- Step 2: 掃描 ticket 目錄
- Step 3: 找到 in_progress ticket 的 wave
- Step 4: 統計同 wave 的 pending 數量
- Step 5: 若 pending == 0，回傳 True；否則 False

安全降級：任何錯誤（檔案讀取失敗、無 in_progress ticket 等）都回傳 False，不阻擋 commit 流程。

### 4. 修改 `main()` 函式
在決定 reminder 後，呼叫 `detect_wave_completion(logger)`。若為 True，將 WAVE_COMPLETION_REMINDER 附加到 reminder：
```python
if is_wave_completion:
    reminder += "\n" + AskUserQuestionReminders.WAVE_COMPLETION_REMINDER
```

### 設計要點
- **情境附加而非替代**：情境 C 是 #16 確認後的額外流程，故附加而非替代
- **安全降級**：無法讀取 current_version/ticket 時，安全回傳 False
- **獨立邏輯**：不 import 其他 Hook 的函式，確保獨立性
- **型別標註**：所有函式完整的型別標註（Optional, Dict, List 等）
