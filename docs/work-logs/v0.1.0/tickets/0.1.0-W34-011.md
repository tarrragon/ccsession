## Problem Analysis

### 根本原因分析

**靜默跳過的機制**：
- 原始代碼在 `_parse_yaml_lines` 第 175 行（修改前）使用條件：
  ```python
  if line.startswith('  ') and not line.startswith('    '):
  ```
  這個條件**明確排除**了 4 格縮排行，導致任何 4 格以上的行都被跳過。

**實際 YAML 格式** (根據 0.1.0-W26-002.md 分析)：
- 使用 2 格縮排的列表項目，格式為 `  - item`
- 例如：
  ```yaml
  where:
    layer: hooks
    files:
    - .claude/hooks/commit-handoff-hook.py
    - .claude/hooks/parallel-suggestion-hook.py
  ```
- 這些 2 格縮排的列表項目也被完全靜默跳過（因為 `_parse_nested_line` 無法處理以 `-` 開頭的行）

### 影響範圍

搜尋實際受影響的 Ticket 檔案：
```bash
grep -rn "^  - " docs/work-logs/v0.1.0/tickets/ | wc -l
# 結果：多個 Ticket 包含 where.files 列表
```

確認的位置（已驗證）：
- `/Users/mac-eric/project/ccsession/docs/work-logs/v0.1.0/tickets/0.1.0-W26-002.md`：files 欄位包含 3 個列表項目
- 其他多個 Ticket 檔案也使用此格式

---

## Solution

### 修復方法

修復流程分為 3 個關鍵修改：

#### 1. 擴展 `_parse_yaml_lines` 支援列表項目

**修改前**（行 175）：
```python
if line.startswith('  ') and not line.startswith('    '):
```

**修改後**：
```python
if line.startswith('  '):  # 接受所有 2+ 格縮排行
```

同時新增狀態變數 `current_nested_key` 追蹤嵌套結構中的子鍵（如 `files`），以便將列表項目正確累積到該子鍵。

#### 2. 擴展 `_parse_nested_line` 處理列表項目

新增列表項目檢測邏輯：
```python
if nested_line.startswith('- ') and current_nested_key and current_key:
    # 列表項目內容
    item_content = nested_line[2:]  # 去除 "- " 前綴
    return _NestedLineResult(
        multiline_marker=None,
        update_action=(current_nested_key, item_content, False)
    )
```

#### 3. 增強 `_parse_yaml_lines` 的狀態管理

新增邏輯識別嵌套字典更新，並記錄最後更新的嵌套鍵：
```python
if is_nested_dict:
    # ...更新嵌套字典...
    if value:
        current_nested_key = list(value.keys())[0]  # 記錄最後的嵌套鍵
```

### 修復驗證

- 新增 3 個測試案例：
  1. `test_two_space_indented_list_items`：驗證 2 格縮排列表項目
  2. `test_four_space_indented_list_items`：驗證 4 格縮排列表項目
  3. `test_mixed_nested_structure_with_lists`：驗證混合嵌套結構

- 所有既有 15 個測試仍通過
- 實際 Ticket 檔案（0.1.0-W26-002.md）解析成功：`where.files` 正確解析為 3 個檔案
