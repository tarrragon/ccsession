## Solution

### Phase 1 功能規格設計

#### 1. 需求分析

**核心問題**：`command-entrance-gate-hook.py` 有 840 行程式碼，11 個可測試函式，現有測試只有 3 個實際案例（其餘 6 個為 `assert True` 佔位符），導致高風險分支路徑無保障。

**功能核心**：Hook 扮演開發命令「閘門」角色，接收用戶 prompt 後判斷是否為開發命令，若是則驗證 Ticket 是否存在、已認領、含決策樹欄位，否則阻止執行。

**關鍵使用者場景**：
- 場景 1：用戶輸入開發命令 + 有效 Ticket → 允許執行
- 場景 2：用戶輸入開發命令 + 無 Ticket → 阻止並提示建立
- 場景 3：用戶輸入開發命令 + Ticket 未認領 → 阻止並提示認領
- 場景 4：用戶輸入管理命令（/ticket、分析等）→ 不驗證，直接放行
- 場景 5：系統內部訊息（task-notification 等）→ 跳過驗證
- 場景 6：開發命令 + Ticket 有效但內容不相關 → 允許但輸出警告

| 工具 | 用途 | 選擇原因 |
|------|------|---------|
| pytest | 單元測試框架 | Phase 3b 執行，標準 Python 測試工具 |
| unittest.mock | Mock 依賴 | 隔離檔案系統和外部依賴 |
| pytest fixture | 測試數據管理 | 為多個測試提供共用的臨時檔案和 logger |
| importlib.util | 動態載入 | 加載包含 uv metadata 的 Hook 檔案 |

#### 1.2 Mock 策略（Sociable Unit Tests 原則應用）

**Mock 對象清單**：

| Mock 對象 | Mock 原因 | Mock 方式 |
|---------|---------|---------|
| `get_latest_pending_ticket` | 外部依賴（檔案系統掃描） | patch 回傳值 |
| `find_ticket_files` | 外部依賴（目錄掃描） | patch 回傳值 |
| `validate_ticket_has_decision_tree` | 外部依賴（字串搜尋） | patch 回傳值 |
| `sys.stdin` | 外部依賴（標準輸入） | io.StringIO 注入 |
| `sys.stdout`/`sys.stderr` | 驗證輸出 | pytest capsys fixture |
| `logger` | 驗證日誌行為 | MagicMock 對象 |

| 對象 | 原因 |

| `is_management_operation` | 純邏輯函式（字串匹配） |
| `is_development_command` | 純邏輯函式（正則和關鍵字匹配） |
| `split_kebab_case` | 純邏輯函式 |
| `tokenize_text` | 純邏輯函式 |

### 6. 權宜方案與技術債務

#### 6.1 最小可行實作

- 所有 62 個測試案例透過 pytest 框架執行
- Mock 使用 unittest.mock.patch 和 pytest fixtures
- 檔案操作使用 tmp_path fixture（標準做法，無技術債）
- 測試覆蓋率目標：>= 95%（行覆蓋率），>= 90%（分支覆蓋率）

#### 6.2 已知限制

| 限制 | 說明 |
|------|------|
| importlib 版本相依 | Python 3.11+ 才有 spec_from_file_location，但 Hook 檔案已要求 >= 3.11 |
| uv script metadata 解析 | Hook 檔案開頭的 `# /// script` 區塊會被 importlib 自動忽略，不影響測試 |
| stdin mock | 測試 TestMain 時需要用 monkeypatch，标準 io.StringIO 可能在某些情況下被 Hook 的 setup_hook_logging 干擾 |
| logger mock 驗證 | mock_logger.debug / warning / error 的呼叫驗證依賴 Hook 內部的日誌邏輯，如果 Hook 改變日誌層級可能需要調整 |

#### 6.3 技術債務標記（Phase 4 建議）

| 债務 | 優先級 | 建議 |
|-----|--------|------|
| conftest.py 的 hook_module fixture | 低 | Phase 4 可考慮提取為共享 fixture（如果其他 Hook 測試也需要）到 `.claude/hooks/tests/conftest.py` 的通用部分 |
| mock logger 的重複 | 低 | 多個測試類都用 mock_logger fixture，Phase 4 可評估是否提取為 autouse fixture |
| patch 路徑的脆弱性 | 中 | 如果 Hook 重構改變函式名或位置，patch 路徑需要調整。建議在 Phase 4 添加 deprecation warning |
| 缺少集成測試 | 低 | 現有測試都是 mocked 單元測試，Phase 4 可考慮添加真實的集成測試（不 mock check_ticket_status）來驗證完整流程 |

### 7. 語言特定實作注意事項

**Python 考量**：
- 使用 `pytest.mark.parametrize` 減少重複程式碼（見 TestIsManagementOperation）
- Mock 使用 `from unittest.mock import patch, MagicMock`（標準庫，無額外依賴）
- 使用 `tmp_path` fixture 進行文件操作測試（pytest 內置）
- 字串比對使用 `in` operator（O(n)），大規模數據可優化為正則預編譯，但此 Hook 規模不需要
- logger 使用 MagicMock 驗證呼叫，避免影響實際日誌輸出

**效能最佳化建議**：
- Hook 呼叫頻繁（每個用戶命令），但驗證邏輯簡單，無需優化
- find_ticket_files 在 check_ticket_status 中被呼叫，Phase 3b 可考慮快取策略（但不在本 Ticket 範圍內）

**可能的技術挑戰**：
1. **importlib 載入失敗**：如果 Hook 檔案有語法錯誤或依賴缺失，動態載入會拋出例外。Phase 3b 需要確保 Hook 檔案的正確性
2. **Mock 隔離**：如果 check_ticket_status 內部呼叫了其他未被 patch 的函式（如 logger），可能出現預期外的行為。Phase 3b 需要小心驗證 patch 的完整性
3. **stdin 干擾**：setup_hook_logging 可能會改變 sys.stdin，需要確保 monkeypatch 應用順序
