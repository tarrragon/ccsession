# Phase 2 Test Design Report: Onboard Checker 新增檢查項目

## 測試策略概述

### 分層決策樹應用

根據 @.claude/methodologies/hybrid-testing-strategy-methodology.md 的分層策略：

- **Layer 5 (Domain Logic - 複雜邏輯)**: onboard_checker.py 中的 6 個新檢查函式
  - 決策：**單元測試** (Sociable Unit Tests)
  - 原因：獨立檢查邏輯，無複雜依賴，應直接測試行為而非實作細節

### Sociable Unit Tests 原則

**核心應用**：
- **Unit = Module**: 每個檢查函式為獨立的測試單元
- **Isolation 策略**: 只隔離外部世界（檔案系統），使用真實的 Path 物件和檔案操作
- **Mock 策略**: 禁止 Mock 檔案系統操作（Path.exists 等），使用 tmp_path fixture 建立真實文件系統
- **測試目標**: Module API 行為（檢查成功/失敗時返回的結果結構）

## 新增檢查項目測試設計

### 1. .gitignore 完整性檢查 (check_gitignore_completeness)

#### 功能說明
驗證 .gitignore 包含所有必須排除的框架檔案：
- coverage/, htmlcov/（覆蓋率報告）
- .claude/hook-logs/（Hook 執行日誌）
- .claude/worktrees/（Git worktree 暫存）
- .claude/tool-results/（工具輸出）
- .claude/handoff/（交接檔案暫存）

#### 資料結構設計

```python
@dataclass
class GitignoreCheckInfo:
    exists: bool                    # .gitignore 檔案存在
    has_coverage_rules: bool        # 包含 coverage/ 規則
    has_hook_logs_rule: bool        # 包含 .claude/hook-logs/ 規則
    has_worktrees_rule: bool        # 包含 .claude/worktrees/ 規則
    has_tool_results_rule: bool     # 包含 .claude/tool-results/ 規則
    has_handoff_rule: bool          # 包含 .claude/handoff/ 規則
    all_required_complete: bool     # 所有必須規則都存在
    missing_rules: list[str]        # 缺失的規則清單
```

#### 測試案例設計

**GWT-1.1: 正常路徑 - 所有規則都存在**
```
Given: 專案根目錄存在 .gitignore，包含所有必須的排除規則
When: 執行 check_gitignore_completeness
Then: 返回 all_required_complete=True，missing_rules=[]
```

**GWT-1.2: 正常路徑 - 規則存在但格式變體**
```
Given: .gitignore 包含規則但變體如 /coverage/, coverage, coverage/*, etc.
When: 執行 check_gitignore_completeness（支援模糊匹配）
Then: 返回 all_required_complete=True（模糊匹配成功）
```

**GWT-1.3: 缺失路徑 - 部分規則缺失**
```
Given: .gitignore 存在但缺少 coverage/ 和 .claude/hook-logs/ 規則
When: 執行 check_gitignore_completeness
Then: 返回 all_required_complete=False，missing_rules=['coverage/', '.claude/hook-logs/']
```

**GWT-1.4: 缺失路徑 - 完全缺失所有規則**
```
Given: .gitignore 存在但內容為空或不含任何必須規則
When: 執行 check_gitignore_completeness
Then: 返回 all_required_complete=False，missing_rules 包含所有 5 項規則
```

**GWT-1.5: 缺失路徑 - .gitignore 檔案不存在**
```
Given: 專案根目錄不存在 .gitignore 檔案
When: 執行 check_gitignore_completeness
Then: 返回 exists=False，all_required_complete=False
```

**GWT-1.6: 邊界情況 - .gitignore 檔案為空**
```
Given: 專案根目錄存在空的 .gitignore 檔案
When: 執行 check_gitignore_completeness
Then: 返回 exists=True，all_required_complete=False，missing_rules 包含全部 5 項
```

**GWT-1.7: 邊界情況 - 檔案編碼錯誤**
```
Given: .gitignore 檔案存在但含有無效編碼
When: 執行 check_gitignore_completeness
Then: 優雅地處理，返回 exists=True，all_required_complete=False
```

## 測試環境設定規劃

### Mock 策略（遵循 Sociable Unit Tests）

| 項目 | 策略 | 原因 |
|------|------|------|
| 檔案系統操作 | 不 Mock，使用 tmp_path fixture | 直接測試行為，檔案系統是 module 的行為邊界 |
| Path 物件 | 直接使用 pathlib.Path | 內部物件，不需 Mock |
| 字串解析（YAML/JSON） | 不 Mock，使用真實檔案 | 測試實際的解析邏輯 |
| 編碼相關 | 使用 tmp_path 建立各種編碼檔案 | 驗證編碼處理的正確性 |

### 測試資料準備

#### Fixture 需求

| Fixture 名稱 | 用途 | 備註 |
|------------|------|------|
| tmp_project_path | 建立臨時專案結構 | 預建立必要目錄和檔案 |
| tmp_gitignore | 建立 .gitignore 變體 | 測試規則完整性 |
| tmp_claude_structure | 建立 .claude 完整結構 | 測試目錄檢查 |
| invalid_json | 無效 JSON 檔案 | 測試格式驗證 |
| invalid_yaml | 無效 YAML 檔案 | 測試格式驗證 |
| various_encodings | 多種編碼檔案 | 測試編碼處理 |

#### 測試資料集合

```python
# .gitignore 規則集
REQUIRED_GITIGNORE_RULES = [
    'coverage/',
    'htmlcov/',
    '.claude/hook-logs/',
    '.claude/worktrees/',
    '.claude/tool-results/',
    '.claude/handoff/',

# .claude 必須目錄

    'rules',

    'hook-language-classification.yaml',
```

## 測試設計決策與理由

### 決策 1：為什麼不 Mock 檔案系統？

**決策**: 使用真實 tmp_path fixture 而非 Mock

**理由**:
1. Sociable Unit Tests 原則：檔案操作是 module 的外部世界邊界
2. 驗證實際的 Path.exists(), read_text() 等行為
3. 編碼、權限等真實問題只能通過真實檔案系統測試
4. tmp_path 成本低，充分隔離

### 決策 2：為什麼設計規則匹配而非精確匹配？

**決策**: .gitignore 規則匹配使用「模糊匹配」（如容許 /coverage/, coverage, coverage/* 都視為有效）

**理由**:
1. 實際使用中規則變體很常見
2. 驗證意圖（覆蓋率目錄被忽略）比精確文字更重要
3. 提升使用者體驗（不會因細微差異報錯）

### 決策 3：為什麼分離 config 目錄檢查和配置檔檢查？

**決策**: 檢查 3（配置檔）和檢查 4（目錄）分開

**理由**:
1. 不同責任：檢查 3 驗證「檔案存在和格式」，檢查 4 驗證「目錄和權限」
2. 允許獨立迭代
3. 更清晰的錯誤報告粒度

### 決策 4：為什麼 README.md 和開發規範是 SHOULD 而非 MUST？

**決策**: 將 .gitignore、.claude 結構、Hook 配置、config 目錄定為 MUST；README 和規範檔定為 SHOULD

**理由**:
1. README 是推薦而非強制（已有 CLAUDE.md）
2. 開發規範檔依語言而異，非所有專案都需要
3. MUST 項目是框架啟動的必要條件
