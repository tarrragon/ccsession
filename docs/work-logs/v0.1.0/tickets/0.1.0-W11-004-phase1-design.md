# 0.1.0-W11-004 Phase 1 功能設計規格：version-shift 命令

## 命令名稱決定

**採用 `version-shift`（子命令形式）**，理由如下：

- `shift` 過短且語義模糊，單獨使用時讓用戶不確定「shift 什麼」
- `version-shift` 明確表達「版本層級的整體位移」，與現有 `migrate`（單一 Ticket 遷移）形成清楚對比
- 符合現有命令命名慣例（`batch-create`、`version-shift` 都是「動作-對象」複合命名）
- 與 `ticket migrate` 並列時，語義邊界一目了然：`migrate` 操作 Ticket，`version-shift` 操作整個版本

## 命令介面規格（CLI API）

### 完整語法

```
ticket version-shift <from_version> <to_version> [OPTIONS]
```

### 位置參數

| 參數 | 必填 | 說明 | 範例 |
|------|------|------|------|
| `from_version` | 是 | 來源版本號（有無 `v` 前綴皆接受） | `0.1.0` 或 `v0.1.0` |
| `to_version` | 是 | 目標版本號（有無 `v` 前綴皆接受） | `0.2.0` 或 `v0.2.0` |

### 選項參數

| 旗標 | 預設 | 說明 |
|------|------|------|
| `--dry-run` | False | 預覽模式：顯示所有預計執行的操作，不實際修改任何檔案 |
| `--no-backup` | False | 跳過備份步驟（不建議，僅限測試環境或已有外部備份時使用） |
| `--skip-todolist` | False | 不更新 `docs/todolist.yaml`（用於版本記錄與 Ticket 目錄不一致的特殊情況） |

### 使用範例

```bash
# 標準使用（建議）
ticket version-shift 0.1.0 0.2.0

# 預覽模式（不執行任何修改）
ticket version-shift 0.1.0 0.2.0 --dry-run

# 不備份（慎用）
ticket version-shift 0.1.0 0.2.0 --no-backup

# 只遷移目錄，不更新 todolist.yaml
ticket version-shift 0.1.0 0.2.0 --skip-todolist

# 版本號帶 v 前綴也接受
ticket version-shift v0.1.0 v0.2.0
```

## 功能流程規格

### 整體操作順序

```
[Step 0] 前置驗證
[Step 1] 備份整個 from_version worklog 目錄
[Step 2] 更新所有 Ticket 的版本欄位
[Step 3] 重新命名 Ticket 檔案
[Step 4] 更新 Ticket 內部交叉引用
[Step 5] 重新命名 worklog 目錄
[Step 6] 更新 todolist.yaml
[Step 7] 輸出操作摘要
```

### Step 0：前置驗證

執行任何修改前，必須完成以下所有驗證，任一失敗則終止並顯示錯誤訊息：

| 驗證項目 | 失敗條件 | 錯誤訊息方向 |
|---------|---------|------------|
| from_version 格式有效 | 不符合 `\d+\.\d+\.\d+` | 格式錯誤提示 |
| to_version 格式有效 | 不符合 `\d+\.\d+\.\d+` | 格式錯誤提示 |
| from_version 不等於 to_version | 相同 | 提示目標版本必須與來源不同 |
| from_version 目錄存在 | `docs/work-logs/v{from_version}/` 不存在 | 提示目錄不存在 |
| to_version 目錄不存在 | `docs/work-logs/v{to_version}/` 已存在 | 提示衝突，要求用戶確認處理方式 |

### Step 1：備份

備份整個 `docs/work-logs/v{from_version}/` 目錄至：

```
.claude/migration-backups/{timestamp}/v{from_version}/
```

- `{timestamp}` 格式：`%Y%m%d-%H%M%S`（與現有 `migrate` 備份格式一致）
- 使用 `shutil.copytree()` 遞迴複製整個目錄
- 備份失敗時：顯示警告，詢問是否繼續（預設為停止）

### Step 2：更新每個 Ticket 的版本欄位

掃描 `docs/work-logs/v{from_version}/tickets/` 下的所有 `*.md` 檔案，對每個 Ticket 更新：

| Ticket 欄位 | 更新規則 | 範例 |
|------------|---------|------|
| `id` | `{from_version}-W{n}-{seq}` → `{to_version}-W{n}-{seq}` | `0.1.0-W3-001` → `0.2.0-W3-001` |
| `version` | 直接替換為 to_version | `0.1.0` → `0.2.0` |
| `blockedBy` | 將列表中所有 `{from_version}-` 開頭的 ID 做版本替換 | `0.1.0-W2-001` → `0.2.0-W2-001` |
| `parent_id` | 若為 `{from_version}-` 開頭則替換 | 同上 |
| `children` | 列表或 dict 形式，將 `{from_version}-` 開頭的 ID 替換 | 同上 |
| `relatedTo` | 將列表中所有 `{from_version}-` 開頭的 ID 替換 | 同上 |
| `spawned_tickets` | 將列表中所有 `{from_version}-` 開頭的 ID 替換 | 同上 |
| `source_ticket` | 若為 `{from_version}-` 開頭則替換 | 同上 |
| `chain.root` | 若為 `{from_version}-` 開頭則替換 | 同上 |
| `chain.parent` | 若不為 null 且為 `{from_version}-` 開頭則替換 | 同上 |

注意：欄位更新後，檔案尚未移動，仍在原目錄下（Step 3 才移動）。

### Step 3：重新命名 Ticket 檔案

在同目錄內將檔案名稱從 `{from_version}-W{n}-{seq}.md` 重新命名為 `{to_version}-W{n}-{seq}.md`。

注意事項：
- 附屬文件（如 `0.1.0-W1-005-phase2-test-design.md`）也需要重新命名
- 附屬文件識別規則：檔名以 `{from_version}-W{n}-{seq}` 開頭，後接 `-` 加描述字串
- 新檔名：將 `{from_version}` 前綴替換為 `{to_version}`，其餘部分保留

### Step 4：更新跨版本交叉引用

此步驟掃描 **其他版本**（非 from_version）的 Ticket 目錄，更新其中對 `{from_version}-` ID 的引用。

掃描範圍：`docs/work-logs/v*/tickets/*.md`，排除剛被更新的 `v{to_version}/tickets/`

更新欄位：與 Step 2 相同的欄位集合（blockedBy、children、relatedTo、spawned_tickets、source_ticket、parent_id）

### Step 5：重新命名 worklog 目錄

將 `docs/work-logs/v{from_version}/` 整個目錄重新命名為 `docs/work-logs/v{to_version}/`。

- 此時目錄內的 Ticket 檔案已在 Step 2-3 更新完畢
- 目錄內的非 Ticket `.md` 主文件（如 `phase4-evaluation-report.md`）也一併移動，**不修改其內容**
- 使用 `shutil.move()` 或 `Path.rename()`

### Step 6：更新 todolist.yaml

讀取 `docs/todolist.yaml`，更新以下欄位（若值等於 from_version）：

| 欄位路徑 | 更新條件 | 說明 |
|---------|---------|------|
| `current_version` | 值等於 `{from_version}` | 直接替換為 `{to_version}` |
| `previous_version` | 值等於 `{from_version}` | 直接替換為 `{to_version}` |
| `tech_debt[*].source_version` | 值等於 `v{from_version}`（有 v 前綴） | 替換為 `v{to_version}` |
| `tech_debt[*].source_ticket` | 值以 `{from_version}-` 開頭 | 進行版本前綴替換 |

設計決策：**source_version 和 source_ticket 欄位都應更新**。理由：
- `source_version` 記錄「此技術債在哪個版本發現」——版本整體後挪時，發現情境也跟著移動
- `source_ticket` 記錄發現技術債的 Ticket ID，而這個 Ticket 已被遷移至新版本
- 若不更新，todolist.yaml 將指向已不存在的版本目錄和 Ticket，造成記錄不一致

若 `--skip-todolist` 旗標啟用，則跳過此步驟並輸出提示訊息。

### Step 7：輸出操作摘要

操作完成後，輸出結構化摘要：

```
============================================================
version-shift 完成摘要
============================================================
來源版本: 0.1.0
目標版本: 0.2.0
備份位置: .claude/migration-backups/20260307-234500/v0.1.0/

處理結果:
  Ticket 更新: 89 個
  附屬文件更新: 3 個
  跨版本引用更新: 0 個
  todolist.yaml 欄位更新: 5 個

目錄操作:
  docs/work-logs/v0.1.0/ → docs/work-logs/v0.2.0/

============================================================
```

## 邊界條件規格

### 邊界條件 1：from_version 不存在

- 條件：`docs/work-logs/v{from_version}/` 目錄不存在
- 處理：在 Step 0 驗證階段終止，輸出錯誤訊息
- 訊息方向：「版本目錄不存在：docs/work-logs/v{from_version}/」
- exit code：1

### 邊界條件 2：to_version 已存在（衝突）

- 條件：`docs/work-logs/v{to_version}/` 目錄已存在
- 處理：在 Step 0 驗證階段終止，輸出錯誤訊息
- 訊息方向：「目標版本目錄已存在：docs/work-logs/v{to_version}/，無法覆蓋」
- 設計決策：**不提供 `--force` 強制覆蓋選項**，理由：
  - 覆蓋已有目標版本是高風險操作，容易造成資料遺失
  - 用戶應先手動確認目標目錄的內容，再決定如何處理
  - 若確實需要，用戶可先手動刪除目標目錄後重試
- exit code：1

### 邊界條件 3：from_version 等於 to_version

- 條件：標準化後兩個版本號相同
- 處理：在 Step 0 驗證階段終止
- 訊息方向：「來源版本與目標版本相同，無需遷移」
- exit code：0（視為無操作，非錯誤）

### 邊界條件 4：worklog 目錄只有主文件，沒有 tickets 子目錄

- 條件：`docs/work-logs/v{from_version}/` 存在但其中沒有 `tickets/` 子目錄
- 處理：跳過 Ticket 更新步驟，直接執行目錄重新命名
- 輸出提示：「警告：找不到 tickets/ 子目錄，跳過 Ticket 更新」
- 繼續執行其他步驟（目錄重新命名、todolist.yaml 更新）

### 邊界條件 5：Ticket 檔案解析失敗

- 條件：某個 Ticket `.md` 檔案無法解析 frontmatter
- 處理：記錄警告，跳過該檔案，繼續處理其他 Ticket
- 不影響整體操作的繼續執行
- 在最終摘要中列出跳過的檔案數量

### 邊界條件 6：備份失敗

- 條件：備份過程中發生 IO 錯誤
- 處理：顯示錯誤訊息，**停止操作**（預設行為，不繼續執行後續步驟）
- 理由：備份失敗時繼續操作有資料遺失風險
- 使用者可使用 `--no-backup` 跳過備份步驟（自行承擔風險）

### 邊界條件 7：worklog 主文件（非 Ticket 的 .md 檔）的內容

- 條件：`docs/work-logs/v{from_version}/` 下有非 Ticket 的 `.md` 主文件（如 `phase4-evaluation-report.md`）
- 處理：**不修改這些文件的內容**，只隨目錄移動
- 設計決策：不嘗試在 worklog 主文件中搜尋並替換版本號引用，理由：
  - worklog 主文件內容為敘述性文字，版本號可能出現在各種語意上下文（「在 v0.1.0 實作了...」是歷史記錄，不應被修改）
  - 修改歷史敘述會破壞記錄的真實性
  - Ticket 和 todolist.yaml 已足夠讓系統維持一致性

### 邊界條件 8：其他版本 Ticket 引用了 from_version 的 Ticket

- 條件：`docs/work-logs/v0.2.0/tickets/` 中有 Ticket 的 `blockedBy` 包含 `0.1.0-W1-001`
- 處理：Step 4 掃描所有其他版本目錄並更新這些引用
- 注意：只替換精確的版本前綴，不做模糊匹配

### 邊界條件 9：todolist.yaml 不存在

- 條件：`docs/todolist.yaml` 檔案不存在
- 處理：跳過 Step 6，輸出警告訊息
- 不視為錯誤，其他步驟正常執行

## 備份策略

### 備份位置與結構

```
.claude/migration-backups/
└── {timestamp}/         # 格式：%Y%m%d-%H%M%S（與現有 migrate 命令一致）
    └── v{from_version}/ # 整個 worklog 目錄的完整副本
        ├── tickets/
        │   ├── 0.1.0-W1-001.md
        │   └── ...
        └── phase4-evaluation-report.md
```

### 備份方式

使用 `shutil.copytree()` 進行整目錄複製（遞迴複製，保留所有子目錄結構和檔案）。

與現有 `migrate` 命令的逐檔備份方式不同，採用整目錄備份，理由：
- 操作規模更大（可能涉及數十至數百個 Ticket）
- 目錄結構完整性比逐檔備份更重要
- 整目錄備份更易於還原（直接複製回原位即可）
- 備份時間幾乎相同，整目錄方式更簡潔

### 備份時機

在 Step 0 驗證通過後、Step 1（實際修改前）立即執行。

### 還原方式（文件說明）

若操作出現問題，用戶可手動將備份目錄複製回原位：

```bash
cp -r .claude/migration-backups/{timestamp}/v{from_version}/ docs/work-logs/v{from_version}/
```

## 驗收標準（供 Phase 2 使用）

### 基本功能驗收

- [ ] `ticket version-shift 0.1.0 0.2.0` 能成功執行
- [ ] 執行後 `docs/work-logs/v0.2.0/tickets/` 目錄存在，`v0.1.0/` 目錄不存在
- [ ] 遷移後所有 Ticket 的 `id` 欄位都是 `0.2.0-W{n}-{seq}` 格式
- [ ] 遷移後所有 Ticket 的 `version` 欄位都是 `0.2.0`
- [ ] 附屬文件（如 `-phase2-test-design.md`）也被正確重新命名
- [ ] worklog 主文件（非 Ticket .md）的內容未被修改

### 交叉引用驗收

- [ ] Ticket 內部的 `blockedBy` 若原本引用同版本 Ticket，遷移後正確指向新版本 Ticket ID
- [ ] Ticket 內部的 `parent_id` 遷移後正確更新
- [ ] Ticket 內部的 `children` 遷移後正確更新（字串形式和 dict 形式都處理）
- [ ] Ticket 內部的 `relatedTo` 遷移後正確更新
- [ ] Ticket 內部的 `spawned_tickets` 遷移後正確更新
- [ ] Ticket 內部的 `source_ticket` 遷移後正確更新
- [ ] Ticket 內部的 `chain.root` 和 `chain.parent` 遷移後正確更新

### todolist.yaml 驗收

- [ ] `current_version` 若等於 from_version 則更新為 to_version
- [ ] `previous_version` 若等於 from_version 則更新為 to_version
- [ ] `tech_debt[*].source_version` 若等於 `v{from_version}` 則更新為 `v{to_version}`
- [ ] `tech_debt[*].source_ticket` 若以 `{from_version}-` 開頭則正確替換版本前綴

### 備份驗收

- [ ] 備份目錄存在於 `.claude/migration-backups/{timestamp}/v{from_version}/`
- [ ] 備份內容與原始目錄結構一致
- [ ] `--no-backup` 旗標能跳過備份步驟

### Dry-run 模式驗收

- [ ] `--dry-run` 後不產生任何實際檔案或目錄變更
- [ ] Dry-run 輸出列出所有預計更新的 Ticket 數量
- [ ] Dry-run 輸出列出目錄重新命名操作
- [ ] Dry-run 輸出列出 todolist.yaml 預計更新的欄位

### 邊界條件驗收

- [ ] from_version 不存在時 exit code 為 1，顯示明確錯誤訊息
- [ ] to_version 已存在時 exit code 為 1，顯示明確錯誤訊息，不修改任何檔案
- [ ] from_version 等於 to_version 時 exit code 為 0，顯示提示訊息，不修改任何檔案
- [ ] 遇到無法解析的 Ticket 檔案時，顯示警告並跳過，繼續處理其他 Ticket
- [ ] `--skip-todolist` 旗標能跳過 todolist.yaml 更新

## 架構決策說明

### 新增獨立命令 vs 擴充 migrate

採用**新增獨立命令** `version-shift.py`，不擴充 `migrate.py`，理由：

1. 操作單元不同：`migrate` 操作「單一 Ticket」，`version-shift` 操作「整個版本計畫」
2. 輸入介面不同：`migrate` 需要 source_id 和 target_id，`version-shift` 只需兩個版本號
3. 副作用範圍不同：`version-shift` 額外操作 todolist.yaml 和整個目錄，擴充 migrate 會使其職責過重
4. 未來維護邊界清晰：兩個命令可獨立演進

### 工具函式複用

`version-shift.py` 可直接複用以下現有函式：

| 現有函式 | 複用方式 |
|---------|---------|
| `_extract_id_components()` | 從 Ticket ID 解析版本、Wave、序號 |
| `_calculate_chain_info()` | 更新 chain.root 和 chain.parent |
| `_load_ticket_from_path()` | 逐一載入 Ticket 檔案 |
| `parse_frontmatter()` | 解析 Ticket YAML frontmatter |
| `save_ticket()` | 儲存更新後的 Ticket |
| `get_project_root()` | 取得專案根目錄 |

交叉引用更新邏輯與 `_update_cross_references()` 類似，但需要處理版本前綴替換而非精確 ID 替換，建議新增專用函式 `_shift_version_in_references()`。
