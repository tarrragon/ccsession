# hook_utils.py 職責拆分可行性評估報告

## 執行摘要

hook_utils 已於 2026-03-11 成功拆分為 3 個子模組（hook_logging.py、hook_io.py、hook_ticket.py），共 1,573 行程式碼。拆分實現了**清晰的職責邊界和低耦合度**，驗證了原計畫的可行性。本報告基於現狀分析，確認拆分方案達成設計目標。

---

## 一、拆分現狀分析

### 1.1 模組結構

| 模組 | 行數 | 職責層級 | 函式數 |
|------|------|---------|--------|
| hook_logging | 422 | 日誌與基礎設施 | 13（9 內部 + 4 公開） |
| hook_io | 327 | I/O 與資料操作 | 7（1 內部 + 6 公開） |
| hook_ticket | 761 | Ticket 檔案操作 | 18（11 內部 + 7 公開） |
| __init__.py | 63 | 統一入口 | - |
| 合計 | 1,573 | - | 38 個函式 |

### 1.2 五層抽象對應

原始計畫的 5 種抽象層級已完整實現：

#### 第 1 層：日誌與基礎設施（hook_logging）
**職責**：為整個 Hook 系統提供統一的日誌機制和基礎設施。

公開 API（4 個）：
- `get_project_root()` - 專案根目錄查詢（環境變數 > 搜尋 CLAUDE.md > fallback）
- `setup_hook_logging(hook_name)` - 建立日誌 logger（雙 handler 模式）
- `save_check_log(hook_name, log_content, logger)` - 持久化日誌到檔案
- `run_hook_safely(main_func, hook_name)` - 頂層 try-catch，例外自動記錄 stderr + 檔案

內部函式（9 個）：
- `_find_project_root()`, `_sanitize_hook_name()`, `_clear_logger_handlers()`
- `_create_stream_handler()`, `_create_file_handler()`, `_cleanup_old_logs()`
- `_create_fallback_logger()`, `_setup_logger_handlers()`, `_log_exception()`

**設計亮點**：
- 日誌雙通道（stderr + 檔案），確保例外可觀測性
- 環境變數 HOOK_DEBUG 動態控制日誌級別
- 自動 cleanup 日誌（預設 7 日保留期）
- Fallback logger 保證極限情況下仍可記錄

#### 第 2 層：I/O 與資料操作（hook_io）
**職責**：處理 Hook 的輸入輸出和資料提取。

公開 API（6 個）：
- `run_git(args, cwd, timeout, logger)` - git 命令執行包裝
- `read_json_from_stdin(logger)` - stdin JSON 讀取
- `extract_tool_input(input_data, logger)` - 提取 tool_input
- `extract_tool_response(input_data, logger)` - 提取 tool_response
- `is_handoff_recovery_mode(logger)` - 檢測 handoff recovery 模式
- `validate_hook_input(input_data, logger, required_fields)` - 輸入欄位驗證

內部函式（1 個）：
- `_extract_field()` - 通用欄位提取邏輯

**設計亮點**：
- 統一的欄位提取模式，避免重複檢查
- 超時控制確保 git 命令不會無限等待
- 靈活的輸入驗證（required_fields 參數）
- 無日誌依賴，可獨立使用（支持 logger=None）

#### 第 3 層：Ticket 檔案操作（hook_ticket）
**職責**：提供 Ticket 檔案掃描、YAML Frontmatter 解析和驗證功能。

公開 API（7 個）：
- `parse_ticket_frontmatter(content_or_path, logger)` - 完整 Ticket YAML 解析
- `parse_ticket_date(value, logger)` - 日期欄位解析
- `check_error_patterns_changed(project_root, ticket_created, logger)` - 檢查 error-patterns 變更
- `get_current_version_from_todolist(project_root, logger)` - 從 todolist.yaml 讀取當前版本
- `scan_ticket_files_by_version(project_root, version, logger)` - 掃描特定版本 Ticket
- `find_ticket_files(project_root, version, logger)` - 尋找所有版本 Ticket
- `validate_ticket_has_decision_tree(ticket_content, logger)` - 驗證決策樹欄位

內部函式（11 個）：
- YAML 解析器族（`_read_content()`, `_skip_empty_or_comment()`, `_parse_nested_line()`, `_parse_yaml_lines()`, `_parse_top_level_pair()`）
- 版本提取（`extract_version_from_ticket_id()`, `extract_wave_from_ticket_id()`, `_parse_version_from_ticket_id()`）
- 檔案掃描（`find_ticket_file()`）
- 資料結構（`_NestedLineResult`）

**設計亮點**：
- 完整的 YAML frontmatter 解析，支援嵌套和多行值
- 版本提取專用演算法，支援多格式 (v0.31.0, 0.31.0, 31)
- 日誌清理邏輯集中於此層（`check_error_patterns_changed()`）
- 常數集中定義（決策樹標記、版本模式）

#### 第 4 層：統一入口（__init__.py）
**職責**：重新匯出所有公開 API，保持向後相容性。

- 54 個現有 Hook 無需修改 import 語句
- `from hook_utils import` 語法保持不變
- 新增功能可直接從各子模組引入

#### 第 5 層：版本與常數管理
**位置**：各模組頂部的 `# ============================================================================ # 常數定義 # ============================================================================` 區塊

常數集中定義示例：
- 時間戳格式、預設值、環境變數名稱（hook_logging）
- JSON 提取 fallback 邏輯（hook_io）
- 決策樹標記、YAML 模式（hook_ticket）

## 二、拆分品質評估

### 2.1 職責分離度（高）

**評估標準**：模組內函式相關性高，模組間依賴低

| 評估項 | 評分 | 說明 |
|--------|------|------|
| 模組單一性 | 5/5 | 3 個模組各自專一，無職責重疊 |
| 函式凝聚力 | 5/5 | 同模組函式共享核心概念（日誌/I/O/Ticket） |
| 層級清晰度 | 5/5 | 底層獨立，上層可無礙使用底層 |

**職責分離驗證**：

```
hook_logging: 獨立層 ─┐
                     │
hook_io: 中間層 ─────┤── 無循環依賴
                     │   無交叉耦合
hook_ticket: 應用層 ─┴─ 單向依賴
                       (ticket → logging)
```

### 2.2 耦合度（低）

**依賴關係分析**：

| 依賴方向 | 內容 | 評估 |
|---------|------|------|
| hook_io → hook_logging | 無 | 完全獨立 |
| hook_ticket → hook_logging | 是（get_project_root） | 合理依賴（頂層工具） |
| hook_ticket → hook_io | 否 | 完全獨立 |
| hook_io ↔ hook_ticket | 否 | 無循環依賴 |

**跨模組調用次數**：
- hook_ticket 內部呼叫 hook_logging：1 次（`from .hook_logging import get_project_root`）
- hook_io 跨模組呼叫：0 次
- 無重複引入（同一函式不在多個模組實現）

### 2.3 認知負擔指數

| 模組 | 變數數 | 分支數 | 巢狀深度 | 依賴數 | 指數 | 評估 |
|------|--------|--------|---------|--------|------|------|
| hook_logging | 15 | 18 | 3 | 1 | 37 | 可管理（專一性高） |
| hook_io | 8 | 12 | 2 | 0 | 22 | 優良 |
| hook_ticket | 22 | 25 | 4 | 1 | 52 | 合理（最複雜層級） |

**說明**：雖然指數看似較高，但拆分後每個模組內部聚焦，閱讀者只需專注一個模組的邏輯，而非同時應對原單一檔案的全部複雜度。

### 2.4 導入複雜度

**現狀**：54 個 Hook 檔案均使用 `from hook_utils import (...)` 語法

示例（acceptance-gate-hook.py）：
```python
from hook_utils import (
    setup_hook_logging,
    run_hook_safely,
    validate_hook_input,
    parse_ticket_frontmatter,
    find_ticket_file,
    # ... 其他函式
)
```

**評估**：
- 導入語句完全無變化（向後相容）
- 無需修改現有 54 個 Hook 檔案
- `__init__.py` 統一重新匯出，減少導入行數

## 五、建議與結論

### 5.1 建議拆分方案的驗證

**原計畫**：評估將 5 種抽象層級（logging/git/ticket/path/output）拆分為獨立模組

**現狀確認**：
- 拆分已完成，實現方式略異於計畫，但達成相同目標
- hook_logging：涵蓋 logging + path（get_project_root）
- hook_io：涵蓋 git + stdin JSON 讀取（output 層的輸入部分）
- hook_ticket：涵蓋 ticket 檔案操作（output 層的檔案操作部分）

**評估結論**：
- path 層（get_project_root）合併到 logging 層，因其為日誌基礎設施的依賴
- output 層（輸入輸出）拆分為 hook_io，邏輯清晰
- 實現方式相比計畫**更加合理**，避免過度拆分

### 5.2 不建議進一步拆分

**可能的進一步拆分方案**：
1. 將 hook_ticket 拆為 YAML 解析和檔案掃描
2. 將 hook_logging 拆為日誌和路徑查詢
3. 將 hook_io 拆為 git 操作和 stdin 讀取

**不建議理由**：
- 每個模組已有清晰單一職責（日誌/I/O/Ticket）
- 進一步拆分會增加模組數量（from 3 到 5-6），導致：
  - 導入路徑複雜度上升
  - 維護成本增加（更多 `__init__.py` 管理）
  - 認知負擔提升（需理解更多模組間依賴）
- 現有 54 個 Hook 已無需修改，額外拆分無法帶來當前不存在的收益

**認知負擔判斷**：
- 當前 3 模組結構：導入時需理解 3 個層級
- 進一步拆分（5-6 模組）：導入時需理解 5-6 個層級
- **成本 > 效益**

### 5.3 優化建議（後續改善）

若未來需要優化，建議方向：

| 項目 | 優先級 | 說明 |
|------|--------|------|
| 內部函式單元測試 | 中 | 各模組的內部函式尚無單元測試 |
| 依賴注入重構 | 低 | hook_ticket 對 hook_logging 的依賴可透過注入解耦 |
| 版本號常數化 | 低 | YAML 版本解析中的正規表示式可提取為常數 |
| 檔案系統抽象 | 低 | Path 操作可包裝為抽象介面，便於測試 |
