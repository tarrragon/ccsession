## 建議方案：保留平行實作（維持現狀）

### 結論

**不建議統一到共用模組**。應保留兩個獨立實作。

### 理由分析

#### 1. 套件隔離性原則（最高優先）
- Hook 套件: 獨立 .py 檔案集合，零依賴
- Skill 套件: 結構化 Python 套件，需要 pyyaml
- 若統一到共用模組，會破壞「零依賴」的 Hook 套件設計

#### 2. 維護成本 vs 重複成本
- 函式極小（20 行），程式邏輯簡單
- 重複維護成本: 若需修改正則表達式，需同步 2 個位置
- 統一成本: 引入共用模組會增加複雜度和潛在循環依賴風險
- **結論**: 重複成本低於統一成本

#### 3. 現有使用情境
- Hook: 內部使用（acceptance-gate-hook.py）
- Skill: 內部使用（ticket_system 套件內）
- **不存在跨套件呼叫的需求**（目前無），統一價值有限

#### 4. 更好的替代方案
若未來發現需要統一，可在以下時機執行：
- 函式邏輯變複雜（目前僅 1 個正則搜尋）
- 發現跨套件呼叫需求（目前無）
- 統一共用庫的內部決策變更（新增 requirements.txt 等）

### 小結

當前保留平行實作是符合「套件隔離」和「維持簡潔」原則的最優選擇。
