## 1. 功能概述

### 1.1 核心問題

IMP-031 揭露代理人在並行派發場景下可能發生「部分完成但報告全部完成」的問題：代理人回報任務完成，但 `git diff --stat` 顯示部分預期修改的檔案實際未變更。

現有的手動防線是：PM 在收到代理人回報後，手動執行 `git diff --stat` 並比對 Ticket 的 `where.files` 清單。此防線依賴人工記憶執行，容易被省略（PC-010 案例顯示 Checkpoint 流程跳過率偏高）。

### 1.2 功能目標

在並行代理人回報完成後，自動比對以下兩個集合：

- **預期變更集合**：Ticket 的 `where.files` 清單（代理人應該修改哪些檔案）
- **實際變更集合**：`git diff --stat` 結果（代理人實際修改了哪些檔案）

當兩者不一致時，自動輸出警告訊息，提醒 PM 介入。

### 1.3 設計哲學（保守設計）

本 Hook 遵循「只驗證+警告，不阻擋、不修改」原則：

- Hook 永遠以 exit 0 結束（不阻擋後續工作流）
- 輸出警告訊息到 additionalContext（提醒但不強制）
- 不自動補完遺漏的修改（修復決策留給 PM）
- 誤報容忍度優先於漏報容忍度（寧願多警告，不省略警告）

## 2. 觸發機制

### 2.1 Hook 類型選擇

**選定：PostToolUse (Bash)**

理由：
- 並行代理人通過執行 `ticket track complete` 命令來回報完成
- PostToolUse (Bash) 是唯一能監聽此命令的 Hook 事件
- 現有 handoff-cleanup-hook.py 和 post-ticket-complete-checkpoint-hook.py 均使用相同觸發模式，可參照其設計

**替代方案排除**：
- UserPromptSubmit：代理人回報使用 Bash 工具，不是用戶 prompt
- PreToolUse：在命令執行前，無法取得 complete 結果
- Stop Hook：時機太晚，可能已離開並行派發的關鍵時間視窗

### 2.2 觸發條件（全部滿足才觸發）

| 條件 | 判斷方式 | 來源欄位 |
|------|---------|---------|
| 工具類型為 Bash | `tool_name == "Bash"` | `input.tool_name` |
| 命令包含 ticket complete | `"ticket track complete"` 在命令中 | `input.tool_input.command` |
| 命令執行成功 | exit code == 0 且輸出含成功標記 | `input.tool_response.stdout` |
| 被完成的 Ticket 有 where.files | Ticket frontmatter 存在且非空 | Ticket 檔案 |

### 2.3 並行派發場景判斷（關鍵設計決策）

**設計問題**：Hook 無法直接知道某 Ticket 是否來自並行派發。

**解決方案：保守策略 — 對所有 ticket complete 都觸發驗證**

理由：
1. 即使是非並行派發，比對 where.files 也是有益的驗證
2. 並行派發場景識別的誤判（false negative）代價高於多餘觸發（false positive）的代價
3. 簡化實作，不需要維護並行派發狀態

**補充判斷（可選增強）**：若 Ticket 的 `chain.parent` 為 null 且同一 Wave 有其他 in_progress Ticket，則標記為「高可能性並行場景」，警告訊息加強提示。

## 3. 輸入規格

### 3.1 Hook 接收的輸入（PostToolUse JSON）

```json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "uv -d .claude/skills/ticket run ticket track complete 0.1.0-W41-002"
  },
  "tool_response": {
    "stdout": "[OK] 已完成 Ticket 0.1.0-W41-002",
    "stderr": "",
    "exit_code": 0
  }
}
```

### 3.2 從 Ticket 讀取的資料

Hook 需要讀取被完成 Ticket 的 `where.files` 欄位。

**Ticket frontmatter 格式**：
```yaml
where:
  layer: framework-quality
  files:
  - .claude/hooks/parallel-dispatch-verification-hook.py
  - .claude/settings.json
```

**`where.files` 路徑格式說明**：
- 相對路徑（相對於 project root）：`.claude/hooks/foo.py`
- 不含 glob 模式（Ticket 系統慣例）
- 可能有多個條目

### 3.3 從 git 讀取的資料

執行 `git diff --name-only HEAD~1` 或 `git diff --stat` 取得實際變更檔案清單。

**git diff 輸出格式說明**：
- 相對路徑（相對於 repo root）：`.claude/hooks/foo.py`
- 路徑格式與 where.files 一致（無需額外轉換）

## 4. 路徑規範化設計

### 4.1 問題場景

| 來源 | 路徑範例 | 潛在問題 |
|------|---------|---------|
| where.files | `.claude/hooks/foo.py` | 前綴 `./`、尾斜線 |
| git diff | `.claude/hooks/foo.py` | 一般無問題 |
| where.files | `./ui/lib/models/user.dart` | 前綴 `./` 需移除 |

### 4.2 規範化規則

所有路徑在比對前必須執行以下規範化：

1. 移除前綴 `./`
2. 統一使用正斜線 `/`（Windows 相容）
3. 移除尾斜線 `/`
4. 轉為小寫（macOS 不分大小寫，但保持一致性）

**規範化函式簽名**：
```python
def normalize_path(path: str) -> str:
    """
    標準化路徑格式，用於跨來源比對。

    Args:
        path: 原始路徑字串

    Returns:
        str - 標準化後的路徑
    """
```

### 4.3 不支援的格式（本版本範圍外）

以下格式在 Phase 1 設計中**不處理**，視為邊界條件：

| 格式 | 處理方式 |
|------|---------|
| glob 模式（`lib/**/*.dart`） | 視為純字串比對，不展開 |
| 絕對路徑 | 規範化移除根路徑後比對 |
| 目錄路徑（`lib/models/`） | 視為前綴匹配：git diff 中任何以此開頭的檔案均視為覆蓋 |

## 5. 比對邏輯設計

### 5.1 比對演算法

```
where_files = 從 Ticket 讀取的預期修改檔案清單（規範化後）
git_changed = 從 git diff 讀取的實際變更檔案清單（規範化後）

missing_files = where_files - git_changed  # 在預期中但不在實際中
extra_files = git_changed - where_files    # 在實際中但不在預期中（正常，不需要警告）
```

**只關注 missing_files**：代理人應修改但未修改的檔案。extra_files（代理人修改了額外檔案）不觸發警告，因為：
1. 代理人可能需要修改輔助檔案（如測試檔案、常數檔等）
2. 過度警告會降低 Hook 的可信度

### 5.2 git diff 基準選擇

**問題**：`git diff` 需要一個「基準點」，才能知道「這次派發之後」改了什麼。

**設計決策**：使用 `git diff HEAD --name-only`

理由：
- 比對 working tree vs HEAD（已提交的狀態），涵蓋所有未提交變更
- 不需要記住派發前的 commit hash（無狀態設計）
- 缺點：若有其他未提交的修改（來自其他任務），可能產生誤判 → 在警告訊息中說明此限制

**備選方案（排除）**：
- `git diff HEAD~1`：假設上次 commit 是派發前，不總是成立
- 儲存派發前 commit hash：需要跨 Hook 執行的持久化狀態，複雜度高

### 5.3 成功判定標準

| 狀態 | 定義 | 輸出行為 |
|------|------|---------|
| 通過 | `missing_files` 為空集合 | 靜默（不輸出任何訊息） |
| 警告 | `missing_files` 非空 | 輸出 additionalContext 警告 |

## 6. 輸出規格

### 6.1 正常通過（靜默）

當 `missing_files` 為空時，Hook 靜默輸出：

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse"
  }
}
```

### 6.2 警告輸出格式

當 `missing_files` 非空時，輸出 additionalContext：

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "[警告] 並行派發驗證：...(見下方訊息格式)"
  }
}
```

### 6.3 警告訊息格式（additionalContext 內容）

```
============================================================
[並行派發驗證警告]
============================================================

Ticket {ticket_id} 完成後，偵測到以下預期修改的檔案未出現在 git diff 中：

缺失檔案（{N} 個）：
  - {file_1}
  - {file_2}

預期修改（where.files 清單，共 {M} 個）：
  - {all_where_files}

實際變更（git diff HEAD --name-only，共 {K} 個相關檔案）：
  - {matching_git_files}

建議動作：
1. 確認代理人是否確實完成了這些檔案的修改
2. 若代理人遺漏了修改，手動補完或重新派發
3. 補完後執行 git diff 確認

注意：此驗證使用 git diff HEAD，包含所有未提交的變更。
若有其他任務的未提交變更，可能影響比對結果。

============================================================
```

### 6.4 雙通道輸出原則（品質基線 Rule 4）

| 輸出通道 | 觸發條件 | 內容 |
|---------|---------|------|
| additionalContext | 有警告時 | 完整警告訊息 |
| stderr | Hook 異常時 | 錯誤訊息和 traceback |
| 日誌檔 | 每次執行 | 執行摘要（含比對結果） |

**日誌目錄**：`.claude/hook-logs/parallel-dispatch-verification/`

**日誌格式**：

```
[{timestamp}]
  TicketID: {ticket_id}
  WhereFiles: {N} 個
  GitChanged: {K} 個
  MissingFiles: {M} 個
  Result: {PASS | WARN}
  MissingList: {list or empty}
```

## 8. 邊界條件與例外處理

### 8.1 正常邊界條件

| 條件 | 處理方式 |
|------|---------|
| `where.files` 為空或不存在 | 跳過驗證，靜默結束（無法比對） |
| git diff 回傳空列表 | 所有 where.files 視為缺失，輸出警告 |
| git 命令執行失敗 | 記錄錯誤到日誌，靜默結束（不阻擋） |
| Ticket 檔案不存在 | 記錄警告到日誌，靜默結束 |
| 無法提取 Ticket ID | 靜默結束（無法定位 Ticket） |

### 8.2 誤報防護設計

**潛在誤報場景 1：Ticket 只需修改 Hook 設定，不修改 .py 檔**

- 設計：`where.files` 應包含 `settings.json`（Ticket 建立時的責任）
- Hook 行為：若 `settings.json` 在 where.files 但不在 git diff → 警告
- 結論：where.files 的準確性決定誤報率，Hook 不引入額外誤報

**潛在誤報場景 2：並行派發多個 Ticket，第一個 complete 時其他的修改尚未提交**

- 設計：每個 Ticket complete 時分別驗證
- Hook 行為：只驗證「已完成」的那個 Ticket 的 where.files
- 但 git diff HEAD 包含所有未提交變更（含其他 Ticket 的修改）→ 實際上減少誤報
- 結論：並行場景下，誤報率反而更低（共享的 git diff 集合更大）

**潛在誤報場景 3：Ticket where.files 列出了「參考用」的檔案，實際上不會修改**

- 設計：where.files 應只列實際會修改的檔案（Ticket 建立規範）
- Hook 行為：視為缺失，輸出警告
- 結論：此為 Ticket 建立品質問題，Hook 的警告是正確的

### 8.3 例外處理矩陣

| 例外情況 | stderr 輸出 | 日誌記錄 | additionalContext | exit code |
|---------|------------|---------|-----------------|-----------|
| JSON 解析失敗 | 是 | 是 | 否 | 0 |
| Ticket 讀取失敗 | 否 | 是（warning） | 否 | 0 |
| git 命令失敗 | 是 | 是 | 否 | 0 |
| 路徑規範化失敗 | 否 | 是（debug） | 否（使用原始路徑） | 0 |
| 任何未預期異常 | 是 | 是（critical） | 否 | 0 |

## 10. 行為場景（Given-When-Then）

### 場景 1：缺失檔案偵測 — 主要場景

```
Given: Ticket 0.1.0-W41-002 的 where.files 為
       [".claude/hooks/foo.py", ".claude/settings.json"]
  And: git diff HEAD --name-only 輸出只有
       ".claude/hooks/foo.py"
  And: 代理人執行 ticket track complete 0.1.0-W41-002 成功（exit 0, 輸出含"已完成"）

When: PostToolUse Hook 觸發

Then: Hook 輸出 additionalContext 包含 "[並行派發驗證警告]"
  And: 警告訊息中列出 ".claude/settings.json" 為缺失檔案
  And: exit code 為 0
```

### 場景 2：全部匹配 — 靜默通過


## 11. 設計約束總結

| 約束項目 | 決策 | 理由 |
|---------|------|------|
| Hook 類型 | PostToolUse (Bash) | 唯一能監聽 ticket complete 的事件 |
| 觸發範圍 | 所有 ticket complete，非僅並行場景 | 簡化設計，誤報率可接受 |
| git 基準 | git diff HEAD | 無狀態設計，不需持久化 |
| 比對方向 | 只警告 missing（where -> git），忽略 extra | 減少誤報，extra 修改是正常現象 |
| 阻擋行為 | 永遠 exit 0 | 保守設計，警告不等於阻擋 |
| glob 支援 | 本版本不支援展開 | 降低複雜度，Ticket where.files 慣例為具體路徑 |
| 自我修改 | 不修改任何 Ticket 或程式碼 | 純驗證角色 |
