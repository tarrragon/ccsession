# Execution Log

## Task Summary

重構 pending handoff 掃描迴圈（list_pending_handoffs 與 _collect_stale_handoffs 共用邏輯）

---

## Problem Analysis

`resume.py` 的 `list_pending_handoffs()` 與 `handoff_gc.py` 的 `_collect_stale_handoffs()` 存在 95%+ 的重複掃描邏輯：

1. **掃描邏輯重複**（glob 模式、迭代順序）：
   - 兩者均使用 `sorted(pending_dir.glob("*.json")) + sorted(pending_dir.glob("*.md"))`
   - 同步掃描 JSON 和 Markdown 檔案

2. **JSON 讀取和錯誤處理相同**：
   - 開檔、json.load()、異常 catch 的邏輯完全相同

3. **Stale 判斷邏輯重複**：
   - 判斷來源 ticket 是否 completed
   - 區分任務鏈與非任務鏈
   - 檢查目標 ticket 啟動狀態
   - 驗證 from_status 欄位

只有**呼叫端的錯誤處理策略不同**：
- `list_pending_handoffs` 計數並輸出 stderr 警告
- `_collect_stale_handoffs` 只 continue（不計數）

---

## Solution

設計**結構化掃描結果** `ParsedHandoff` 資料類，包含：
- 檔案資訊、解析結果、錯誤標記
- 支援呼叫端自訂錯誤處理

在 `handoff_utils.py` 新增 `scan_pending_handoffs()` 函式：
1. 統一實現掃描和基本驗證（JSON 讀取、必填欄位檢查）
2. 返回完整的 `ParsedHandoff` 清單
3. 每個記錄包含 parse_error 和 schema_error 欄位

重構兩個函式：
- `list_pending_handoffs()`: 使用共用掃描，記錄計數並輸出 stderr
- `_collect_stale_handoffs()`: 使用共用掃描，跳過錯誤記錄

**優勢**：
- DRY 原則達成（重複代碼消除）
- 易於維護（stale 判斷邏輯一處）
- 支援靈活的錯誤處理（呼叫端自訂）
- 認知負擔降低（複雜迴圈從 2 個減至 1 個）
