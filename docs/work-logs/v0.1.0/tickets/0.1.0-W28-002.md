## 現有問題

22+ 個 Hook 檔案各自重複實現 tool_input/tool_response 的解析邏輯：

1. **模式 A：tool_input 提取**（22+ 個 Hook）
   - 所有 Hook 都用相同的邏輯：先 .get('tool_input') 再 or {} 再逐個欄位提取
   
2. **模式 B：tool_response 提取**（8 個 Hook）
   - 重複提取 stdout、stderr、exit_code 等標準欄位
   
3. **模式 C：tool_name 驗證**（15+ 個 Hook）
   - 重複的 tool_name 檢查和比對邏輯

## 根本原因

共用提取邏輯缺失，導致 DRY 原則違反。

## 新增共用函式

在 hook_utils.py 中新增兩個公開 API 函式（419-520 行）：

### 1. extract_tool_input(input_data, logger=None)

簽名：
- 入參：input_data (dict | None), logger (logging.Logger | None)
- 回傳：dict（始終返回 dict，無欄位時返回 {}）

功能：
- 安全提取 input_data['tool_input'] 欄位
- 處理 None、空值、型別錯誤等異常情況
- 可選記錄詳細日誌

### 2. extract_tool_response(input_data, logger=None)

簽名：
- 入參：input_data (dict | None), logger (logging.Logger | None)
- 回傳：dict（始終返回 dict，無欄位時返回 {}）

功能：
- 安全提取 input_data['tool_response'] 欄位
- 處理 None、空值、型別錯誤等異常情況
- 可選記錄詳細日誌

## 實作特色

- 防禦性設計：處理 None、空、非 dict 型別輸入
- 型別檢查：兩層驗證（input_data、欄位本身）
- 日誌友善：可選 logger 參數，支援詳細除錯記錄
- DRY 優先：統一了 22+ Hook 的重複邏輯
- 無副作用：純函式，不修改輸入資料
- Python 3.9 相容：使用字串型別標註而非 PEP 604
