## 3b 拆分子任務結構設計規格

### 一、原始 3b Ticket 角色定義

拆分後，原始 Phase 3b Ticket **轉為協調任務（coordinator）**：

| 屬性 | 規則 |
|------|------|
| status | 維持 in_progress 直到所有子任務完成 |
| where.files | 更新為所有子任務 files 的聯集（參考用） |
| who | PM（協調角色，不執行實作） |
| tdd_phase/tdd_stage | 維持 Phase 3b 標記 |
| acceptance | 改為「所有子任務完成 + 整合測試通過」 |

**PM 操作流程**：
1. 評估觸發拆分 → 原始 3b Ticket 不 complete
2. 建立 N 個子任務（`/ticket create --parent {3b_id}`）
3. 並行派發 parsley 執行各子任務
4. 所有子任務完成 → complete 原始 3b Ticket
5. 進入 Phase 4a

### 二、子任務 Ticket 結構規則

#### 2.1 欄位繼承

| 欄位 | 繼承方式 |
|------|---------|
| version | 繼承父 Ticket |
| wave | 繼承父 Ticket |
| type | 繼承（IMP） |
| tdd_phase | 繼承（3b） |
| who | parsley-flutter-developer |
| parent_id | 設為原始 3b Ticket ID |
| priority | 繼承父 Ticket |

#### 2.2 where.files 分配規則

| 規則 | 說明 |
|------|------|
| 無交集（強制） | 子任務的 where.files 之間不可重疊 |
| 測試跟隨產品碼 | test/module_a_test.dart 跟隨 lib/module_a/ 歸同一子任務 |
| 共用檔案歸屬見第三節 | constants、共用型別等有專門規則 |
| 聯集 = 父 Ticket scope | 所有子任務 files 加總應覆蓋父 Ticket 的完整範圍 |

#### 2.3 子任務命名

子任務 title 格式：`Phase 3b-{N}: {模組/層級描述}`

範例：
- Phase 3b-1: WebSocket 連線模組實作
- Phase 3b-2: JSONL Parser 模組實作

### 三、共用檔案歸屬規則

Phase 3a 策略文件通常會標識出共用基礎設施（shared constants、型別定義、基底類別）。

#### 3.1 判斷流程

```
共用檔案被 2+ 子任務需要
    |
    v
共用檔案的修改是「新增」還是「修改既有」？
    |
    +-- 新增（新常數、新型別）→ 歸入最先使用它的子任務
    |
    +-- 修改既有 → 影響範圍大嗎？
        |
        +-- 小（加一個常數）→ 歸入修改量最大的子任務
        +-- 大（重構共用介面）→ 獨立為「共用基礎設施」子任務，其他子任務 blockedBy 它
```

#### 3.2 常見共用檔案處理

| 檔案類型 | 歸屬規則 |
|---------|---------|
| constants.dart | 歸入修改量最大的子任務（通常是最核心的模組） |
| 共用 Entity/VO | 如果只新增欄位 → 歸入需要該欄位的子任務 |
| 基底類別/抽象介面 | 獨立子任務（其他 blockedBy） |
| barrel files (index.dart) | 歸入最後完成的子任務（或協調任務） |
| pubspec.yaml 新依賴 | 歸入需要該依賴的子任務 |

### 四、依賴關係模式與 blockedBy 規則

#### 4.1 三種依賴模式

**模式 A：完全獨立（最常見）**

```
Parent 3b
├── Child A (模組 X) - blockedBy: []
├── Child B (模組 Y) - blockedBy: []
└── Child C (模組 Z) - blockedBy: []
→ 全部並行派發
```

適用：各模組無共用程式碼修改、無介面依賴。

**模式 B：共用基礎設施優先**

```
Parent 3b
├── Child A (共用基礎) - blockedBy: []
├── Child B (模組 X) - blockedBy: [Child A]
└── Child C (模組 Y) - blockedBy: [Child A]
→ A 先完成，B/C 並行
```

適用：有共用介面/型別需先定義，其他模組依賴它。

**模式 C：分層序列**

```
Parent 3b
├── Child A (domain 層) - blockedBy: []
├── Child B (service 層) - blockedBy: [Child A]
└── Child C (UI 層) - blockedBy: [Child B]
→ 嚴格序列
```

適用：按架構層拆分（策略 1），每層依賴下層。

#### 4.2 blockedBy 判斷標準

| 條件 | 設定 blockedBy |
|------|--------------|
| 子任務 B 修改的程式碼呼叫子任務 A 新建的介面 | 是（B blockedBy A） |
| 子任務 A/B 修改完全獨立的模組 | 否 |
| 子任務 B 的測試需要 A 的產出物 | 是 |
| 子任務 A/B 都讀（但不寫）同一個檔案 | 否（Read-Read 不衝突） |
| 兩個子任務都新增到同一個 barrel file | 是（合併到同一子任務，或最後由協調任務處理） |

### 五、完成條件與異常處理

#### 5.1 父 Ticket 完成條件

| 條件 | 必要性 |
|------|--------|
| 所有子任務 status = completed | 強制 |
| 整合測試通過（如有跨模組互動） | 強制 |
| where.files 聯集完整覆蓋 | 強制（git diff --stat 驗證） |

#### 5.2 子任務失敗處理

```
子任務失敗
    |
    v
[強制] /pre-fix-eval → incident-responder
    |
    v
修復 Ticket 建為失敗子任務的子任務
    |
    v
其他並行子任務不受影響（繼續執行）
    |
    v
失敗子任務修復後 → 繼續等待所有子任務完成
```

**重要**：一個子任務失敗不需要暫停其他子任務（除非有 blockedBy 依賴）。
