## 二、實作策略與虛擬碼

### 2.1 `validate_tool_input()` 的實作策略

#### 2.1.1 責任邊界

**前置條件**：呼叫端已執行 `validate_hook_input(input_data, logger, ("tool_name", "tool_input"))`，確保 tool_name 和 tool_input 的頂層存在性。

**責任**：驗證 tool_input 內的指定子欄位。

**防禦性**：即使前置條件未滿足，函式仍執行 None 檢查，確保健壯性（不會 raise）。

#### 2.1.2 虛擬碼

```
函式: validate_tool_input(input_data, logger, required_subfields)
回傳: bool

1. 防禦性檢查 input_data
   IF input_data 為 None OR 非 dict:
       IF logger 存在:
           記錄 error "輸入資料為 None 或非 dict，無法驗證 tool_input 子欄位"
       RETURN False

2. 取得 tool_input 欄位
   tool_input = input_data.get("tool_input")
   IF tool_input 為 None:
       IF logger 存在:
           記錄 error "tool_input 欄位缺失或為 None"
       RETURN False

3. 驗證 tool_input 為 dict 型別
   IF tool_input 非 dict:
       IF logger 存在:
           記錄 error "tool_input 非 dict 型別: {type}"
       RETURN False

4. 子欄位驗證迴圈
   IF required_subfields 非空:
       FOR 每個 field IN required_subfields:
           IF field 不在 tool_input:
               IF logger 存在:
                   記錄 error "tool_input 缺少必要子欄位: {field}"
               RETURN False

           IF tool_input[field] 為 None:
               IF logger 存在:
                   記錄 error "tool_input 子欄位值為 None: {field}"
               RETURN False

5. 驗證通過
   IF logger 存在:
       記錄 debug "tool_input 子欄位驗證通過"
   RETURN True
```

#### 2.1.3 實作注意事項

- **回傳值型別**：`bool`（不拋出異常）
- **日誌策略**：logger 為 None 時靜默執行（不拋出異常）
- **順序**：防禦→型別檢查→子欄位驗證（early return）
- **空字串視為有效**：只檢查 None，不檢查 falsy 值
- **多欄位缺失**：發現第一個缺失欄位即返回，不檢查所有欄位

### 2.2 `validate_ticket_unified()` 的實作策略

#### 2.2.1 責任邊界

**責任**：整合 `find_ticket_file()` + 檔案讀取 + `validate_ticket_has_decision_tree()` 三步，提供統一的 Ticket 驗證入口。

**回傳值**：Tuple[bool, Optional[str]]，包含驗證結果和錯誤訊息。

#### 2.2.2 虛擬碼

```
函式: validate_ticket_unified(ticket_id, project_root, logger)
回傳: Tuple[bool, Optional[str]]

1. 解析 project_root
   IF project_root 為 None:
       project_root = get_project_root()

2. 尋找 Ticket 檔案
   ticket_path = find_ticket_file(ticket_id, project_root, logger)
   IF ticket_path 為 None:
       msg = "找不到 Ticket: {ticket_id}"
       IF logger 存在:
           記錄 error msg
       RETURN (False, msg)

3. 讀取 Ticket 內容
   TRY:
       content = ticket_path.read_text(encoding="utf-8")
   EXCEPT Exception as e:
       msg = "無法讀取 Ticket 檔案: {ticket_id}"
       IF logger 存在:
           記錄 error "{msg}: {e}"
       RETURN (False, msg)

4. 驗證內容非空
   IF NOT content:
       msg = "無法讀取 Ticket 檔案: {ticket_id}"
       IF logger 存在:
           記錄 error msg
       RETURN (False, msg)

5. 驗證決策樹欄位
   IF NOT validate_ticket_has_decision_tree(content, logger):
       msg = "Ticket {ticket_id} 缺少決策樹欄位，為無效 Ticket"
       IF logger 存在:
           記錄 error msg
       RETURN (False, msg)

6. 驗證通過
   IF logger 存在:
       記錄 info "Ticket {ticket_id} 驗證通過"
   RETURN (True, None)
```

#### 2.2.3 實作注意事項

- **project_root 自動探測**：為 None 時呼叫 `get_project_root()`
- **例外捕獲**：read_text() 的任何例外都返回 (False, msg)，不向上拋出
- **內容非空檢查**：讀取成功但內容為空字串視為失敗
- **決策樹驗證委派**：使用既有的 `validate_ticket_has_decision_tree()`，複用其邏輯
- **錯誤訊息精確**：每種失敗情況有不同的錯誤訊息，便於診斷

def validate_tool_input(input_data, logger=None, required_subfields=None):
    """docstring - 參考 Phase 1 設計文件第 2.1 節"""

    # 防禦性檢查 input_data
    if input_data is None or not isinstance(input_data, dict):
        if logger:
            logger.error("輸入資料為 None 或非 dict，無法驗證 tool_input 子欄位")
        return False

    # 取得 tool_input
    tool_input = input_data.get("tool_input")
    if tool_input is None:
        if logger:
            logger.error("tool_input 欄位缺失或為 None")
        return False

    # 檢查 tool_input 型別
    if not isinstance(tool_input, dict):
        if logger:
            logger.error("tool_input 非 dict 型別: {}".format(type(tool_input)))
        return False

    # 子欄位驗證
    if required_subfields:
        for field in required_subfields:
            if field not in tool_input:
                if logger:
                    logger.error("tool_input 缺少必要子欄位: {}".format(field))
                return False
            if tool_input.get(field) is None:
                if logger:
                    logger.error("tool_input 子欄位值為 None: {}".format(field))
                return False

    if logger:
        logger.debug("tool_input 子欄位驗證通過")
    return True
```

def validate_ticket_unified(ticket_id, project_root=None, logger=None):
    """docstring - 參考 Phase 1 設計文件第 2.2 節"""

    # 步驟 1：解析 project_root
    if project_root is None:
        project_root = get_project_root()

    # 步驟 2：尋找 Ticket 檔案
    ticket_path = find_ticket_file(ticket_id, project_root=project_root, logger=logger)
    if not ticket_path:
        msg = "找不到 Ticket: {}".format(ticket_id)
        if logger:
            logger.error(msg)
        return False, msg

    # 步驟 3：讀取檔案內容
    try:
        content = ticket_path.read_text(encoding="utf-8")
    except Exception as e:
        msg = "無法讀取 Ticket 檔案: {}".format(ticket_id)
        if logger:
            logger.error("{}: {}".format(msg, e))
        return False, msg

    # 步驟 4：驗證內容非空
    if not content:
        msg = "無法讀取 Ticket 檔案: {}".format(ticket_id)
        if logger:
            logger.error(msg)
        return False, msg

    # 步驟 5：驗證決策樹欄位
    if not validate_ticket_has_decision_tree(content, logger):
        msg = "Ticket {} 缺少決策樹欄位，為無效 Ticket".format(ticket_id)
        if logger:
            logger.error(msg)
        return False, msg

    # 步驟 6：驗證通過
    if logger:
        logger.info("Ticket {} 驗證通過".format(ticket_id))
    return True, None
```

## 五、關鍵技術決策與理由

### 5.1 為什麼分為兩個函式？

**決策**：`validate_tool_input()` 和 `validate_ticket_unified()` 分別處理兩個不同層面的驗證。

**理由**：
- **單一責任原則**：tool_input 子欄位驗證 vs Ticket 完整性驗證是兩個獨立的關注點
- **重用性**：tool_input 驗證可被多個 Hook 重用；Ticket 驗證主要服務於派發驗證
- **測試複雜度降低**：分離的函式更容易單獨測試，邊界條件更清晰

### 5.2 為什麼 validate_ticket_unified 回傳 Tuple？

**決策**：回傳 `Tuple[bool, Optional[str]]` 而非布林值。

**理由**：
- **錯誤診斷**：返回的錯誤訊息幫助呼叫端識別失敗原因
- **呼叫端簡化**：不需要額外呼叫 logger.error 單獨處理錯誤
- **向後相容**：原 `validate_ticket()` 函式已回傳此 tuple，修改後行為一致

### 5.3 為什麼不 Mock Domain 物件？

**決策**：使用 Sociable Unit Tests，不 mock dict、logger、Path 等域物件。

**理由**：
- **真實性**：測試使用真實資料結構，更接近生產環境
- **簡化測試邏輯**：無需複雜的 Mock 設定
- **邊界發現**：能夠發現原始碼中可能的邊界情況（如空字串 vs None）

### 5.4 為什麼保留現有 API 而不修改？

**決策**：保留 `find_ticket_file()`, `validate_hook_input()` 等現有 API，新增統一 API 而非修改。

**理由**：
- **零破壞原則**：現有 54 個 Hook 中的 50+ 個使用現有 API，無需修改
- **漸進式重構**：新增 API 先服務於兩個 Hook，未來可逐步推廣
- **回滾安全**：若新 API 有問題，現有 Hook 不受影響

## 六、常見陷阱與防護措施

### 6.1 陷阱 1：型別檢查不完整

**陷阱**：只檢查 None，忽略其他錯誤型別（如 list、int）。

**防護**：
```
if not isinstance(tool_input, dict):
    logger.error("tool_input 非 dict 型別: {}".format(type(tool_input)))
    return False
```

### 6.2 陷阱 2：Early Return 不完整

**陷阱**：繼續執行後續檢查，即使前一步已失敗。

**防護**：每個檢查失敗點都有 `return False`，不繼續執行。

## 十、預期風險與緩解策略

### 10.1 風險 1：類型標註不相容

**風險**：`Tuple[bool, Optional[str]]` 在 Python 3.9 中需要 `from __future__ import annotations`。

**緩解**：
- 檢查 hook_ticket.py 頂部是否有 `from __future__ import annotations`
- 若無，在 `from typing import` 之前加入
- 或使用 `Union[Tuple[bool, str], Tuple[bool, None]]` 替代（更繁瑣但相容）

### 10.2 風險 2：test fixture 的 tmp_path 隔離

**風險**：多個測試並行時，tmp_path 可能互相干擾。

**緩解**：
- pytest 預設為每個測試提供獨立的 tmp_path
- 若有干擾，使用 `scope="function"` 明確 fixture 範圍

### 10.3 風險 3：logger 為 None 時的 AttributeError

**風險**：使用 `logger.error()` 前忘記檢查 `if logger:`。

**緩解**：所有日誌操作都在 `if logger:` 保護下，確保靜默模式安全。
