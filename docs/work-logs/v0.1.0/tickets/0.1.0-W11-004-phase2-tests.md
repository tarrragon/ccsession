## 測試策略概述

根據 Phase 1 功能設計規格，`ticket version-shift` 命令涉及以下核心操作：

1. **前置驗證** (Step 0) — 參數格式、版本衝突檢查
2. **備份操作** (Step 1) — 整目錄備份至 `.claude/migration-backups/`
3. **Ticket 版本更新** (Step 2-3) — 遍歷所有 Ticket，更新 ID、version 欄位、內部交叉引用
4. **跨版本引用更新** (Step 4) — 掃描其他版本的 Ticket，更新指向已遷移 Ticket 的引用
5. **目錄重新命名** (Step 5) — 重命名 worklog 目錄
6. **todolist.yaml 更新** (Step 6) — 更新版本記錄和技術債務元資料
7. **摘要輸出** (Step 7) — 結構化輸出操作結果

### 測試分層策略

遵循「Sociable Unit Tests」原則，測試目標是**模組行為**（端對端），而非內部實作細節：

| 層級 | 測試對象 | Mock 策略 |
|-----|---------|---------|
| Unit | 個別函式（frontmatter 解析、版本號標準化、引用替換邏輯） | 使用真實 YAML 資料，不 Mock I/O |
| Integration | 完整 version-shift 工作流（各 Step 的銜接） | Mock 檔案系統（使用 `tmp_path`），不 Mock 業務邏輯 |
| E2E (簡化) | 實際檔案與目錄操作（驗收條件） | 無 Mock，使用隔離臨時目錄 |

### 測試數據準備

所有測試使用 pytest 的 `tmp_path` fixture 建立隔離的測試目錄結構，確保測試之間無副作用。

**測試目錄結構模板**：

```
tmp_path/
├── docs/
│   ├── work-logs/
│   │   ├── v0.1.0/
│   │   │   ├── tickets/
│   │   │   │   ├── 0.1.0-W1-001.md
│   │   │   │   ├── 0.1.0-W1-002.md
│   │   │   │   ├── 0.1.0-W1-005-phase2-design.md (附屬文件)
│   │   │   │   └── ...
│   │   │   └── phase4-evaluation-report.md
│   │   └── v0.2.0/ (某些邊界條件測試用)
│   └── todolist.yaml
├── .claude/
│   └── migration-backups/
└── pubspec.yaml (標記專案根目錄)
```

## 最複雜的 3 個測試案例摘要

### 1. **I.3 Ticket 內部交叉引用更新**

**複雜度原因**：
- 涉及 6 種不同的欄位類型（blockedBy、parent_id、children、chain.root、chain.parent、relatedTo 等）

- 涉及相互依賴的 Ticket（A → B → C 的依賴鏈），需要驗證全鏈的一致性更新

**關鍵測試點**：
- 列表型欄位 `blockedBy: ["0.1.0-W1-002"]` 中的元素替換
- 字串型欄位 `parent_id: "0.1.0-W1-001"` 的替換
- 巢狀結構 `children: ["0.1.0-W2-002", "0.1.0-W2-003"]` 的逐元素替換
- chain 結構中多欄位的同時更新

### 2. **I.5 跨版本交叉引用更新 (Step 4)**

**複雜度原因**：
- 需要掃描多個版本目錄（v0.2.0、v0.3.0 等），避免重複遍歷已更新的版本
- 跨版本檔案修改時需要驗證去重邏輯：同一檔案不被更新兩次
- 需要測試邊界情況：部分版本不存在時的優雅處理

**關鍵測試點**：
- 正確識別「其他版本」的 Ticket（排除 to_version）
- 版本掃描的順序和完整性
- 重複引用的避免（例如同一 Ticket 中多個欄位都包含舊版本 ID）
- 交叉版本邊界的驗證（確保 v0.2.0 內的引用被正確更新）

---

### 3. **I.6 todolist.yaml 更新**

**複雜度原因**：
- YAML 多層結構操作（current_version、previous_version、tech_debt 陣列）
- 條件性更新邏輯（只有在值匹配時才更新）
- 版本號格式的差異性（tech_debt.source_version 有 v 前綴，source_ticket 無）
- 需要驗證 YAML 格式的完整性（不破壞其他欄位、保留縮排、保留註解等）

**關鍵測試點**：
- 多個欄位同時更新時的一致性
- source_version 和 source_ticket 欄位的格式差異処理（v 前綴）
- 部分欄位匹配、部分不匹配的場景
- YAML 序列化時的格式一致性驗證

## 已知測試困難點與解決方案

### 困難點 1：時間戳唯一性驗證

**問題**：測試可能在同一秒內執行多次，導致時間戳相同
**解決方案**：Mock `datetime.now()` 或使用遞增計數器作為備份目錄後綴

### 困難點 2：Platform 相關的權限模擬

**問題**：`--no-backup` 旗標測試和備份失敗模擬在不同平台上行為不同
**解決方案**：使用 `unittest.mock.patch` 模擬檔案系統 I/O 錯誤

### 困難點 3：YAML 格式一致性驗證

**問題**：YAML 讀寫可能改變原始檔案的格式（縮排、註解、順序）
**解決方案**：讀寫後使用 `yaml.safe_load()` 比對資料結構，而非文本比對
