# Execution Log

## Task Summary

建立 並行派發後自動化驗證機制 — git diff 比對代理人報告 vs 實際變更

---

## Problem Analysis

**IMP-031 揭露的問題**：並行派發場景下，代理人可能報告任務已完成，但 `git diff --stat` 結果顯示部分預期修改的檔案實際未變更。

**現有防線分析**：
- 手動驗證：PM 收到代理人回報後，手動執行 `git diff --stat` 並比對 Ticket 的 `where.files` 清單
- 問題：此防線依賴人工執行，容易被忽略或遺漏（PC-010 案例顯示 Checkpoint 流程跳過率偏高）

**核心需求**：自動化比對，在並行代理人回報完成後自動驗證 where.files 與實際變更的一致性

**設計約束**：
- 保守策略：只驗證+警告，不阻擋工作流（exit code 永遠 0）
- 適用場景：所有 ticket complete 操作，非僅限於並行派發
- 技術選擇：PostToolUse (Bash) Hook，監聽 `ticket track complete` 成功信號

## Solution

### Phase 1 功能規格設計（完成）

**設計文件**：`docs/work-logs/v0.1.0/tickets/0.1.0-W41-002-phase1-feature-spec.md`

#### 核心設計決策摘要

**觸發機制**：PostToolUse (Bash)，監聽 `ticket track complete` 成功執行

**觸發範圍**：所有 ticket complete（非僅並行場景），保守策略避免漏報

**比對邏輯**：
- 預期集合：Ticket `where.files` 清單
- 實際集合：`git diff HEAD --name-only`
- 只警告「預期中但缺失的檔案」（missing = where_files - git_changed）

**設計哲學**：只驗證+警告，永遠 exit 0，不阻擋工作流

#### 主要函式清單

| 函式 | 用途 |
|------|------|
| `is_ticket_complete_success(input_data)` | 判斷觸發條件 |
| `extract_ticket_id(input_data)` | 從命令/輸出提取 Ticket ID |
| `read_ticket_where_files(ticket_id, project_root)` | 讀取 Ticket where.files |
| `get_git_changed_files(project_root)` | 執行 git diff HEAD 取得變更清單 |
| `find_missing_files(where_files, git_changed)` | 比對找出缺失檔案 |
| `normalize_path(path)` | 路徑規範化（移除 ./、統一正斜線） |
| `format_warning_message(...)` | 生成結構化警告訊息 |

#### 驗收標準（精化後 7 個 AC + 3 個 NAC）

AC-1: 有缺失時 additionalContext 含 `[並行派發驗證警告]`
AC-2: 全部匹配時靜默通過（無 additionalContext）
AC-3: 警告列出所有缺失檔案名稱
AC-4: where.files 為空時靜默跳過
AC-5: git 失敗時 exit 0，不阻擋
AC-6: 警告同時寫入日誌檔
AC-7: settings.json 新增 PostToolUse > Bash Hook 設定
