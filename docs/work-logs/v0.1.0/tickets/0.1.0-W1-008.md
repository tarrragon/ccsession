# Execution Log

## Task Summary

釐清 UC-007 子事件重組設計決策（後端組裝 vs 前端重組）

---

## Problem Analysis

### 問題背景

Claude Code 的 assistant message 的 `content` 欄位是一個陣列，可能包含多個不同類型的元素：
- `text`: 文字回應
- `tool_use`: 工具呼叫
- `tool_result`: 工具執行結果
- `thinking`: 思考過程

當一個 JSONL 行的 content array 包含多個元素時（例如 text + tool_use + text），需要決定如何組裝這些子事件。

### 當前設計問題

W1-006 Linux 視角審查指出：UC-007 的子事件前端重組是「補丁思維」。
- 當前方案（前端重組）：Frontend 收到所有子事件後自行判斷哪些屬於同一 message
- 被批評原因：職責邊界模糊，看起來像臨時解決方案而非永久架構

### 待評估的三種方案

1. **後端組裝**：Go Backend 將同一 message 的所有 content items 組裝成一個完整的 SessionEvent
2. **Boundary 信號**：Backend 推送子事件時附加 message boundary 信號（isLastContent），Frontend 據此重組
3. **前端重組**（現有方案）：Frontend 收到所有子事件後自行判斷哪些屬於同一 message

---

## Solution

### 三種方案詳細對比

#### 方案 1：後端組裝

**設計**：Backend 在解析 assistant message 時，將 content array 的所有元素組裝成一個完整的 SessionEvent。

**優點**：清晰的職責分離、語義完整、前端實作簡單。

**缺點**：Backend 複雜度大幅上升、即時性延遲（需等待 content array 完整）、不支援 streaming、重構成本高、與 spec.md v1.2.0 新增欄位邏輯衝突。

#### 方案 2：Boundary 信號（推薦）

**設計**：Backend 仍然推送子事件，但在最後一個子事件上設置 `isLastContent: true` 信號。Frontend 據此進行重組。

**優點**：
- 保持即時性，無延遲
- 支援 streaming（邊接收邊呈現）
- 最小化結構改動（只需新增一個布林欄位）
- 清晰的邊界信號，解決「補丁思維」疑慮
- 向後相容，容錯性好

**缺點**：Frontend 仍需重組邏輯、邊界信號可靠性取決於信號傳遞。

#### 方案 3：前端重組（現有方案）

**設計**：Frontend 根據 messageId 和 contentIndex 自行猜測和重組。

**優點**：Backend 最簡單、無邊界依賴、容錯性優秀。

**缺點**：Frontend 複雜度高、職責模糊（「補丁思維」）、與完整 message 語義距離遠。

### 方案對比表

| 考量維度 | 後端組裝 | Boundary 信號 | 前端重組 |
|--------|--------|------------|--------|
| **即時性** | ✗ 有延遲 | ✓ 無延遲 | ✓ 無延遲 |
| **Streaming** | ✗ 不支援 | ✓ 支援 | ✓ 支援 |
| **Backend 複雜度** | ✗ 高 | ✓ 低 | ✓ 最低 |
| **Frontend 複雜度** | ✓ 最低 | ○ 中 | ○ 中-高 |
| **結構改動** | ✗ 大改 | ✓ 最小 | ✓ 無改 |
| **向後相容** | ✗ 破壞 | ✓ 保持 | ✓ 保持 |
| **職責清晰度** | ✓ 最清晰 | ○ 清晰 | ✗ 模糊 |

### 設計決策：選擇方案 2（Boundary 信號）

**決策理由**：

1. **最佳的平衡點**：Backend 簡單、Frontend 邏輯清晰、同時支援即時性和 streaming

2. **符合 MVP 進度**：影響範圍最小，不阻塞其他任務

3. **解決 Linux 視角的疑慮**：Backend 層明確定義邊界，而非 Frontend 被迫猜測

4. **向前相容和升級路徑**：無損降級、便於未來升級

5. **易於測試和除錯**：邊界信號可追蹤，超時機制可容錯
