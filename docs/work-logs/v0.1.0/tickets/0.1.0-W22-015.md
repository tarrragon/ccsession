## 現況分析

### 單一 Ticket 週期 AskUserQuestion 觸發地圖

典型 IMP Ticket（1 次 commit）必定觸發的場景：

| 觸發點 | 場景 | Hook | 每週期次數 |
|--------|------|------|-----------|
| ticket complete 前 | #1 驗收方式確認 | acceptance-gate | 1 次 |
| ticket complete 後 | #2 complete 後下一步 | acceptance-gate | 1 次 |
| git commit 後 | #16 錯誤學習確認 | commit-handoff | 每次 commit |
| git commit 後 | #11 Handoff 確認 | commit-handoff | 每次 commit |

最小觸發：#1 + #16 + #11 + #2 = 4 次（單一 commit）
多次 commit：4 + (#16 + #11) × (commit 數 - 1)

### 場景 #1 問題分析

**現況**：acceptance-gate-hook 已讀取 type 和 priority 欄位，但 #1 提醒對所有 ticket 類型無差別觸發。



Hook 已有 is_doc_type() 判斷（用於豁免驗收記錄檢查），但此判斷未用於抑制 #1 提醒。

**核心問題**：驗收方式可由 type + priority 欄位自動決定，不需要每次詢問。

### 場景 #16 問題分析

**現況**：commit-handoff-hook 對每次 git commit 無條件觸發，不考慮 commit 內容性質。

三個選項設計缺陷：
- 「稍後記錄」（稀後記錄）：承諾虛假 — 實際上幾乎不會被執行，但佔用使用者決策頻寬
- 「記錄錯誤學習經驗」：觸發成本高（進入 /error-pattern add 流程）
- 結果：使用者習慣性選「無需記錄」，#16 成為形式性干擾

**兩個結構性問題**：
1. 缺乏 commit message 語義分析（docs:/chore:/style: 通常不含錯誤學習）
2. 三選一 vs 二元（Yes/No）—「稍後記錄」是偽選項

## 優化方案

### 方案一：#1 依 task_type + priority 自動決定驗收方式

**決策矩陣**：

| 條件 | 自動選擇 | 是否觸發 #1 提醒 |
|------|---------|----------------|
| type = DOC | 簡化驗收 | 否（自動） |
| type = ANA | 簡化驗收 | 否（自動） |
| priority = P0 | 顯示 2 選項 | 是（2 選項：標準驗收 / 先完成後補）|
| 其他（IMP/ADJ/TST + 非 P0） | 標準驗收 | 否（自動） |
| type 未知/空值 | 標準驗收 | 否（保守預設） |

**預計影響**：95%+ 的 Ticket 不再需要 #1 互動（P0 屬罕見例外）。

**實作範圍**：
1. acceptance-gate-hook.py — 在 AcceptanceCheckResult 加入 ticket_type 和 priority 欄位，build_hook_output() 加入自動決定邏輯
2. askuserquestion-rules.md — 記錄自動決定規則
3. askuserquestion-scene-details.md — 更新 #1 場景說明

---

### 方案二：#16 簡化為二元確認 + commit message 語義過濾

**選項 A（最小變更）：移除「稍後記錄」虛假選項，改為二元**



**選項 B（更激進）：新增 commit message 語義過濾**

| commit message 前綴 | #16 行為 |
|--------------------|---------|
| docs:, chore:, style:, refactor:, test: | 自動跳過（非錯誤相關） |
| fix:, bug:, patch: | 強制觸發（錯誤學習機會高） |
| 其他（feat:, 無前綴等） | 觸發二元確認 |

**推薦實作**：選項 A + 選項 B 組合

**實作範圍**：
1. commit-handoff-hook.py — 解析 commit message，加入前綴過濾邏輯
2. hook_messages.py — 新增二元 #16 訊息模板
3. askuserquestion-scene-details.md — 更新 #16 場景說明

### 預計觸發次數改善

| Ticket 場景 | 優化前 | 優化後 |
|-------------|--------|--------|
| DOC ticket, 1 次 docs: commit | 4 次 | 2 次（#11 + #2）|
| IMP ticket, 1 次 docs: commit | 4 次 | 2 次（#11 + #2）|

| IMP ticket, 3 次 commit（各類型） | 8 次 | 4-5 次 |

整體降幅：25-50%。P0 緊急 Ticket 無影響。
