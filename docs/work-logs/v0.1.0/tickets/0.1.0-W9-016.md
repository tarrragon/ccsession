# Execution Log

## Task Summary

研究並實作 context 用量感知機制：調查 Claude Code 是否暴露 token 使用量，建立 Hook 在 context 達到閾值時主動建議 handoff；同時調整 AskUserQuestion #11 預設路由為 handoff-first

---

## Problem Analysis

### 研究結論：Claude Code 不暴露 token 用量

調查 Claude Code hook 事件結構（UserPromptSubmit / PreToolUse / PostToolUse / Stop），
確認這些事件的 JSON payload 均不包含 token 使用量或 context window 剩餘量。

**代理指標選擇**：使用「session 內完成的 Ticket 數量」作為 context 累積的代理指標。
- 每個 `ticket track complete` 成功 = 一個完整的工作單元完成
- 3 個以上 Ticket 在同一 session → context 累積風險顯著

**參考**：`/strategic-compact` skill 使用「tool call 次數」作為類似的代理指標。

### 現有問題分析（PC-009）

- `COMMIT_HANDOFF_REMINDER` 訊息中「情境 A/B」描述未明確強調 handoff 優先
- `askuserquestion-rules.md` #11 選項表中雖然 Handoff 已是 Recommended，但缺少原則說明
- 沒有主動偵測 context 累積並警告的機制

---

## Solution

### 實作項目

1. **新建 `session-context-guard-hook.py`（PostToolUse/Bash）**
   - 偵測：`ticket track complete` 命令 + stdout 包含 "已完成"
   - 計數器：`/tmp/claude-session-completed-tickets-{ppid}`
   - 閾值 2：輕度提醒
   - 閾值 3+：強烈建議 handoff

2. **更新 `COMMIT_HANDOFF_REMINDER`（hook_messages.py）**
   - 新增 `[核心原則 - PC-009]` 段落
   - 情境 A/B 說明明確標注 Handoff 為 Recommended，繼續是例外

3. **更新 `askuserquestion-rules.md` #11 章節**
   - 新增「Handoff first（PC-009）」核心原則區塊
   - 補充第 3 條共通規則：Handoff 選項必須排第一

4. **更新 `settings.json`**
   - 新增 session-context-guard-hook.py 至 PostToolUse/Bash
