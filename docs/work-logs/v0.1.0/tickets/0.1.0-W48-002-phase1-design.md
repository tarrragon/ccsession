# Phase 1 功能設計：場景 #8 Hook 覆蓋

## 1. 功能需求分析

### 1.1 核心問題

場景 #8（執行方向確認）目前在 `askuserquestion-rules.md` 的 Hook 提醒欄位標記為 `-`（無覆蓋）。這意味著當 PM 向用戶詢問「先做哪個任務」「任務執行順序如何」等問題時，系統不會自動提醒使用 AskUserQuestion 工具，依賴 PM 的記憶觸發。

### 1.2 場景 #7 vs #8 的核心區別

| 維度 | 場景 #7（派發方式選擇） | 場景 #8（執行方向確認） |
|------|----------------------|----------------------|
| 核心問題 | 用什麼機制派發？ | 任務執行順序？ |
| 決策對象 | subagent / Agent Teams / 序列派發 | 先做哪個 / 哪些並行 / 哪些序列 |
| 典型語言 | 「要用 agent team 嗎」「並行派發還是序列」 | 「先做哪個任務」「接下來做什麼」「執行順序是？」 |
| 現有 Hook | askuserquestion-reminder-hook | 無（本 Ticket 目標） |

### 1.3 使用者場景

| 場景 | 用戶輸入示例 | 預期行為 |
|------|------------|---------|
| 詢問執行順序 | 「接下來先做哪個比較好」 | 提醒 PM 使用 AskUserQuestion |
| 任務優先排序 | 「這幾個任務要按什麼順序做」 | 提醒 PM 使用 AskUserQuestion |
| 並行或序列選擇 | 「這些任務要先後做還是同時做」 | 提醒 PM 使用 AskUserQuestion |
| 下一步方向 | 「接下來要做什麼」 | 提醒 PM 使用 AskUserQuestion |

## 2. 功能規格設計

### 2.1 輸入定義

觸發條件：用戶的 prompt（小寫化後）同時包含以下任一關鍵字組合。

### 2.2 輸出規格

當 `check_decision_keywords()` 匹配到場景 #8 的關鍵字時，回傳場景名稱字串 `"執行方向確認"`，由 `check_skill_suggestion()` 包裝成 `AskUserQuestionMessages.DECISION_REMINDER.format(scenario="執行方向確認")` 輸出至 stdout。

### 2.3 正常流程

1. 用戶輸入 prompt，Hook 觸發
2. `check_skill_suggestion()` 在第五優先級呼叫 `check_decision_keywords()`
3. `check_decision_keywords()` 掃描 `DECISION_KEYWORDS` 字典
4. 若匹配到場景 #8 關鍵字組合，回傳 `"執行方向確認"`
5. `check_skill_suggestion()` 格式化 `DECISION_REMINDER` 並回傳
6. `main()` 輸出提醒訊息

### 2.4 錯誤處理

- 否定詞語境（如「不需要確認執行順序」）：`_is_keyword_negated()` 過濾，不觸發提醒
- 邊界情況：只有一個關鍵字存在時（組合關鍵字要求兩者都出現）不觸發

## 3. 邊界條件分析

### 3.1 與場景 #7 的關鍵字重疊風險

現有 #7 關鍵字：`("並行", "派發")`、`("agent team",)`

場景 #8 目標關鍵字必須確保不與上述重疊。

重疊風險分析：
- `"並行"` 單獨存在可能同時描述「派發方式」(#7) 和「執行順序」(#8)
- 解決方案：#8 的關鍵字組合選擇不含 `"並行"+"派發"` 的組合，改用更具體的執行順序語義

### 3.2 與場景 #5（優先級確認）的重疊風險

現有 #5 關鍵字：`("優先", "處理")`、`("先做", "哪個")`

分析：`("先做", "哪個")` 目前歸類為 `"優先級確認"`（#5），語義與 #8 有重疊。

設計決策：`("先做", "哪個")` 保持歸屬 #5（優先級），#8 的新關鍵字專注於「執行順序/方向」語義，避免重複。

### 3.3 誤觸發防護

高誤觸發風險組合（應避免）：
- `("做", "什麼")`：過於通用，大量日常語句會匹配
- `("先",)`：單詞過短，誤觸發率高
- `("接下來",)`：單一關鍵字，與日常詢問難以區分

設計原則：使用雙關鍵字組合，確保語義特異性。

## 4. API 介面設計

### 4.1 新增 DECISION_KEYWORDS 條目

**位置**：`prompt-submit-hook.py` 第 154-164 行，`DECISION_KEYWORDS` 字典

**新增條目設計**：

```python
# 場景 #8：執行方向確認（執行順序/先後/並行安排）
("順序", "任務"): "執行方向確認",       # 「任務順序怎麼排」「按什麼順序做任務」
("接下來", "做"): "執行方向確認",        # 「接下來要做什麼」「接下來先做哪個」
("先後", "執行"): "執行方向確認",        # 「先後執行順序」「先後執行哪些」
```

**設計理由**：

| 關鍵字組合 | 覆蓋語義 | 典型觸發句 | 誤觸發風險 |
|------------|---------|------------|------------|
| `("順序", "任務")` | 任務排序詢問 | 「任務順序如何」「任務的執行順序是什麼」 | 低（需同時含「順序」+「任務」） |
| `("接下來", "做")` | 下一步方向 | 「接下來要做什麼」「接下來先做哪個好」 | 低（需同時含「接下來」+「做」） |
| `("先後", "執行")` | 執行先後安排 | 「先後執行的順序」「哪些要先後執行」 | 極低（較為特定的組合） |

**與現有關鍵字的差異化確認**：

| 組合 | 是否與 #4-#7 重疊 | 說明 |
|------|-------------------|------|
| `("順序", "任務")` | 否 | 現有無此組合 |
| `("接下來", "做")` | 否 | 現有無此組合；`("先做","哪個")` 是 #5 但組合不同 |
| `("先後", "執行")` | 否 | 現有無此組合 |

### 4.2 場景名稱字串

回傳值：`"執行方向確認"`

此字串透過 `AskUserQuestionMessages.DECISION_REMINDER.format(scenario="執行方向確認")` 格式化，輸出提醒：

```
============================================================
[AskUserQuestion 提醒] 偵測到執行方向確認
============================================================

此場景建議使用 AskUserQuestion 工具而非文字提問，
避免用戶回答被 Hook 系統誤判為開發命令。

提示: ToolSearch("select:AskUserQuestion") 載入後使用。
詳見: .claude/rules/core/decision-tree.md
============================================================
```

### 4.3 hook_messages.py 評估

**結論：不需要新增訊息常數。**

`AskUserQuestionMessages.DECISION_REMINDER` 已使用 `{scenario}` 佔位符格式化輸出，僅需傳入 `"執行方向確認"` 字串即可，無需在 `hook_messages.py` 或 `ask_user_question_reminders.py` 新增常數。

### 4.4 askuserquestion-rules.md 更新

**修改位置**：場景表第 #8 行的 Hook 提醒欄

**修改前**：

```markdown
| 8 | 執行方向確認 | 並行/序列、先後順序 | 決策樹第負一層 | - |
```

**修改後**：

```markdown
| 8 | 執行方向確認 | 並行/序列、先後順序 | 決策樹第負一層 | prompt-submit-hook |
```

同步更新 Hook 提醒機制表格中 `prompt-submit-hook` 的覆蓋場景欄，從：

```
| prompt-submit-hook | 用戶提問含決策關鍵字 | #4 方案 + #5 優先級 + #6 拆分 |
```

更新為：

```
| prompt-submit-hook | 用戶提問含決策關鍵字 | #4 方案 + #5 優先級 + #6 拆分 + #8 執行方向 |
```

## 5. 驗收標準

### 5.1 功能正確性驗證

| 驗收條件 | 驗證方式 |
|---------|---------|
| DECISION_KEYWORDS 新增至少 3 組場景 #8 關鍵字 | 程式碼審查：計算 `"執行方向確認"` 值的條目數 |
| 三組關鍵字與現有 #4-#7 無重疊 | 逐一比對現有字典條目 |
| 偵測到場景 #8 類提問輸出 AskUserQuestion 提醒 | 執行 `check_decision_keywords("接下來要做什麼任務")` → 回傳 `"執行方向確認"` |
| askuserquestion-rules.md 場景 #8 Hook 提醒欄更新 | 文字比對：欄位從 `-` 變為 `prompt-submit-hook` |
| 所有測試通過（含新增 #8 測試案例） | `uv run pytest .claude/hooks/tests/` 全部 pass |

### 5.2 測試案例設計（供 Phase 2/3 使用）

**正向測試（應觸發）**：

```python
# 場景 #8 觸發案例
check_decision_keywords("這幾個任務的順序怎麼安排")  → "執行方向確認"
check_decision_keywords("接下來要做哪個任務比較好")   → "執行方向確認"
check_decision_keywords("先後執行的順序要怎麼決定")  → "執行方向確認"
```

**負向測試（不應觸發）**：

```python
# 否定語境
check_decision_keywords("不需要確認任務順序")    → None（否定詞過濾）
check_decision_keywords("無需考慮接下來怎麼做")  → None（否定詞過濾）

# 只有單一關鍵字
check_decision_keywords("任務很多")              → None（缺少「順序」）
check_decision_keywords("接下來呢")              → None（缺少「做」）

# 確保 #7 不被誤觸發為 #8
check_decision_keywords("要用並行派發嗎")         → "派發方式選擇"（#7，非 #8）
```

**與現有 #5 的邊界測試**：

```python
# ("先做", "哪個") 仍是 #5，不變
check_decision_keywords("先做哪個好")            → "優先級確認"（#5，非 #8）
```

### 5.3 邊界條件測試

```python
# 大小寫不影響（check_decision_keywords 接收小寫輸入）
check_decision_keywords("task 順序如何")          → None（缺少「任務」）

# 長文本仍正確偵測
check_decision_keywords("我想了解一下這幾個任務的執行順序應該如何安排比較好")
→ "執行方向確認"
```

## 6. 行為場景提取（Given-When-Then）

**場景 1：正常流程 - 任務順序詢問**
- Given：PM 想詢問用戶任務執行順序
- When：用戶輸入含「順序」+「任務」的文字
- Then：Hook 輸出場景 #8 的 AskUserQuestion 使用提醒

**場景 2：正常流程 - 下一步方向詢問**
- Given：PM 想詢問接下來執行方向
- When：用戶輸入含「接下來」+「做」的文字
- Then：Hook 輸出場景 #8 的 AskUserQuestion 使用提醒

**場景 3：異常流程 - 否定語境**
- Given：用戶表達不需要確認順序
- When：用戶輸入「不需要考慮任務順序了」
- Then：`_is_keyword_negated()` 過濾「順序」，Hook 不輸出提醒

**場景 4：邊界條件 - 只有單一關鍵字**
- Given：用戶句子只含組合關鍵字的其中一個詞
- When：用戶輸入「接下來呢？」（無「做」）
- Then：雙關鍵字要求未滿足，Hook 不觸發

**場景 5：邊界條件 - 與 #7 的優先級**
- Given：用戶同時觸發 #7 和 #8 關鍵字（不可能，因為關鍵字互不重疊）
- When：所有 DECISION_KEYWORDS 依序掃描
- Then：先匹配的場景優先回傳（字典迭代順序）

**設計決策記錄**：

1. 選擇 3 組雙關鍵字組合而非單一關鍵字：降低誤觸發率，提高語義特異性
2. 不修改 `("先做", "哪個")` 的歸屬（保留 #5）：雖有語義重疊，但「先做哪個」更偏優先級選擇，「執行順序」更偏安排規劃
3. 不新增 hook_messages.py 常數：現有 `DECISION_REMINDER` 的 `{scenario}` 佔位符已足夠，增加常數反而增加維護負擔
4. `"接下來"+"做"` 組合的選擇：「接下來」是方向性詞彙，搭配「做」限縮為行動指示語境，避免純粹「接下來」的過度觸發
