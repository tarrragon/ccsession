## Problem Analysis

### 4x 過濾邏輯分析
在 `execute_agent()` 函式中發現的重複邏輯：
1. 第一次遍歷：過濾出代理人相關的 Ticket（340-351 行）
2. 初始化狀態分組字典（354-359 行）
3. 第二次遍歷：按狀態分組（360-367 行）
4. 提取各狀態的列表（364-367 行，實際上這只是變數指派）

**原始程式碼問題**：agent_tickets 先建立為單獨列表，然後遍歷再分組，涉及兩次循環遍歷。

### Ticket 驗證邏輯重複分析

搜尋指令：`grep -n "if not.*ticket\|load_ticket" ticket_system/commands/track_relations.py`

確認的位置（已確認）：
- track_relations.py:138（_execute_set_relation_field 中驗證目標 Ticket）
- track_relations.py:151（_execute_set_relation_field 中驗證被引用 Ticket）
- track_relations.py:238（execute_add_child 中驗證父 Ticket）
- track_relations.py:244（execute_add_child 中驗證子 Ticket）
- track_relations.py:302（execute_phase 中驗證 Ticket）

所有位置都使用相同模式：`load_ticket() + if not ticket: print error + return 1`

---

## Solution

### 1. 合併 execute_agent 的 4x 過濾邏輯為單次遍歷

**改進策略**：
- 移除獨立的 agent_tickets 列表和第二次遍歷
- 在第一次遍歷時直接按狀態分組到 status_groups
- 最後從 status_groups 中重建 agent_tickets（供計數使用）

**程式碼變更位置**：
- `execute_agent()` 函式（326-367 行）
- 從兩次遍歷減少為一次遍歷，同時初始化狀態分組

### 2. 提取 Ticket 驗證邏輯為共用函式

**新增函式**：`validate_ticket_exists(version: str, ticket_id: str) -> tuple[dict | None, bool]`

**設計特點**：
- 回傳 tuple（ticket, success），避免呼叫者重複 load_ticket()
- 簡化所有驗證點的程式碼
- 統一錯誤訊息輸出

**應用位置**：
1. `_execute_set_relation_field()`：驗證目標 Ticket 和被引用 Ticket
2. `execute_add_child()`：驗證父 Ticket 和子 Ticket
3. `execute_phase()`：驗證 Ticket

**程式碼變更前後**：
```python
# 舊
target_ticket = load_ticket(version, target_id)
if not target_ticket:
    print(format_error(...))
    return 1

# 新
target_ticket, success = validate_ticket_exists(version, target_id)
if not success:
    return 1
```
