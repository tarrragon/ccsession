新增 post-ticket-completion-askuserquestion Hook（偵測 Ticket complete 後的純文字問句）

## Problem Analysis

### 問題背景

根據 W15-001 分析，AskUserQuestion 現有 12/17 場景已有 Hook 覆蓋，但場景 #2（Complete 後下一步選擇）仍缺乏 Hook 提醒。

**觀察情況**：
- PM 在 `ticket track complete` 執行成功後，以純文字詢問「要繼續執行下一個 Ticket 嗎？」
- 此場景應使用 AskUserQuestion tool（避免用戶回答被誤判為開發命令）
- 現有 acceptance-gate-hook.py 在 PreToolUse 階段僅提醒驗收方式（場景 #1），未涵蓋完成後下一步引導

### 分析決策

評估兩個實作方案：

| 維度 | 選項 A：擴充 acceptance-gate-hook.py | 選項 B：新建 wave-next-step-guard-hook.py |
|------|------|------|
| 改動規模 | 修改 1 個檔案 + 訊息檔 | 新建 1 個 Hook + settings.json |
| 單一職責 | acceptance-gate 負責整個 complete 流程 | 責任切割不清，分散 |
| 維護成本 | 低（集中一個檔案） | 中（分散多個檔案） |
| 實現成本 | 低（5 行程式碼改動） | 中（100+ 行新代碼） |

**決策：選項 A（擴充 acceptance-gate-hook.py）**

**正當性**：
- acceptance-gate-hook 的核心職責是驗證和引導 `ticket track complete` 的完整生命週期
- 擴充職責到「complete 流程完整引導」（包含驗收方式 #1 + 完成後下一步 #2）是自然延伸
- 不引入新的單一職責，而是加深同一職責的覆蓋深度
- 符合 DRY 原則，避免重複的 Ticket ID 提取邏輯

| `.claude/hooks/lib/hook_messages.py` | 新增 `COMPLETE_NEXT_STEP_REMINDER` | 場景 #2 的標準提醒訊息 |

## Solution

### 實作內容

#### 1. acceptance-gate-hook.py 修改

**修改 generate_hook_output()** 簽名和邏輯：
```python
def generate_hook_output(
    should_block: bool,
    message: Optional[str],
    ask_user_reminder: bool = False,
    next_step_reminder: bool = False
) -> Dict[str, Any]:
```

- 新增 `next_step_reminder` 參數
- 調整邏輯：
  - message（錯誤/警告）優先級最高
  - ask_user_reminder（場景 #1 驗收方式）次之
  - next_step_reminder（場景 #2 完成後下一步）最後（僅在無訊息時追加）

**修改 main()** 決策邏輯：
```python
next_step_prompt = not should_block and not message
output = generate_hook_output(
    should_block,
    message,
    ask_user_reminder=True,
    next_step_reminder=next_step_prompt
)
```

- 當檢查全部通過（無阻塊、無警告）時，追加場景 #2 提醒
- 優先級排序確保不同情況下呈現適當訊息

#### 2. hook_messages.py 新增常數

**新增 COMPLETE_NEXT_STEP_REMINDER**：
- 覆蓋場景 #2（Complete 後下一步選擇）
- 說明四個可能的下一步選項：
  1. 繼續執行下一個 Ticket
  2. Wave 收尾確認
  3. 版本發布檢查
  4. 清空 Session 結束
- 引用規則來源：askuserquestion-rules.md 場景 #2
- 包含 ToolSearch 載入提示和規則連結

#### 3. Hook 描述更新

更新頂部註釋，說明：
- 完整的 Hook 職責（驗收檢查 + 流程引導）
- 覆蓋場景 #1 和 #2
- 補充說明時序（PreToolUse 階段提醒涵蓋 complete 前後流程）

### 訊息優先級設計

```
場景順序（PreToolUse 階段）：

1. 如有 should_block=true（子任務未完成）
   → 阻塊訊息 + 場景 #1 提醒

2. 如有 should_block=false 但有警告（未驗收）
   → 警告訊息 + 場景 #1 提醒

3. 如全部通過（無阻塊、無警告）
   → 場景 #1 提醒 + 場景 #2 提醒（完整的 complete 流程引導）
```
