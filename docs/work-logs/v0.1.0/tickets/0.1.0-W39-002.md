## Phase 1 功能規格設計

### 1. 功能需求分析

#### 1.1 問題根源

**is_handoff_recovery_mode() 效能問題**：
- 現有實作每次呼叫都執行 glob 掃描（Path.glob）
- 觸發場景：多個 UserPromptSubmit Hook 在同一 session 內多次觸發
- 問題：glob 是 OS 系統呼叫，在高頻觸發場景重複掃描同一目錄，產生不必要的 I/O
- 目錄結構固定（.claude/handoff/pending/），同一進程內結果不會改變

**check_error_patterns_changed() 效能問題**：
- 現有實作每次呼叫執行 rglob 遞迴掃描，掃描 .claude/error-patterns/ 下所有子目錄
- 觸發場景：acceptance-gate-hook 在每次 ticket track complete 時執行
- 問題：rglob 遍歷整個目錄樹，隨 error-patterns 數量增長，效能線性下降
- 漸進掃描快取目標：快取每個檔案的 mtime，避免重複 stat() 系統呼叫

**手寫 YAML 解析器限制**：
- parse_ticket_frontmatter() 依賴約 200 行手寫解析邏輯（_parse_yaml_lines 等）
- 已知支援子集：頂層 key-value、多行字串（|, >）、簡單嵌套、簡單列表
- 潛在風險：邊界情況未完整測試，特殊字元處理不全，維護成本高
- PyYAML 是 Python 標準生態的 YAML 函式庫，依賴方式需評估（inline script dependencies）

#### 1.2 使用者場景

場景 A: 每次用戶在 session 中提交 Prompt
- handoff-prompt-reminder-hook 觸發
- is_handoff_recovery_mode() 被呼叫
- 若有快取：直接返回，0 次 I/O
- 若無快取：執行一次 glob，快取結果

場景 B: 執行 ticket track complete
- acceptance-gate-hook 觸發
- check_error_patterns_changed() 被呼叫
- 漸進掃描：快取每個 .md 檔案的 mtime，避免重複 stat() 呼叫

### 2. API 介面設計

#### 2.1 is_handoff_recovery_mode() 快取機制

**快取策略**：Process-level 記憶體快取（模組級變數）

**快取生命週期**：同一 Python 進程內（即單次 Hook 執行），進程結束後自動失效

**失效條件**：進程結束（每次 Hook 觸發是獨立進程，快取自然失效）

**介面設計（hook_io.py）**：

模組級快取變數：
  _handoff_recovery_cache: Optional[bool] = None

函式簽名（不變，向後相容）：
  def is_handoff_recovery_mode(logger=None) -> bool

新增測試輔助函式：
  def clear_handoff_recovery_cache() -> None
    功能：清空 _handoff_recovery_cache 為 None
    用途：測試隔離（生產環境不呼叫）

**快取邏輯流程**：
  1. 呼叫 is_handoff_recovery_mode()
  2. 若 _handoff_recovery_cache is not None：直接返回快取結果（DEBUG log）
  3. 否則：執行現有 glob 掃描邏輯，計算結果
  4. 設定 _handoff_recovery_cache = 結果
  5. 返回結果

#### 2.2 check_error_patterns_changed() mtime 快取

**快取策略**：檔案 mtime 快取（模組級字典），以 file_path 字串為 key

**快取生命週期**：同一 Python 進程內

**快取的資料**：每個 .md 檔案的 st_mtime 值（float）

**介面設計（hook_ticket.py）**：

模組級快取變數：
  _error_pattern_mtime_cache: Dict[str, float] = {}

函式簽名（不變，向後相容）：
  def check_error_patterns_changed(project_root, ticket_created, logger=None) -> Tuple[bool, List[str]]

新增測試輔助函式：
  def clear_error_pattern_mtime_cache() -> None
    功能：清空 _error_pattern_mtime_cache
    用途：測試隔離（生產環境不呼叫）

**快取邏輯流程**：
  1. 呼叫 check_error_patterns_changed()
  2. 執行 error_patterns_dir.rglob("*.md")（目錄遍歷，保留）
  3. 對每個 file_path：
     a. 若 str(file_path) in _error_pattern_mtime_cache：使用快取的 mtime
     b. 否則：呼叫 file_path.stat().st_mtime，加入快取
  4. 比較 mtime > ticket_created_timestamp，收集 changed_files
  5. 返回 (has_changed, changed_files)

**注意**：rglob 目錄遍歷本身保留（不快取），快取的是每個檔案的 stat() 呼叫。

#### 2.3 PyYAML 替代評估標準

**評估面向**：

1. 功能等價性：
   - 測試所有現有 Ticket frontmatter 格式（頂層 key-value、多行字串、嵌套、列表）

   - 方法：timeit 模組，各執行 1000 次

   - Hook script 使用 uv inline script dependencies（/// script ... ///）

   - 方案：列出所有使用 parse_ticket_frontmatter() 的 Hook 檔案

4. 替換決策標準：

   - 建議保留條件：任一條件不滿足，或引入依賴的維護成本更高

## Test Case Design

### 測試策略概述

採用 **Sociable Unit Tests** 架構：
- 使用真實的檔案系統（通過 `tmp_path` 隔離）
- 使用真實的 logging 系統，不 mock logging
- Mock 外部依賴：檔案系統變化、時間、系統呼叫（glob/stat）
- 測試聚焦於模組行為而非內部實作

### 測試分層決策
- **快取測試**：單元測試（測試快取機制，mock glob/stat 呼叫計數）

- **邊界條件**：單元測試（空目錄、多檔案、時間邊界）
- **PyYAML 對比**：單元測試（功能等價性評估）

## Mock 設計策略

### 何時 Mock
| 依賴 | Mock 策略 | 理由 |
|------|---------|------|
| `Path.glob()` | Mock | 外部 OS 系統呼叫，快取測試的關鍵（驗證 call_count） |
| `Path.stat()` | Mock | 外部 OS 系統呼叫，mtime 快取的關鍵（驗證 call_count） |
| logging | 不 Mock | 使用真實 logging，便於偵錯 |
| 檔案讀寫 | 不 Mock | 使用 tmp_path 實際建立檔案 |

### 快取隔離策略
- 使用 `clear_caches` fixture 在每個測試後自動重設快取
- 確保測試間無狀態污染
