## 根因分析（三個維度）

### 維度 1：流程設計（60% 問題來源）

**檢查結果**：
- ✓ atomic-ticket-methodology.md 定義了「何時建立子任務」的清晰表格（4 種情境）
- ✓ 工具 `/ticket create --parent` 實現完整
- ✗ **缺口**：plan-to-ticket-flow.md「執行中額外發現」強制建 Ticket，但**未指明應建為子任務還是新 Wave Ticket**
- ✗ **缺口**：decision-tree.md 第三層強調「立即建立 Ticket 不詢問」，但未區分兩種建立方式
- ✗ **缺口**：ticket-lifecycle.md 建立後審核但無子任務相關指引

**核心問題**：當代理人在執行 Ticket 時遇到「相關的額外工作」（如 W27-001 的邏輯遺漏），流程指引會直接說「建立新 Ticket」，但**沒有決策分支判斷「是否應建為子任務」**。

**結果**：開發者傾向選擇「新建獨立 Ticket」因為這樣做的流程成本更低（不需判斷）。

### 維度 2：工具支援（5% 問題來源）

**檢查結果**：
- ✓ `/ticket create --parent {parent_id}` 命令存在且功能完整
- ✓ SKILL 定義正確支援子任務建立
- ✗ **可見性不足**：沒有文件或教程展示「實際何時使用 --parent」的場景
- ✗ **發現性低**：沒有提示機制告訴開發者「這個新 Ticket 應該建為子任務」

**核心問題**：工具本身沒有問題，但開發者可能不知道這個功能，或不知道何時應該使用。

**結果**：W25-W34 期間沒有任何人嘗試使用此功能。

### 維度 3：文件指引（35% 問題來源）

**檢查結果**：

子任務使用場景定義（atomic-ticket-methodology.md）：
```
1. TDD 階段任務 - 清晰
2. 問題衍生修復 - 清晰
3. 功能細分 - 清晰
4. 阻塞處理 - 清晰
```

但**判斷標準不清晰**：
- ✗ task-splitting.md 教「如何拆分」但未教「拆分後用子任務還是新 Wave Ticket」
- ✗ 無清晰的決策樹：「同一個功能的多個實施步驟」vs「獨立的新需求」的判斷方法
- ✗ W28-001 title 顯示 PM 意識到問題（"W27-001 遺漏"），但無相應的流程指引告訴開發者應建為子任務

**核心問題**：文件定義了子任務的「四種用途」，但沒有教「在實際執行中遇到問題時，何時應該建為子任務」。

## Solution

### 結論

**複合問題，以流程設計和文件指引為主**：

| 維度 | 權重 | 問題性質 |
|------|------|---------|
| 流程設計缺口 | 60% | 「執行中發現」時的決策指引不明確，缺少「子任務 vs 新 Wave」判斷分支 |
| 文件指引不足 | 35% | 子任務使用場景已定義，但實際判斷標準和決策流程缺失 |
| 工具不易發現 | 5% | 工具本身完整，但使用指引和示例不足 |

**不需要改變子任務機制本身**，只需改進：
1. 決策流程（在流程中補充「何時用子任務」的判斷點）
2. 文件指引（補充判斷標準和實例）
3. 教育示例（展示實際場景）

### 具體改善建議

#### 改善 1：補充 decision-tree.md（優先級：高）

**位置**：第三層命令處理流程

**新增內容**：
```
開發命令錯誤 → 涉及「補完相關工作」?
    ├─ 是 → 是「同一個功能的不同實施步驟」?
    │        ├─ 是 → 建子任務 /ticket create --parent {parent_id}
    │        │       例如：W27-001 遺漏的邏輯應建為 W27-001.1
    │        └─ 否 → 建新 Wave Ticket /ticket create --wave {wave+1}
    │
    └─ 否 → 新獨立 Ticket /ticket create --wave {wave}
```

**參考案例**：
- W28-001 應建為 W27-001.1（同一功能，補完邏輯）
- W29-002 應建為 W28-002.1（同一重構，遷移實施）

#### 改善 2：補充 plan-to-ticket-flow.md（優先級：高）

**位置**：「執行中額外發現」章節

**新增內容**：
```
執行中發現額外工作時：
┌─ 屬於當前 Ticket 的同一功能? 
│  ├─ 是 → /ticket create --parent {current_ticket}
│  │       例如：W27 實施結束後發現邏輯遺漏，建 W27-001.1
│  │
│  └─ 否 → /ticket create --wave {current_wave or next_wave}
│          根據優先級決定同 Wave 還是下個 Wave
│
└─ 判斷標準：同一「what」的不同「how」 → 子任務
           不同「what」 → 新 Wave Ticket
```

#### 改善 3：補充 task-splitting.md（優先級：中）

**位置**：決策樹末尾

**新增分支**：
```
任務拆分後 → 各子任務如何組織?
    ├─ TDD 各階段 → 全建為子任務（根為功能，子為各 Phase）
    ├─ 同功能不同步驟 → 全建為子任務（同一 Wave）
    ├─ 多個功能 → 建不同 Wave Ticket（Wave 獨立性原則）
    └─ 執行中發現額外工作 → 見 plan-to-ticket-flow.md 判斷標準
```
