## 3. 函式設計與虛擬碼

### 3.1 函式 1: `get_ticket_created_time()`

**職責**: 從 Ticket frontmatter 讀取 `created` 欄位，解析為 datetime 物件

**虛擬碼**:

```
函式 get_ticket_created_time(ticket_file: Path, logger) → datetime | None:

  前置條件:
    - ticket_file 存在（呼叫前驗證）
    - logger 物件可用

  邏輯流程:
    Step 1: 嘗試讀取檔案內容
      IF 讀取失敗:
        logger.warning("無法讀取 ticket 檔案")
        RETURN None

    Step 2: 呼叫 parse_ticket_frontmatter() 解析 frontmatter
      frontmatter = parse_ticket_frontmatter(content)
      IF frontmatter 為空:
        logger.warning("frontmatter 為空")
        RETURN None

    Step 3: 取得 created 欄位
      created_str = frontmatter.get("created", "").strip()
      IF created_str 為空:
        logger.warning("frontmatter 缺少 created 欄位")
        RETURN None

    Step 4: 嘗試多種格式解析日期
      嘗試格式 1: ISO 8601 → datetime.fromisoformat(created_str)
      IF 成功:
        logger.info(f"Ticket created at: {datetime_object}")
        RETURN datetime_object

      嘗試格式 2: 簡單日期 (YYYY-MM-DD) → datetime.strptime(created_str, "%Y-%m-%d")
      IF 成功:
        logger.info(f"Ticket created at: {datetime_object}")
        RETURN datetime_object

      嘗試格式 3: RFC 3339 含時區 → dateutil.parser.isoparse(created_str)
      IF 成功:
        logger.info(f"Ticket created at: {datetime_object}")
        RETURN datetime_object

      所有格式都失敗:
        logger.warning(f"無法解析 created 欄位: {created_str}")
        RETURN None

  後置條件:
    - 返回 datetime 物件（日期有效）或 None（解析失敗）
```

**實作細節**:
- 依賴現有的 `parse_ticket_frontmatter()` 函式（已在 acceptance-gate-hook.py 中實現）
- 使用標準的 datetime 解析方法，優先級由簡到複雜
- 所有異常情況都記錄 warning 日誌，不拋出例外
- 返回 None 時不影響主流程（優雅降級）

**邊界條件處理**:
- created 欄位缺失 → 返回 None（預期情況）
- created 值為空字串 → 返回 None（預期情況）
- created 值格式無效 → 返回 None（預期情況，日誌記錄）
- 檔案讀取失敗 → 返回 None，warning 日誌

### 3.2 函式 2: `check_error_patterns_changed()`

**職責**: 掃描 `.claude/error-patterns/` 目錄，找出所有 mtime > ticket_created 的 .md 檔案

**虛擬碼**:

```
函式 check_error_patterns_changed(project_root: Path, ticket_created: datetime, logger)
  → tuple[bool, list[str]]:

  前置條件:
    - project_root 為有效的專案根目錄
    - ticket_created 為有效的 datetime 物件（或 None）
    - logger 物件可用

  邏輯流程:
    Step 1: 驗證前置條件
      IF ticket_created 為 None:
        logger.warning("ticket created time 為 None，跳過檢查")
        RETURN (False, [])

    Step 2: 檢查 error-patterns 目錄是否存在
      error_patterns_dir = project_root / ".claude" / "error-patterns"
      IF error_patterns_dir 不存在:
        logger.info("error-patterns 目錄不存在，跳過檢查")
        RETURN (False, [])

    Step 3: 初始化結果容器
      changed_files = []
      ticket_created_timestamp = ticket_created.timestamp()

    Step 4: 遞迴掃描目錄
      FOR EACH 檔案 IN error_patterns_dir.rglob("*.md"):
        嘗試:
          file_mtime = 檔案.stat().st_mtime

          IF file_mtime > ticket_created_timestamp:
            相對路徑 = 檔案.relative_to(project_root)
            changed_files.append(str(相對路徑))
            logger.debug(f"找到新增檔案: {相對路徑}")

        捕獲異常 (OSError, PermissionError):
          logger.warning(f"無法讀取檔案 stat: {檔案}")
          CONTINUE（跳過此檔案）

    Step 5: 生成日誌和返回
      logger.info(f"掃描完成：發現 {len(changed_files)} 個新增/修改檔案")

      has_changed = len(changed_files) > 0
      RETURN (has_changed, changed_files)

  後置條件:
    - 返回 (False, []) 當：目錄不存在、ticket_created=None、無新增檔案、讀取失敗
    - 返回 (True, [file_paths]) 當：找到 1+ 個 mtime > ticket_created 的檔案
```

**實作細節**:
- 使用 `rglob("*.md")` 遞迴掃描所有子目錄
- mtime 比較使用 timestamp（秒級精度，符合檔案系統粒度）
- 相對路徑從 project_root 開始（一致的路徑表示）
- OSError 和 PermissionError 被優雅捕獲，不中斷掃描
- 若掃描過程中任何異常，不影響已找到的檔案（最大努力掃描）

**邊界條件處理**:
- 目錄不存在 → 返回 (False, [])，info 日誌（預期情況）
- ticket_created = None → 返回 (False, [])，warning 日誌（安全考量）
- 無檔案或所有檔案 mtime <= ticket_created → 返回 (False, [])
- 無讀取權限 → 返回 (False, [])，warning 日誌（安全降級）
- 目錄只有子目錄無 .md 檔案 → 返回 (False, [])

### 3.4 修改函式 2: `generate_hook_output()`

**目的**: 調整優先級順序，新增場景 #17 輸出邏輯

**新簽名**:
```python
def generate_hook_output(
    should_block: bool,
    message: Optional[str],
    ask_user_reminder: bool = False,
    next_step_reminder: bool = False,
    has_new_error_patterns: bool = False,
    new_error_pattern_files: list[str] | None = None
) -> Dict[str, Any]:
```

**虛擬碼（優先級邏輯）**:

```
函式 generate_hook_output(...) → Dict[str, Any]:

  初始化:
    context_parts = []  # 用於收集待輸出的訊息部分

  優先級順序（第一優先級到第四優先級）:

    [第一優先級] 錯誤或警告訊息
    ================================
    IF message IS NOT None:
      context_parts.append(message)

    [第二優先級（新增）] error-pattern 場景 #17 提醒
    ================================
    IF (has_new_error_patterns) AND (message IS None):
      # 邏輯：只在無其他訊息時輸出 #17

      檔案清單格式化:
        file_list_formatted = "\n".join(f"  - {f}" for f in (new_error_pattern_files or []))

      訊息格式化:
        reminder_msg = AskUserQuestionMessages.ERROR_PATTERN_REMINDER.format(
          file_count=len(new_error_pattern_files or []),
          file_list=file_list_formatted
        )

      context_parts.append(reminder_msg)

    [第三優先級（原第二）] complete 流程提醒（驗收方式，場景 #1）
    ================================
    IF (ask_user_reminder) AND (message IS None):
      # 邏輯：無訊息時輸出（與 #17 無關）

      context_parts.append(AskUserQuestionMessages.COMPLETE_REMINDER)

    [第四優先級（原第三）] complete 後下一步提醒（場景 #2）
    ================================
    IF (next_step_reminder) AND (message IS None) AND (NOT has_new_error_patterns):
      # 邏輯：只在無訊息且無 error-pattern 時輸出
      # 設計意圖：#17 提醒會接管後續路由，#2 被抑制

      context_parts.append(AskUserQuestionMessages.COMPLETE_NEXT_STEP_REMINDER)

  生成輸出:
    output = {
      "hookSpecificOutput": {
        "hookEventName": "PreToolUse",
        "permissionDecision": "deny" if should_block else "allow"
      }
    }

    IF len(context_parts) > 0:
      output["hookSpecificOutput"]["additionalContext"] = "\n\n".join(context_parts)

    output["check_result"] = {
      "should_block": should_block,
      "timestamp": datetime.now().isoformat()
    }

    RETURN output
```

**優先級設計說明**:
- 優先級 1（訊息）：阻擋/警告最重要，直接中斷後續邏輯
- 優先級 2（#17）：error-pattern 是本次新增的邏輯，高於場景 #1/#2
- 優先級 3（#1）：驗收方式確認總是保留，不受 #17 影響
- 優先級 4（#2）：被 #17 抑制，因為 #17 已接管後續路由決策

**修改實現（關鍵條件）**:
```python
# 場景 #17 條件：has_new_error_patterns AND NOT message
if has_new_error_patterns and not message:
    # 輸出 #17

# 場景 #2 條件：next_step_reminder AND NOT message AND NOT has_new_error_patterns
if next_step_reminder and not message and not has_new_error_patterns:
    # 輸出 #2
```

## 4. 資料結構選擇

### 4.1 `check_acceptance_status()` 返回值

**選項 A：5-tuple（已選）**
```python
Tuple[bool, bool, Optional[str], bool, list[str]]
# (should_block, has_acceptance, message, has_new_error_patterns, new_error_pattern_files)
```

**優點**:
- 簡潔，直接使用 tuple unpacking
- 最小改動現有程式碼
- 一致性：現有邏輯也使用 tuple 返回

**缺點**:
- 返回值位置隱性，容易誤用
- 若需要新增第 6 個欄位，又要改簽名

**決策理由**: 此實作暫時只需要 5-tuple，未來若有變更可考慮 dataclass，但當前簡潔性優先。

### 4.2 檔案清單表示

**選項 A：相對路徑（已選）**
```python
["error-patterns/implementation/IMP-020.md",
 "error-patterns/process-compliance/PC-006.md"]
```

**優點**:
- 相對路徑易於讀者理解（相對於專案根目錄）
- 一致性：與 `new_error_pattern_files` 定義一致
- 訊息清晰：用戶可直接在專案根目錄找到檔案

**選項 B：完整路徑**
- 浪費空間，使用者不關心絕對路徑

**決策理由**: 相對路徑是標準做法，與專案結構一致。

### 4.3 datetime 物件類型

**選項 A：datetime.datetime（已選）**
```python
from datetime import datetime
ticket_created = datetime(2026, 3, 8, 10, 30, 45)
```

**優點**:
- 標準庫，無額外依賴
- 易於時間計算和比較
- mtime 轉換簡單：`Path.stat().st_mtime` 可直接 timestamp() 比較

**選項 B：字串格式**
- 需要每次重複解析，低效

**決策理由**: datetime 物件是最自然的選擇，符合 Python 最佳實踐。

```
     └─── 否 → EXIT_SUCCESS (0)
```

### 5.2 error-pattern 檢查流程（詳細）

```
get_ticket_created_time(ticket_file, logger)
│
├─ 讀取檔案內容
│  └─ 失敗 → return None (warning log)
│
├─ 呼叫 parse_ticket_frontmatter()
│  └─ 失敗 → return None
│
├─ 取得 created 欄位值
│  ├─ 欄位不存在 → return None (warning log)
│  ├─ 欄位為空 → return None (warning log)
│  └─ 欄位存在 → 進行格式解析
│
└─ 嘗試解析日期（優先級）
   ├─ ISO 8601 (datetime.fromisoformat)
   │  └─ 成功 → return datetime object
   │
   ├─ 簡單日期 YYYY-MM-DD (strptime)
   │  └─ 成功 → return datetime object
   │
   ├─ RFC 3339 (isoparse)
   │  └─ 成功 → return datetime object
   │
   └─ 全部失敗 → return None (warning log)

check_error_patterns_changed(project_root, ticket_created, logger)
│
├─ ticket_created IS None? → return (False, [])
│
├─ error-patterns 目錄存在?
│  ├─ 不存在 → return (False, []) (info log)
│  └─ 存在 → 進行掃描
│
├─ 遞迴掃描 *.md 檔案
│  ├─ FOR EACH file:
│  │  ├─ 讀取 mtime
│  │  │  └─ 失敗 → 記錄 warning，繼續
│  │  │
│  │  └─ mtime > ticket_created?
│  │     ├─ 是 → 加入 changed_files
│  │     └─ 否 → 跳過
│  │
│  └─ 掃描完成
│
└─ return (has_changed, changed_files)
```

## 6. 邊界條件實作指引

### 6.1 邊界條件表

| 邊界條件 | 觸發情況 | 處理方式 | 日誌級別 | 影響 |
|---------|---------|---------|---------|------|
| ticket_created = None | frontmatter 缺少 created 欄位或格式無效 | 跳過 error-pattern 檢查，返回 (False, []) | warning | 無場景 #17 提醒，正常流程 |
| error-patterns 目錄不存在 | `.claude/error-patterns/` 目錄不存在 | 返回 (False, []) | info | 無場景 #17 提醒，正常流程 |
| 無 .md 檔案 | 目錄存在但無檔案或所有 mtime <= created | 返回 (False, []) | info | 無場景 #17 提醒，正常流程 |
| 無讀取權限 | `.claude/error-patterns/` 無讀取權限 | 捕獲 OSError，返回 (False, []) | warning | 無場景 #17 提醒，正常流程 |
| mtime = created（邊界） | 檔案 mtime 恰好等於 ticket created | 不計入（使用 > 嚴格大於） | debug | 安全起見，等於不視為「新增」 |
| should_block = True | 子任務未完成 | 不執行 error-pattern 檢查 | 取決於子任務檢查 | 優先顯示阻擋訊息，#17 被抑制 |
| message 存在 | 驗收記錄缺失或其他警告 | 不輸出場景 #17（訊息優先） | 取決於訊息類型 | #17 被訊息抑制 |
| has_new_error_patterns = True && message != None | 同時有新增 error-pattern 和訊息 | 輸出 message，抑制場景 #17 | 訊息日誌 | 訊息優先級最高 |

### 6.2 邊界條件虛擬碼

```
// 邊界 1: 日期無法解析
IF ticket_created IS None:
  logger.warning("無法解析 ticket created 時間")
  has_new_error_patterns = False
  new_error_pattern_files = []
  RETURN  // 不影響主流程

// 邊界 2: 目錄不存在
IF error_patterns_dir 不存在:
  logger.info("error-patterns 目錄不存在")
  has_new_error_patterns = False
  new_error_pattern_files = []
  RETURN

// 邊界 3: 無讀取權限
TRY:
  FOR file IN error_patterns_dir.rglob("*.md"):
    file.stat()
EXCEPT (OSError, PermissionError) AS e:
  logger.warning(f"讀取失敗: {e}")
  // 不中斷，已找到的檔案保留

// 邊界 4: mtime 邊界（精度問題）
IF file_mtime > ticket_created_timestamp:  // 使用 > 而非 >=
  // 只計入嚴格新增

// 邊界 5: 阻擋時跳過 error-pattern 檢查
IF should_block:
  // 不執行 error-pattern 檢查邏輯
  has_new_error_patterns = False
  new_error_pattern_files = []
```
