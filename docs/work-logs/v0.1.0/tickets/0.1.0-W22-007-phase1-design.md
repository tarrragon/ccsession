# W22-007 Phase 1 設計規格：acceptance-gate-hook.py 場景 #17

## 1. 現有架構分析

### 1.1 現有 acceptance-gate-hook.py 結構

Hook 的核心執行流程（lines 653-673）：
1. **驗收狀態檢查** → `check_acceptance_status()`
   - 返回 3-tuple: `(should_block, has_acceptance, message)`
   - `should_block`: 子任務未完成時為 True（EXIT_BLOCK）
   - `has_acceptance`: 是否有驗收記錄
   - `message`: 錯誤或警告訊息

2. **輸出生成** → `generate_hook_output()`
   - 參數：`should_block`, `message`, `ask_user_reminder`, `next_step_reminder`
   - 邏輯：錯誤/警告訊息 → 驗收方式提醒（場景 #1） → 下一步提醒（場景 #2）

3. **Exit Code 決定**（lines 680-685）
   - `should_block=True` → EXIT_BLOCK (2)
   - 其他 → EXIT_SUCCESS (0)

### 1.2 hook_messages.py 訊息組織

三層訊息模型：
- **GateMessages**: 阻擋/警告訊息（共用於 5 個 gate hooks）
- **AskUserQuestionMessages**: 決策點提醒（場景 #1/#2/#3...#17）
- **format_message()**: 模板格式化函式

現有場景 #1/#2 訊息（lines 584-629）：
- `COMPLETE_REMINDER`: 驗收方式確認（場景 #1）
- `COMPLETE_NEXT_STEP_REMINDER`: 完成後下一步（場景 #2）

### 1.3 關鍵約束

**ticket-lifecycle.md v5.2.0 場景 #17 規則**（來源任務）：
```
ticket track complete 執行時（PreToolUse Hook）：
1. 比對 ticket 的 created 時間 vs .claude/error-patterns/ 目錄中檔案 mtime
2. 若有任何 error-pattern 檔案 mtime > ticket created 時間 → 觸發 AskUserQuestion #17
3. 若無 → 正常完成，不觸發額外提醒
```

**邏輯位置**（SA 審查結論）：
- 新邏輯插入位置：`check_acceptance_status()` 函式內（步驟 6.5）
- 條件：`should_block = false && has_error_pattern = true` → 輸出 #17 提醒

## 2. 新增函式設計

### 2.1 `get_ticket_created_time()`

**目的**：從 ticket frontmatter 讀取 `created` 欄位並解析為 datetime

**函式簽名**：
```python
def get_ticket_created_time(ticket_file: Path, logger) -> Optional[datetime]:
```

**輸入**：
- `ticket_file: Path` - Ticket MD 檔案路徑
- `logger` - Hook 日誌物件

**輸出**：
- `datetime` - 解析後的建立時間
- `None` - 若無法讀取或解析

**實作邏輯**：
1. 讀取 ticket_file 內容
2. 呼叫既有的 `parse_ticket_frontmatter()` 提取 frontmatter
3. 取得 `created` 欄位值（字串格式，ISO 8601 或其他常用格式）
4. 解析為 datetime object
   - 支援格式：
     - ISO 8601: `2026-03-08T10:30:45`
     - RFC 3339: `2026-03-08T10:30:45Z` 或 `2026-03-08T10:30:45+00:00`
     - 簡單日期: `2026-03-08`
5. 返回 datetime 或 None（解析失敗時）

**日誌**：
- info: `"Ticket {ticket_id} created at: {datetime}"`
- warning: `"無法解析 created 欄位: {value}"`

**邊界條件**：
- frontmatter 缺少 `created` 欄位 → 返回 None，warning 日誌
- `created` 值為空字串 → 返回 None，warning 日誌
- 時間格式無法解析 → 返回 None，warning 日誌

### 2.2 `check_error_patterns_changed()`

**目的**：掃描 `.claude/error-patterns/` 目錄，找出所有 mtime > ticket_created 的檔案

**函式簽名**：
```python
def check_error_patterns_changed(
    project_root: Path,
    ticket_created: datetime,
    logger
) -> tuple[bool, list[str]]:
```

**輸入**：
- `project_root: Path` - 專案根目錄
- `ticket_created: datetime` - Ticket 建立時間
- `logger` - Hook 日誌物件

**輸出**：
- `bool` - 是否有新增/修改的 error-pattern（任何 mtime > ticket_created）
- `list[str]` - 新增/修改的檔案相對路徑清單，例如：
  ```
  [
    "error-patterns/implementation/IMP-020-refactoring-scope-regression.md",
    "error-patterns/process-compliance/PC-006-cli-misuse.md"
  ]
  ```

**實作邏輯**：
1. 檢查 `.claude/error-patterns/` 目錄是否存在
   - 不存在 → 返回 `(False, [])`，info 日誌
2. 遞迴掃描目錄中所有 `.md` 檔案
3. 對每個檔案：
   - 讀取 mtime（使用 `Path.stat().st_mtime` → datetime）
   - 比較 `mtime > ticket_created`
   - 符合條件 → 加入結果清單
4. 返回 `(has_changed, file_list)`

**日誌**：
- info: `"掃描 error-patterns 目錄完成：發現 {count} 個新增/修改檔案"`
- debug: `"新增/修改檔案: {relative_path} (mtime: {mtime})"`
- info: `"error-patterns 目錄不存在，跳過檢查"`（若目錄不存在）

**邊界條件**：
- `.claude/error-patterns/` 不存在 → 返回 `(False, [])`，無錯誤
- 目錄存在但無檔案 → 返回 `(False, [])`
- 目錄中只有子目錄無 `.md` 檔案 → 返回 `(False, [])`
- 無讀取權限 → log warning，返回 `(False, [])`
- ticket_created 為 None（無法解析） → log warning，跳過檢查，返回 `(False, [])`

## 3. 修改函式設計

### 3.1 `check_acceptance_status()` 修改

**現有簽名**（line 413）：
```python
def check_acceptance_status(
    ticket_id: str,
    project_dir: Path,
    logger
) -> Tuple[bool, bool, Optional[str]]:
    """
    返回: (should_block, has_acceptance, message)
    """
```

**新簽名**：
```python
def check_acceptance_status(
    ticket_id: str,
    project_dir: Path,
    logger
) -> Tuple[bool, bool, Optional[str], bool, list[str]]:
    """
    返回: (should_block, has_acceptance, message, has_new_error_patterns, new_error_pattern_files)
    """
```

**修改位置**：在現有邏輯完成後（line 490），新增步驟：

```python
# [新增步驟 6.5] 檢查是否有新增 error-pattern
has_new_error_patterns = False
new_error_pattern_files = []

# 僅在不阻擋且需要檢查的情況下執行
if not should_block and should_check_acceptance:
    # 取得 ticket 建立時間
    ticket_created = get_ticket_created_time(ticket_file, logger)

    if ticket_created:
        # 檢查 error-patterns 目錄
        has_new_error_patterns, new_error_pattern_files = check_error_patterns_changed(
            project_dir,
            ticket_created,
            logger
        )

        if has_new_error_patterns:
            logger.info(f"發現 {len(new_error_pattern_files)} 個新增/修改的 error-pattern")
    else:
        logger.warning(f"無法取得 ticket {ticket_id} 的建立時間，跳過 error-pattern 檢查")

# 返回 5-tuple（增加新欄位）
return should_block, has_acceptance, message, has_new_error_patterns, new_error_pattern_files
```

**邏輯限制**：
- `has_new_error_patterns` 僅在 `should_block=False` 且 `should_check_acceptance=True` 時才計算
- 若 ticket_created 無法解析 → `has_new_error_patterns=False, new_error_pattern_files=[]`（不影響主流程）
- 若目錄不存在 → `has_new_error_patterns=False, new_error_pattern_files=[]`

### 3.2 `generate_hook_output()` 修改

**現有簽名**（line 501）：
```python
def generate_hook_output(
    should_block: bool,
    message: Optional[str],
    ask_user_reminder: bool = False,
    next_step_reminder: bool = False
) -> Dict[str, Any]:
```

**新簽名**：
```python
def generate_hook_output(
    should_block: bool,
    message: Optional[str],
    ask_user_reminder: bool = False,
    next_step_reminder: bool = False,
    has_new_error_patterns: bool = False,
    new_error_pattern_files: list[str] = None
) -> Dict[str, Any]:
```

**修改邏輯**（line 526-541）：

現有優先級順序：
1. 錯誤或警告訊息
2. complete 流程提醒（場景 #1）
3. complete 後下一步提醒（場景 #2）

**新優先級順序**：
1. 錯誤或警告訊息
2. error-pattern 場景 #17 提醒（新增，優先級 2.5，僅在無訊息且 has_new_error_patterns=True 時追加）
3. complete 流程提醒（場景 #1）
4. complete 後下一步提醒（場景 #2）

```python
context_parts = []

# 第一優先級：錯誤或警告訊息
if message:
    context_parts.append(message)

# [新增] 第二優先級：error-pattern 場景 #17 提醒（僅在無訊息時）
if has_new_error_patterns and not message:
    context_parts.append(
        AskUserQuestionMessages.ERROR_PATTERN_REMINDER.format(
            file_count=len(new_error_pattern_files or []),
            file_list="\n".join(f"  - {f}" for f in (new_error_pattern_files or []))
        )
    )

# 第三優先級（原第二）：complete 流程提醒（驗收方式，場景 #1）
if ask_user_reminder and not message:
    context_parts.append(AskUserQuestionMessages.COMPLETE_REMINDER)

# 第四優先級（原第三）：complete 後下一步提醒（路由選擇，場景 #2，僅在無訊息時追加）
if next_step_reminder and not message and not has_new_error_patterns:
    context_parts.append(AskUserQuestionMessages.COMPLETE_NEXT_STEP_REMINDER)
```

**邊界條件**：
- `new_error_pattern_files` 為 None → 使用 `[]`
- 僅當 `has_new_error_patterns=True` 且無其他訊息時才輸出 #17 提醒
- 若有 error-pattern 提醒，下一步提醒（場景 #2）被抑制，由場景 #17 接管後續路由

```python
# ========================================================================
# 場景 17: 錯誤學習經驗確認（ticket complete 時）

ERROR_PATTERN_REMINDER = """============================================================
```

### 4.2 訊息內容設計

**訊息組成**：
- 標題：`[AskUserQuestion 強制提醒] 錯誤學習經驗確認（場景 #17）`
- 偵測訊息：說明發現了新的 error-pattern（帶檔案清單）
- 規範引用：ticket-lifecycle.md v5.2.0
- 三個選項說明：
  1. **建立改進 Ticket（Recommended）** - 優先選項，建立 Ticket 追蹤修復
  2. **已有對應 Ticket** - 備選，若修復已在進行中
  3. **延後處理** - 備選，用於 P2/P3 問題的暫緩處理
- 工具提示：ToolSearch 載入方式
- 規範連結

**訊息變數**：
- `{file_list}` - 新增/修改檔案清單（markdown 清單格式，前綴 2 個空格）
- 範例：
  ```
  - error-patterns/implementation/IMP-020-refactoring-scope-regression.md
  - error-patterns/process-compliance/PC-006-cli-misuse.md
  ```

### 4.3 訊息輸出位置

**優先級順序**（在 `generate_hook_output()` 中）：

```
無訊息情況下的優先級：
1. 錯誤或警告訊息 （message != None）
   ↓
2. error-pattern 場景 #17 提醒（has_new_error_patterns=True）
   ↓
3. complete 流程提醒（ask_user_reminder=True）
```

**關鍵邏輯**：
- 場景 #17 提醒僅在 `has_new_error_patterns=True && message=None` 時輸出
- 若輸出場景 #17，則抑制場景 #2（下一步提醒）
- 場景 #1（驗收方式提醒）保留，因為 complete 前驗收方式必須確認

## 5. 邊界條件處理

### 5.1 `.claude/error-patterns/` 目錄不存在

**處理方式**：
- 不視為錯誤，返回 `(False, [])`
- 日誌級別：info
- 日誌內容：`"error-patterns 目錄不存在，跳過檢查"`
- 流程：無場景 #17 提醒，正常流程（場景 #1/#2）

### 5.2 Ticket created 時間無法解析

**處理方式**：
- `get_ticket_created_time()` 返回 `None`
- 在 `check_acceptance_status()` 中捕獲 None，跳過 `check_error_patterns_changed()`
- 日誌級別：warning
- 日誌內容：`"無法取得 ticket {ticket_id} 的建立時間，跳過 error-pattern 檢查"`
- 流程：無場景 #17 提醒，正常流程（場景 #1/#2）
- **不阻擋執行**（exit code 仍為 0）

### 5.3 Frontmatter 缺少 `created` 欄位

**處理方式**：
- `get_ticket_created_time()` 返回 `None`
- 日誌級別：warning
- 日誌內容：`"Ticket {ticket_id} frontmatter 缺少 created 欄位"`
- 流程：同 5.2

### 5.4 有新增 error-pattern 且 should_block=true

**處理方式**：
- error-pattern 檢查不執行（條件限制）
- 日誌：記錄子任務未完成的阻擋原因
- 流程：優先顯示阻擋訊息，不輸出場景 #17
- **設計意圖**：子任務未完成是更嚴重的問題，應優先提示

### 5.5 error-patterns 目錄讀取失敗

**處理方式**：
- 捕獲 `OSError` 或權限錯誤
- 日誌級別：warning
- 日誌內容：`"讀取 error-patterns 目錄失敗: {error}"`
- 流程：返回 `(False, [])`，無場景 #17 提醒
- **不阻擋執行**
