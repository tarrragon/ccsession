## 問題分析

### 背景

當前 `hook_utils/hook_logging.py` 同時承擔兩個不相關的職責：

1. **日誌系統**：setup_hook_logging()、run_hook_safely()、save_check_log()
2. **專案路徑查詢**：get_project_root()、_find_project_root()、環境變數常數

### 根本問題

| 問題 | 影響層級 | 具體後果 |
|------|--------|--------|
| **職責混雜** | 架構設計 | 違反單一責任原則，模組邊界不清 |
| **依賴順序錯誤** | 模組設計 | 基礎設施（路徑查詢）應被上層依賴，卻混在日誌模組中 |
| **複用受限** | 維護成本 | 新模組若需 get_project_root，被迫依賴整個 logging 配置邏輯 |
| **測試困難** | 品質 | 測試 get_project_root 時被迫初始化日誌系統 |
| **循環依賴風險** | 系統穩定 | 模組擴張時容易引入循環依賴 |

### 現狀結構圖

```
問題狀態：
├─ hook_logging.py
│  ├─ 日誌功能（FileHandler、StreamHandler）
│  └─ 路徑查詢（get_project_root、_find_project_root）←── 不相關
│
├─ hook_ticket.py
│  └─ from .hook_logging import get_project_root
│     └─ 為了取得路徑，被迫依賴整個日誌模組 ❌
│
└─ hook_io.py
   └─ from .hook_logging import get_project_root
      └─ 同上，不必要的耦合 ❌
```

---

## 解決方案

### 設計目標

新建 `hook_base.py` 作為基礎設施層，分離職責並建立清晰的模組分層。

### 目標結構

```
目標狀態：
├─ hook_base.py（基礎層，新建）
│  ├─ get_project_root()
│  ├─ _find_project_root()
│  ├─ ENV_PROJECT_DIR = "CLAUDE_PROJECT_DIR"
│  └─ CLAUDE_MD_SEARCH_DEPTH = 5
│
├─ hook_logging.py（日誌層）
│  ├─ 日誌功能（原有，不變）
│  └─ from .hook_base import get_project_root ✓ Re-export
│
├─ hook_ticket.py（應用層）
│  └─ from .hook_base import get_project_root ✓ 直接依賴
│
└─ hook_io.py（應用層）
   └─ from .hook_base import get_project_root ✓ 直接依賴
```

### 修改清單

| 檔案 | 修改類型 | 變更內容 |
|------|--------|--------|
| `hook_base.py` | 新建 | 包含 get_project_root() + _find_project_root() + 常數 |
| `hook_logging.py` | 修改 | 移除定義，添加 re-export：`from .hook_base import get_project_root` |
| `hook_ticket.py` | 修改 | 第 27 行：`from .hook_base import get_project_root` |
| `hook_io.py` | 修改 | 第 273 行：`from .hook_base import get_project_root as _get_project_root` |
| `__init__.py` | 無修改 | Re-export 鏈路完整，無需變更 |

### 向後相容性保證

re-export 策略確保 54+ 外部 Hook 零修改：

```python
# 舊 import 路徑（外部 Hook）- 繼續有效 ✓
from hook_utils import get_project_root
from hook_utils.hook_logging import get_project_root

# 新 import 路徑（內部模組）- 直接依賴基礎層 ✓
from hook_utils.hook_base import get_project_root
```

## 1. 測試策略規劃

### 1.1 分層決策樹

本重構涉及模組級別的結構調整，適用以下測試策略：

| 模組 | 層級 | 複雜度 | 測試策略 |
|------|------|--------|---------|
| hook_base（新） | 基礎層 | 中等 | 單元測試 + 整合測試 |
| 模組依賴（import 轉換） | 結構層 | 高 | 整合測試 + 循環依賴檢查 |
| 向後相容性（re-export） | 系統層 | 低 | 整合測試 + 外部驗證 |

### 1.2 Sociable Unit Tests 原則

#### Mock 策略判斷

| 依賴 | Mock 策略 | 理由 |
|-----|---------|------|
| 環境變數 CLAUDE_PROJECT_DIR | Mock | 外部狀態，測試需控制 |
| 工作目錄（cwd） | Mock | 外部狀態，測試需隔離 |
| 檔案系統（CLAUDE.md） | 真實（tmp_path） | Domain 行為，使用 fixture 隔離 |
| Path 物件 | 真實 | Domain Entity，直接使用 |
| os 模組 | 真實 | 內部邏輯，保持真實 |

#### 測試隔離級別

- **Module** = 單個函式（get_project_root、_find_project_root） + 其直接使用的常數
- **Mock 邊界**：環境變數、工作目錄
- **真實互動**：檔案系統（tmp_path 隔離）、Path 物件操作

    assert 'hook_ticket' not in imports
    assert 'hook_io' not in imports

## 5. 設計決策記錄

### 決策 1：為什麼分離到 hook_base？

職責混雜導致設計混亂。基礎設施應該被上層依賴，而非混在日誌模組中。

### 決策 2：為什麼使用 re-export？

向後相容成本最低，54+ Hook 無需修改，future 可逐步遷移。

### 決策 3：為什麼 Mock 環境變數但不 Mock 檔案系統？

Sociable Unit Tests 原則：測試行為而非實作，檔案系統檢查是行為的一部分。
