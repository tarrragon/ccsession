**問題**：_setup_logger_handlers 每次執行都讀寫 .cleanup_trigger 計數器檔案（2 次 I/O），54 個 hook 目錄各一個。清理只在每 10 次觸發。一次用戶操作 ~10 個 hook = 20 次額外 I/O。

**影響範圍**：54 個 .cleanup_trigger 檔案，所有 hook 的 setup_hook_logging 路徑。

**效能影響評估**：計數器 read/write 為毫秒級，在 hook 總執行時間（50-200ms）中佔比 <5%，非效能瓶頸。

## 替代方案分析

### 方案 A：基於 mtime 時間觸發（推薦）
- 用 .cleanup_trigger 的 mtime 判斷是否需要清理（每 N 小時一次）
- I/O：stat() 取代 read+write，減少約 50%
- 語義更好：時間間隔比計數器更合理
- 改動最小：僅改 _setup_logger_handlers 內部

### 方案 B：隨機觸發
- random.randint(1, 10) == 1 代替計數器
- I/O 減少至 0（無需檔案）
- 缺點：不可預測，可能長時間不觸發或短時間多次觸發

### 方案 C：全域單一觸發檔
- 一個共用 .cleanup_trigger 替代 54 個
- 改動大，收益小，需處理並行存取問題

## 結論
建議採用方案 A（mtime），但 P2 優先級，非效能瓶頸。可在下次修改 hook_utils.py 時順便實施，不需單獨排 IMP Ticket。
