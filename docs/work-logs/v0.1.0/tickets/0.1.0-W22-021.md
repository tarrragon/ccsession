調查結果：

三個 hook 各自維護不同版本的 YAML frontmatter 解析邏輯：

1. **hook_utils.py** - 簡化版本
   - 簽名：parse_ticket_frontmatter(file_path: Path, logger: logging.Logger) -> Optional[dict]
   - 輸入：Path 物件
   - 只解析頂層 key-value，不支援多行字串和嵌套結構

2. **acceptance-gate-hook.py** - 高級版本
   - 簽名：parse_ticket_frontmatter(content: str) -> Dict[str, str]
   - 輸入：文件內容字串
   - 支援多行字串（|, >, |-, >-）和嵌套結構
   - 無 logger 參數，無法記錄解析錯誤

3. **parallel-suggestion-hook.py** - 中等版本
   - 簽名：parse_ticket_frontmatter(content: str) -> Dict[str, Any]
   - 輸入：文件內容字串  
   - 支援嵌套鍵值對（2 個空格縮排）
   - 無 logger 參數

問題：
- 簽名不一致（Path vs str 輸入）
- 功能覆蓋不完整（有些函式缺少某些功能）
- 重複維護成本高
- 三個版本缺少統一的錯誤日誌記錄
## Solution

<!-- To be filled by executing agent -->

---

統一方案已實現：

在 hook_utils.py 中設計新的統一函式：

**新函式簽名：**
```python
def parse_ticket_frontmatter(
    content_or_path: str | Path,
    logger: logging.Logger | None = None
) -> Optional[dict]
```

**功能特性：**
1. 支援雙重輸入模式：
   - 接受 str（文件內容）- 直接解析
   - 接受 Path（檔案路徑）- 先讀取再解析

2. 統一 YAML 解析能力：
   - 頂層 key-value 對
   - 多行字串（|, >, |-, >-）
   - 嵌套結構（縮排鍵值對）

3. 錯誤處理改進：
   - logger 參數可選（向後相容）
   - 解析失敗時返回空 dict（不返回 None）
   - 若提供 logger，自動記錄錯誤

**修改內容：**
1. ✓ 更新 hook_utils.py - 新增統一函式
2. ✓ 更新 acceptance-gate-hook.py：
   - 導入統一的 parse_ticket_frontmatter
   - 刪除本地定義（68 行減少）
3. ✓ 更新 parallel-suggestion-hook.py：
   - 導入統一的 parse_ticket_frontmatter  
   - 刪除本地定義（70 行減少）
4. ✓ 驗證無其他 hook 有重複定義

**向後相容性：**
- 接受 Path 輸入（hook_utils 舊 API）
- 接受 str 輸入（兩個 hook 舊 API）
- 始終返回 dict（兼容各種用途）
- logger 參數可選
