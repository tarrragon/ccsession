# 0.1.0-W37-002: track_board.py 測試案例設計

## 執行摘要

track_board.py 包含 17 個公開函式，需要設計 95+ 個測試案例以達到全面覆蓋。本文件詳細列出所有測試案例（GWT 格式）和測試策略。

---

## Problem Analysis

### 背景

track_board.py 是看板命令的核心模組，提供 Kanban 風格的看板視覺化。截至當前版本，該模組存在以下問題：

1. **零測試覆蓋**：17 個公開函式完全無測試
2. **視覺寬度計算複雜**：涉及中英文混合、全形字元、emoji 等多個邊界
3. **多個 bug 修復記錄**：
   - W33-001: 某個功能 bug
   - W33-003: 排版相關問題
   - W34-006: 寬度計算問題
   - 均由多視角審查發現，未被測試攔截

4. **高風險函式**：
   - `calculate_visual_width` - 基礎寬度計算，影響所有排版
   - `ljust_with_chinese_width` - 中文對齐基礎
   - `truncate_title` - 複雜的中文截斷邏輯
   - `render_board_unicode` - 看板渲染核心

### 分析結果

通過靜態分析 728 行程式碼，識別以下重點：

1. **複雜度分層**：
   - Layer 5 (Domain): `calculate_visual_width`, `ljust_with_chinese_width`, `truncate_title` - 純粹的數據轉換邏輯
   - Layer 2 (Behavior): `build_tree_structure`, `render_tree_node`, `prepare_cards`, `calculate_layout` - 業務邏輯組合
   - Layer 1 (UI): `render_board_unicode`, `render_board_tree`, `render_board_ascii` - 使用者介面呈現

2. **邊界條件複雜度**：
   - `calculate_visual_width`: 中文(2寬) + 英文(1寬) + emoji(2寬) + 全形標點(2寬)
   - `ljust_with_chinese_width`: 寬度不足、恰好、超過三種情況
   - `truncate_title`: 截斷位置在字元邊界，需考慮省略符號寬度

3. **依賴關係**：
   - `ljust_with_chinese_width` 依賴 `calculate_visual_width`
   - `truncate_title` 依賴 `get_char_display_width`
   - `render_board_unicode` 依賴 `ljust_with_chinese_width` 進行對齁

### 根因分析

過去 bug 無法被測試攔截的原因：

1. **無單元測試**：寬度計算邏輯未被隔離測試
2. **手動測試局限**：終端寬度、字體等影響，難以重現邊界條件
3. **缺乏邊界條件驗證**：特殊字元組合（如中英混合）未被系統驗證

## Solution

### 設計目標

1. **覆蓋率**：>= 95% 行覆蓋率，>= 90% 分支覆蓋率
2. **邊界條件**：100% 覆蓋高風險函式的邊界條件
3. **測試案例**：95+ 個測試案例，分層設計
4. **可維護性**：清晰的 GWT 格式，便於後續調整

### 設計策略

#### 1. 分層測試策略

**Layer 5 (Domain Logic - 單元測試)**
- 目標：驗證純粹的數據轉換邏輯
- 函式：
  - `calculate_visual_width` (9 個 test cases)
  - `ljust_with_chinese_width` (9 個 test cases)
  - `truncate_title` (12 個 test cases)
  - `get_char_display_width` (9 個 test cases)
- Mock 策略：無需 Mock，直接使用標準庫 unicodedata
- 預期效果：快速、確定性高、易於調試

**Layer 2 (Behavior - 單元測試)**
- 目標：驗證業務邏輯組合
- 函式：
  - `build_tree_structure` (6 個 test cases)
  - `render_tree_node` (5 個 test cases)
  - `prepare_cards` (多個 test cases)
  - `calculate_layout` (多個 test cases)
- Mock 策略：Mock `list_tickets` 結果
- 預期效果：驗證邏輯正確性，無排版依賴

**Layer 1 (UI - 整合測試)**
- 目標：驗證最終呈現效果
- 函式：
  - `render_board_unicode` (多個 test cases)
  - `render_board_tree` (多個 test cases)
  - `render_board_ascii` (多個 test cases)
  - `execute_board` (多個 test cases)
- Mock 策略：Mock `list_tickets`、`resolve_version`、終端寬度
- 預期效果：驗證整體流程，確保用戶可見內容正確

#### 2. Sociable Unit Tests 原則

**應用**：
- **不 Mock 標準庫**：`unicodedata.east_asian_width` 是確定性的標準行為
- **不 Mock Domain 物件**：直接使用真實的 ticket dictionary
- **只 Mock 外部系統**：`list_tickets`、終端 I/O 等
- **測試行為而非結構**：驗證寬度計算結果，不驗證迴圈實作

**優勢**：
- 測試與實作的耦合度低
- 重構時測試保持穩定
- 發現真實的整合問題

#### 3. 邊界條件設計

**寬度計算邊界**（`calculate_visual_width`）：
- 空字串：0
- 單字元（英文/中文/emoji）：1 或 2
- 混合（英+中+全形+emoji）：各取其值
- 特殊字元（日文、韓文）：2

**填充邊界**（`ljust_with_chinese_width`）：
- 寬度不足：補充空格
- 寬度達到：無補充
- 寬度超過：無截斷（保留原文本）
- 目標寬度 0：無補充

**截斷邊界**（`truncate_title`）：
- max_length=0：返回空字串
- max_length < 2：無法容納 ".."，可能返回空字串
- 截斷位置在字元邊界：選擇前一個完整字元
- 省略符寬度：2（計入總寬度）

**樹狀結構邊界**（`build_tree_structure`）：
- 無子任務：只有根任務清單
- 孤兒子任務：父任務缺失時，作為根任務處理
- 多層嵌套：深度無限制，但測試驗證 3 層
- 排序：按字典序排序

#### 4. Mock 和 Fixture 設計

**Test Data Fixtures**：
```
TICKET_SIMPLE：簡單英文票，用於基本流程
TICKET_CHINESE：純中文票，用於寬度計算驗證
TICKET_MIXED：中英混合票，用於邊界驗證
TICKET_SUBTASK：子任務票，用於樹狀結構驗證
TICKET_BLOCKED：被阻塞票，用於狀態過濾驗證
TICKET_LONG_TITLE：長標題票，用於截斷驗證
```

**Mock 策略**：
- `list_tickets(version)`：用 fixture 返回預先定義的 tickets
- `resolve_version(args)`：直接使用版本字串
- `shutil.get_terminal_size()`：可選 monkeypatch，主要測試邏輯
- `TrackBoardMessages`：使用真實常數，驗證訊息內容

### 3. `truncate_title` (L309) - 標題截斷

**功能**：按視覺寬度截斷標題，超過上限加 ".." 省略符

- Given: title = "very long english title" (len=22, width=22), max_length = 15

  Expected: something like "very long en.."

- Given: title = "這是一個很長的標題文字" (2*8=16), max_length = 15
- When: truncate_title("這是一個很長的標題文字", 15)

#### Test Case 3.3: 中英混合截斷

- When: truncate_title("AB中文CDEFGHij", 10)
- Then: return "AB中.." (1+1+2+2 = 6 width < 10)

#### Test Case 3.4: 無需截斷
- Given: title = "Short" (width=5), max_length = 15
- When: truncate_title("Short", 15)
- Then: return "Short" (no "..")

#### Test Case 3.5: 恰好達到限制
- Given: title = "中文AB" (2+2+1+1=6), max_length = 6

- Given: title = "中文" (2+2=4), max_length = 3

#### Test Case 3.7: 空字串
- Given: title = "", max_length = 15

#### Test Case 3.8: max_length = 0
- Given: title = "任意內容", max_length = 0

#### Test Case 3.9: max_length = 1

#### Test Case 3.10: max_length = 2 (exactly for "..")
- Given: title = "很長的標題" (2*5=10), max_length = 2

- Given: title = "A很長的標題", max_length = 3
- When: truncate_title("A很長的標題", 3)
- Then: return "A.." (1 + 2 = 3)

#### Test Case 3.12: 截斷點在中英邊界
- Given: title = "ABCDEFGHij中文" (1*8=8, 2+2=4, total=12), max_length = 9
- When: truncate_title("ABCDEFGHij中文", 9)

## 測試設計決策記錄

### 決策 1：使用 Parametrize vs 分開 Test Cases

決定：分開 Test Cases（更清晰的失敗訊息和邊界說明）

### 決策 2：Mock vs 不 Mock unicodedata

決定：不 Mock（測試實際行為，unicodedata 是標準庫）

### 決策 3：Fixture 粒度

決定：細粒度 fixtures（每個測試單獨創建，減少狀態耦合）

### 決策 4：整合測試 vs 單元測試

決定：
- 寬度、填充、截斷 → 單元測試
- 樹狀結構、卡片準備 → 單元測試
- 看板渲染 → 整合測試（涉及多個函式）
