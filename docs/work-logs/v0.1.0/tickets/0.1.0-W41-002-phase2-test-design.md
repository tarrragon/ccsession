## 1. 測試策略規劃

### 1.1 測試分層決策

根據 Sociable Unit Tests 原則和混合測試策略，本 Hook 適用以下分層決策：

**代碼層級**：Hook Python 腳本（utility layer，不涉及業務邏輯）
**測試層級**：**單元測試**（pytest）

**理由**：
- Hook 是純工具層，功能為「讀取 → 比對 → 輸出」
- 外部依賴有限（git 命令、Ticket 檔案、日誌系統）
- 無需複雜的整合測試或 BDD
- Sociable Unit Test 原則：Mock 外部依賴（git、檔案），使用真實的內部邏輯

### 1.2 測試覆蓋率要求

| 維度 | 目標 |
|------|------|
| **功能點覆蓋** | 100% - 所有 6 個行為場景 |
| **邊界條件覆蓋** | 100% - 路徑規範化、特殊字元、空清單等 |
| **例外情境覆蓋** | 100% - 檔案讀取失敗、git 命令失敗、JSON 解析錯誤 |
| **代碼行數覆蓋** | >= 95% - 無需要測試的死碼（判定樹覆蓋完整） |

### 1.3 測試設計哲學

**保守設計 + 誤報容忍度優先**：
- 測試場景遵循 Phase 1 的「只驗證+警告」設計
- 永遠期望 exit code 為 0（永不阻擋）
- Mock 外部依賴，驗證邏輯輸出（警告訊息、日誌記錄）
- 無需測試 Hook 框架（Hook Runner 由框架層測試）

## 2. 測試案例設計

### 2.1 主要功能場景（6 個 GWT 測試）

#### 場景 1：缺失檔案偵測 — 主要功能場景

**測試目標**：驗證 Hook 能正確偵測 where.files 中有但 git diff 中沒有的檔案

```
Given:
  - Ticket 0.1.0-W41-002 存在於 Ticket 目錄
  - Ticket frontmatter where.files = [".claude/hooks/foo.py", ".claude/settings.json"]
  - git diff HEAD --name-only 輸出只有 ".claude/hooks/foo.py"
  - Bash 命令為 "ticket track complete 0.1.0-W41-002"
  - 命令執行成功（exit 0, stdout 含"已完成"）

When:
  - PostToolUse Hook 觸發
  - Hook 讀取 Ticket 的 where.files
  - Hook 執行 git diff 取得實際變更
  - Hook 執行比對邏輯

Then:
  - Hook exit code 為 0
  - Hook 輸出的 additionalContext 包含 "[並行派發驗證警告]"
  - additionalContext 列出 ".claude/settings.json" 為缺失檔案
  - additionalContext 展示完整的 where.files 清單
  - additionalContext 展示 git diff 的結果
  - 警告訊息寫入日誌檔 .claude/hook-logs/parallel-dispatch-verification/
```

**測試配置**：
- 使用 `pytest fixture` 建立臨時 Ticket 檔案
- Mock `git diff` 的 subprocess 呼叫
- Capture stdout/stderr 驗證輸出內容
- 驗證日誌檔內容（或 mock logging）

**斷言清單**：
- [ ] exit_code == 0
- [ ] output["hookSpecificOutput"]["additionalContext"] 包含 "[並行派發驗證警告]"
- [ ] "缺失檔案" 在輸出中且列出 ".claude/settings.json"
- [ ] ".claude/hooks/foo.py" 出現在「預期修改」清單中
- [ ] ".claude/settings.json" 出現在「預期修改」清單中
- [ ] ".claude/hooks/foo.py" 出現在「實際變更」清單中
- [ ] ".claude/settings.json" 不出現在「實際變更」清單中

#### 場景 6：路徑前綴差異 — 規範化比對

**測試目標**：驗證路徑規範化函式能正確處理前綴、斜線等差異，使不同格式的路徑視為相同

```
Given:
  - Ticket where.files = ["./ui/lib/models/user.dart", "ui/lib/views/home.dart"]
  - git diff 輸出 = ["ui/lib/models/user.dart", "./ui/lib/views/home.dart"]

When:
  - 執行路徑規範化和比對

Then:
  - 兩個來源的路徑規範化後完全相同
  - 比對結果為全部匹配（無缺失警告）
```

**測試配置**：
- 測試 normalize_path() 函式單獨行為
- 測試路徑規範化後的比對結果

**規範化規則驗證**：
- `"./ui/lib/x.dart"` → `"ui/lib/x.dart"`
- `"ui/lib/x.dart"` → `"ui/lib/x.dart"`
- `"ui\\lib\\x.dart"` → `"ui/lib/x.dart"`（Windows 路徑）
- `"ui/lib/x.dart/"` → `"ui/lib/x.dart"`（尾斜線）
- `"UI/LIB/X.DART"` → `"ui/lib/x.dart"`（大小寫）

**斷言清單**：
- [ ] normalize_path("./ ...") 移除前綴 ./
- [ ] normalize_path("... /") 移除尾斜線
- [ ] normalize_path 統一使用正斜線 /
- [ ] 規範化後的路徑相等

### 2.3 否定測試（NAC — Negative Acceptance Criteria）

#### NAC-1：Hook 永不以非 0 exit code 結束

**測試目標**：驗證 Hook 在任何情況下都返回 exit code 0（永不阻擋）

```
Scenario: 任何異常情況
Given: 任何可能導致異常的輸入（JSON 解析失敗、檔案不存在等）
When: Hook 執行
Then: exit_code == 0
```

**測試場景清單**：
- [ ] JSON 解析失敗（無效 JSON）
- [ ] Ticket 檔案不存在
- [ ] Ticket 無 where.files
- [ ] git diff 命令失敗
- [ ] git diff 輸出無效格式
- [ ] 日誌寫入失敗（日誌目錄不存在）
- [ ] 訊息格式化失敗（不應發生，但防禦性測試）
- [ ] 任何未預期的異常

**斷言清單**：
- [ ] 所有場景下 exit_code == 0
- [ ] 無拋出異常傳播到上層

---

#### NAC-2：Hook 不修改任何檔案

**測試目標**：驗證 Hook 是純驗證角色，不執行任何修改操作

```
Scenario: Hook 執行前後檔案無變更
Given: Ticket 檔案、程式碼檔案、git 狀態
When: Hook 執行
Then: 執行後無新增/修改檔案（除日誌檔）
```

**測試配置**：
- 記錄執行前的 `git status`
- 執行 Hook
- 記錄執行後的 `git status`
- 比較兩者，檢查是否有差異（日誌檔除外）

**斷言清單**：
- [ ] `git status` 執行前後對比，無新增或修改檔案
- [ ] 日誌檔可能增加（允許），但其他檔案不可變

---

#### NAC-3：where.files 為空時不輸出警告

**測試目標**：驗證 AC-4 的反面 — 必須確保空 where.files 時絕不警告

```
  - 日誌記錄 "No where.files" 或類似（不算「警告輸出」）
```

**斷言清單**：
- [ ] additionalContext 不存在
- [ ] 警告關鍵字（如「缺失」「未出現」）不出現在 additionalContext

## 15. 測試設計決策記錄

### 決策 D1：為什麼不拆分為多個測試檔案？

**決策**：單一測試檔案 `test_parallel_dispatch_verification_hook.py`

**理由**：
- Hook 功能單一，測試邏輯相對獨立
- 所有測試共享相同的 fixture 和 mock 策略
- 便於維護和查閱（不超過 500 行）

---

### 決策 D2：為什麼使用參數化而非複製測試？

**決策**：normalize_path 和 find_missing_files 使用 `@pytest.mark.parametrize`

**理由**：
- 減少重複代碼
- 一次測試失敗可知道哪個參數失敗
- 便於添加新的邊界測試用例

---

### 決策 D3：為什麼 Mock git diff 而非真實執行？

**決策**：所有測試中 git diff 都被 Mock

**理由**：
- 測試不應依賴外部環境（git 可能不存在、repo 狀態可能改變）
- 提升測試執行速度
- 便於模擬各種 git 失敗場景（超時、權限、非 git repo）

---

### 決策 D4：為什麼驗證日誌輸出？

**決策**：測試驗證日誌檔寫入和內容

**理由**：
- 日誌是 Hook 的重要輸出（雙通道原則）
- 便於生產環境問題追蹤
- 驗證「靜默失敗」防護
