# W39-004 規則間接性分析報告

## 執行摘要

本分析檢查 `.claude/rules/` 目錄對 `.claude/references/` 目錄的依賴情況，評估跳轉成本和認知負擔。

**分析範圍**：28 個 rules 檔案 vs 54+ 個 references 檔案（含 hooks 子目錄）

---

## 核心發現

### 1. 跳轉規模統計

| 指標 | 數值 | 說明 |
|------|------|------|
| 引用總次數 | 47 次 | rules → references 的直接引用 |
| 引用檔案數 | 13 個 | 有對外引用的 rules 檔案 |
| 被引用最多 | 8 次 | quality-common-details |
| 被引用最少 | 1 次 | 7 個 references 檔案 |
| 內部跳轉 | 16 次 | references 內部相互引用 |
| 跳轉密度 | 1.1-4.9% | 每 100 行引用次數（全為低密度） |

### 2. 引用頻率分布

```
引用次數  檔案數
────────────────
8次       1 個  (quality-common-details) - 非常高價值
4次       2 個  (tdd-flow-details, incident-response-details)
3次       2 個  (verification-framework-details, ticket-lifecycle-phases)
2次       8 個  (中等价値)
1次       7 個  (低價值，合併候選)
```

**關鍵觀察**：
- 高度集中：質量最好的 5 個 references 佔 64%（30/47）的引用
- 低價值尾部：7 個 references 各被引用 1 次（總計 15%）
- 沒有不用的 references：最低也有 1 次引用（零引用的是純 hook 技術文件）

### 3. 檔案大小分析

| 類別 | 檔案數 | 平均行數 | 狀態 |
|------|--------|---------|------|
| 超大（>350 行） | 5 | 479 | 保留，有重要內容 |
| 大型（200-350 行） | 10 | 278 | 保留，是關鍵參考 |
| 中型（150-200 行） | 8 | 177 | 需評估密度 |
| 小型（<150 行） | 6 | 113 | 合併候選 |

**合併候選清單**（低引用 + 超大檔案）：
1. `decision-tree-diagrams.md`（142 行，1 次引用）- 可回併到 decision-tree.md
2. `query-research-scenario-examples.md`（141 行，1 次引用）- 可回併到 query-vs-research.md
3. `version-decision-case-studies.md`（201 行，1 次引用）- 可回併到 version-progression.md
4. `plan-to-ticket-mapping-details.md`（238 行，1 次引用）- 邊界，可考慮保留

---

## 認知負擔評估

### 跳轉密度（低風險）

規則檔案的引用密度均在 **1.1% ～ 4.9%**（低於認知負擔閾值 10%）。

**評估結論**：
- ✅ 所有 rules 檔案的跳轉成本都很低
- ✅ 閱讀者不會因跳轉導致認知超載
- ✅ 現狀不需要緊急改進

### 多層跳轉（可接受）

references 內部存在 16 次跳轉，分布如下：
- decision-tree-checkpoint-details → 6 次（最多）
- verification-framework-details → 4 次
- 其他 → 6 次

**多層跳轉深度** = 最多 3 層（rules → references → references → 內容）

**評估結論**：
- ⚠️ 存在輕度多層跳轉，但規模有限
- ✅ 只有專門需要深度理解的檔案才會觸發多層跳轉（如 decision-tree-checkpoint 多視角細節）
- ✅ 不是普遍問題

### 讀者旅程分析

三種典型讀者場景：

| 場景 | 跳轉次數 | 複雜度 |
|------|---------|--------|
| 快速查詢（只讀核心規則） | 0 次 | 低 |
| 理解概念（讀規則 + 1 個 reference） | 1 次 | 中 |
| 深度學習（讀規則 + 多層 reference） | 2-3 次 | 高 |

**評估結論**：
- ✅ 大多數用戶只需跳轉 0-1 次
- ✅ 深度學習場景（2-3 層）是自願的，不是強制的

## 優化建議

### 方案 A：保持現狀（推薦）

**優勢**：
- 認知負擔指標完全合格（密度 < 5%）
- 無多層跳轉的大規模問題
- 高價值 references 已被充分引用
- 修改成本高（需調整 13 個 rules 檔案）

**成本**：
- 讀者需要 1-2 次跳轉（可接受範圍）

**結論**：現狀設計良好，**不建議主動調整**

---

### 方案 B：小規模合併（可選）

如果確實需要優化，建議只合併低價值檔案（1 次引用 + <150 行）：

1. **`decision-tree-diagrams.md` → 回併 `decision-tree.md`**
   - 成本：編輯 1 個 rules 檔案
   - 收益：減少 1 個 reference 檔案，但 decision-tree.md 會從 600 行增至 745 行（+145 行）
   - **評估**：不推薦（會超過認知負擔閾值）

2. **`query-research-scenario-examples.md` → 回併 `query-vs-research.md`**
   - 成本：編輯 1 個 rules 檔案
   - 收益：減少 1 個 reference 檔案，query-vs-research.md 從 67 行增至 208 行（+141 行）
   - **評估**：推薦（target 檔案較小，且邏輯相關）

3. **保留 `version-decision-case-studies.md` 和 `plan-to-ticket-mapping-details.md`**
   - 理由：雖然引用次數少，但內容複雜，合併會導致父檔案超過認知負擔閾值

**方案 B 成本效益**：
- 工作量：2 個檔案修改
- 收益：減少 1 個 reference 檔案
- 新成本：+141 行內容進入 query-vs-research.md
- **淨收益**：低（不強烈建議）

---

### 方案 C：分類整理（長期優化）

不修改檔案本身，改進導航體驗：

1. 在 `.claude/references/README.md` 中新增分類索引
   - 按頻率分級（高頻/中頻/低頻）
   - 按深度分級（快速參考/詳細文件/技術細節）

2. 在 rules 檔案中新增「快速導航」標記
   - 標記必讀 vs 選讀的 references
   - 標記深度及跳轉成本

3. 建立「使用者路線圖」
   - 新手路線：僅讀 rules，不跳轉
   - 進階路線：讀 rules + 關鍵 references
   - 專家路線：所有檔案

**方案 C 成本效益**：
- 工作量：低（僅加索引和標記）
- 收益：顯著改進閱讀體驗
- **淨收益**：高（強烈推薦）

## 附錄：計算方法

### 認知負擔指數（Cognitive Load Index）

```
CL = (引用次數 / 檔案行數) × 100
```

- CL < 5%：低（✅ 推薦）
- CL 5-10%：中
- CL > 10%：高（❌ 需改進）

**本分析結果**：所有 rules 檔案 CL < 5% ✅

### 多層跳轉深度

```
深度 = 最長路徑: rules → references → references → 內容
```

**本分析結果**：深度 ≤ 3 層，佔總跳轉 < 5% ✅

### 被引用覆蓋度

```
覆蓋度 = 被引用檔案數 / 總檔案數 × 100
```

**本分析結果**：
- rules → references：13/28 = 46% ✅（合理，非所有規則都需要詳細說明）
- references 內部：6/54 = 11% ✅（正常的參考交叉引用）
