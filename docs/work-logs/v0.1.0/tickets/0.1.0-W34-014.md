#### 2. main-thread-edit-restriction-hook.py (L121-149)

```python
def normalize_path(file_path: str) -> str:
    """
    正規化檔案路徑為相對於專案根目錄的路徑

    Args:
        file_path: 原始檔案路徑（可能是絕對或相對路徑）

    Returns:
        str - 正規化後的相對路徑（使用正斜杠）
    """
    # 統一使用正斜杠
    normalized = file_path.replace("\\", "/")

    # 取得專案根目錄（絕對路徑）
    project_dir = os.getenv("CLAUDE_PROJECT_DIR", str(Path.cwd()))
    project_dir = project_dir.replace("\\", "/")

    # 如果是絕對路徑且在專案目錄內，轉換為相對路徑
    if normalized.startswith(project_dir):
        normalized = normalized[len(project_dir):]
        # 移除開頭的斜杠
        if normalized.startswith("/"):
            normalized = normalized[1:]

    # 移除開頭的 ./ 如果有的話
    if normalized.startswith("./"):
        normalized = normalized[2:]

    return normalized
```

**特徵**：
- 參數：file_path (str) 僅一個
- 操作：
  1. 轉換反斜杠
  2. 查詢環境變數 CLAUDE_PROJECT_DIR（fallback to cwd）
  3. 絕對路徑轉相對路徑（strip project dir prefix）
  4. 移除開頭斜杠
  5. 移除 "./" 前綴
- 最全面的實現，處理絕對路徑情況
- 有詳細文件字串和多步驟邏輯

### 語意衝突分析

**衝突點 1：logger 參數**
- ticket-path-guard 需要 logger，但 ticket-file-access-guard 和 main-thread-edit-restriction 不需要
- 調用上下文不同，導致簽名不一致
- ticket-path-guard 的 logger 用於 debug，非核心邏輯

**衝突點 2：絕對路徑處理**
- main-thread-edit-restriction 假設輸入可能是絕對路徑，主動轉為相對
- ticket-path-guard 和 ticket-file-access-guard 假設輸入已是相對路徑
- 如果 main-thread-edit-restriction 接收相對路徑，該邏輯變成無操作（CLAUDE_PROJECT_DIR 匹配失敗）

**衝突點 3：使用上下文差異**
- main-thread-edit-restriction: 檔案編輯限制，可能接收絕對路徑（來自 tool_input）
- ticket-path-guard、ticket-file-access-guard: 只檢查特定 regex pattern，相對路徑即可

### 可行性評估

1. **統一為簡單版本**（ticket-path-guard / ticket-file-access-guard 的邏輯）
   - 缺點：main-thread-edit-restriction 可能收到絕對路徑，需要上游確認 tool_input 格式
   - 風險：未驗證 tool_input 中的 file_path 是否確實為相對路徑

2. **統一為完整版本**（main-thread-edit-restriction 的邏輯）
   - 優點：容納所有輸入格式（相對、絕對、帶 "./"）
   - 缺點：ticket-path-guard 的 logger 參數需要判斷是否必要
   - 風險：環境變數 CLAUDE_PROJECT_DIR 不總是可用

3. **保留差異，改名區分**
   - normalize_path_simple() vs normalize_path_with_absolute()
   - 優點：明確意圖，無需統一邏輯
   - 缺點：增加維護負擔，三個 hook 各有名稱

## Solution

### 建議方案：統一為完整版本（方案 2）

**理由**：
1. **容錯性最佳**：三個函式的使用上下文都是接收 tool_input 中的 file_path，而 tool_input 可能包含絕對路徑（特別是 main-thread-edit-restriction 的場景）
2. **未來擴展性**：若後續新增 hook 也需要路徑規範化，完整版本可直接複用
3. **最小化假設**：不依賴「輸入一定是相對路徑」的假設
4. **logger 參數可選**：可以在簽名中加入 `logger=None` 使其可選

### 提議的統一實現

建立 `.claude/hooks/hook_utils/hook_path.py` 新模組：

```python
def normalize_path(file_path: str, logger=None) -> str:
    """
    正規化檔案路徑為相對於專案根目錄的相對路徑

    處理的情況：
    - 絕對路徑 → 相對路徑（若在專案目錄內）
    - Windows 反斜杠 → Unix 正斜杠
    - ./ 前綴 → 移除

    Args:
        file_path: 原始檔案路徑（絕對或相對）
        logger: 可選日誌物件，用於 debug

    Returns:
        str - 正規化後的相對路徑
    """
    # 統一使用正斜杠
    normalized = file_path.replace("\\", "/")

    # 取得專案根目錄
    project_dir = os.getenv("CLAUDE_PROJECT_DIR", str(Path.cwd()))
    project_dir = project_dir.replace("\\", "/")

    # 如果是絕對路徑且在專案目錄內，轉換為相對路徑
    if normalized.startswith(project_dir):
        normalized = normalized[len(project_dir):]
        if normalized.startswith("/"):
            normalized = normalized[1:]

    # 移除開頭的 ./ 前綴
    if normalized.startswith("./"):
        normalized = normalized[2:]

    if logger:
        logger.debug(f"正規化路徑: {file_path} → {normalized}")

    return normalized
```
