## 1. 測試策略規劃

### 1.1 分層決策樹

本功能跨越三個主要模組，適用不同的測試策略：

| 模組 | 層級 | 複雜度 | 測試策略 | 觸發方式 |
|------|------|--------|---------|---------|
| **模組 A: Context Refresh Handoff** | UseCase/CLI | 高（邏輯決策複雜） | BDD (Given-When-Then) | 命令參數驗證 |
| **模組 B: 路徑常數化** | 構成/工具層 | 低（機械替換） | 單元測試 | 常數導入驗證 |
| **模組 C: set-blocked-by/set-related-to** | UseCase/Domain | 高（複雜驗證邏輯） | 單元測試 + 集成 | 關係管理命令 |

### 1.2 Sociable Unit Tests 原則應用

#### Mock 策略判斷

| 依賴 | Mock 策略 | 理由 |
|------|---------|------|
| `load_ticket()`（外部 I/O） | Mock | 測試隔離，避免檔案系統依賴 |
| `save_ticket()`（外部 I/O） | Mock | 測試隔離，驗證呼叫但不寫檔 |
| `get_project_root()`（環境） | Mock/Fixture | 測試隔離 |
| Ticket 字典對象（Domain） | 真實 | 直接建立，測試行為而非實作 |
| 命令參數解析（Unit） | 真實（Mock argparse） | 簡單純邏輯 |
| 檔案系統（模組 B） | Fixture/臨時檔案 | 常數化需實際驗證導入 |

#### 測試隔離級別

- **Module** = 單個命令函式（handoff.py, track_relations.py）+ 其直接依賴
- **Mock 外部邊界**：load_ticket, save_ticket, get_project_root, 檔案 I/O
- **保持真實**：Ticket dict, 命令參數, 驗證邏輯

## 3. 補充的邊界條件和迴歸測試

### 邊界條件測試

| 場景 | 預期行為 |
|------|---------|
| **空依賴清單** | 移除所有依賴（`--remove` 全部刪除） |
| **自引用** | `blockedBy` 包含自身 → 循環依賴錯誤，拒絕 |
| **雙向依賴** | A blockedBy B 且 B blockedBy A → 檢測到循環，警告或拒絕 |
| **版本不匹配** | 引用其他版本的 Ticket（如 0.2.0-W1-001）→ 拒絕或警告 |
| **特殊字元和空格** | Ticket ID 中含空格或特殊字元 → 驗證和拒絕 |

| 測試項 | 驗證內容 |

## 8. 測試設計決策記錄

### 為什麼選擇 Mock 而非完整集成測試？

**決策**: 對 load_ticket/save_ticket 使用 Mock
- **優點**: 測試執行速度快，隔離檔案系統依賴，便於故障排查
- **缺點**: 可能遺漏文件 I/O 邊界錯誤
- **補償**: 通過 handoff 檔案建立的實際驗證（B-1/B-2）來補償

### 為什麼 C-5 要求 Fail Fast？

**決策**: 在發現不存在的 Ticket 時立即拒絕，不部分修改
- **理由**: Atomic 操作，保証資料一致性，避免中間狀態
- **實作**: 在迴圈前驗證所有被引用 Ticket

### 為什麼分開 set-blocked-by 和 set-related-to？

**決策**: 兩個獨立命令但共用邏輯
- **理由**: 語義清晰，易於使用者理解
- **實作**: 提取共用函式 `_execute_set_relation_field()`，兩個命令呼叫它
