## 一、問題分析

### 1.1 初始問題

**W39-001 Phase 3b 發現的缺陷**：

在 Phase 3b 新增了 `validate_tool_input()` 和 `validate_ticket_unified()` 兩個函式到 hook_utils 時，**Phase 2 設計的測試案例未被實作**。

**具體症狀**：
- 74 個既有測試通過
- 但新函式缺乏測試覆蓋（設計有 23 個測試案例，實作為零）
- 無法確保新函式按規格正常工作
- 無法驗證邊界條件和異常路徑

**影響範圍**：
- `validate_tool_input()` 用於 Hook 輸入驗證（PreToolUse/PostToolUse Hook 依賴）
- `validate_ticket_unified()` 用於 Ticket 驗證（agent-ticket-validation-hook 依賴）
- 缺乏測試導致無法確認這些 Hook 的正確性

### 1.2 根本原因

Phase 3b 實作時執行了程式碼，但在「補充測試」階段未完整執行。Phase 2 設計的 26 個測試案例（包括 Sociable Unit Tests 和行為等價測試）未被實作到測試檔案中。

---

## 二、解決方案

### 2.1 設計原則

遵循 **Sociable Unit Tests** 方法論：
- 使用真實 dict、Logger、Path 物件，不 mock 業務邏輯
- 只 mock 外部依賴（檔案系統用 `tmp_path`、環境用 `monkeypatch`）
- 測試行為而非實作細節

### 2.2 實作策略

**新增 3 個測試類別到 test_hook_utils.py**：

1. **TestValidateToolInput（14 個測試）**
   - 正常路徑：3 個測試（子欄位完整、額外欄位、空字串）
   - 異常路徑：8 個測試（缺失欄位、None 值、型別錯誤）
   - 特殊情況：3 個測試（靜默模式、無要求、空 tuple）

2. **TestValidateTicketUnified（9 個測試）**
   - 正常路徑：2 個測試（有效 Ticket、自動探測）
   - 異常路徑：5 個測試（不存在、缺決策樹、讀取失敗、空檔案、空 ID）
   - 特殊情況：2 個測試（靜默模式、非標準格式）

3. **TestImportsAndExports（3 個測試）**
   - 驗證新函式可匯入
   - 驗證既有符號保留（向後相容）

### 2.3 實作遵循的規範

- **Phase 2 設計的 100% 遵循**：所有 23 個 GWT 場景、20 個邊界情況、13 個驗收標準都實現為具體測試
- **PEP 8 和 Python 品質規則**：遵循 Python 品質規則補充（quality-python.md）
- **Sociable Unit Tests**：複用既有 Fixture，不破壞隔離性
