# Phase 2 測試設計：Hook 共用模式統一

**對應 Ticket**: 0.1.0-W39-001
**TDD 階段**: Phase 2 測試設計
**設計者**: sage-test-architect
**基礎**: Phase 1 功能設計文件 (0.1.0-W39-001-phase1-design.md)

---

## 一、測試設計概述

### 1.1 測試目標

基於 Phase 1 設計的 9 個 GWT 場景和 13 個驗收標準（A1-A11, B1-B3），設計完整的單元測試和行為等價測試，確保：

1. **新增 API 的正確性**：`validate_tool_input()` 和 `validate_ticket_unified()` 按規格正常工作
2. **邊界條件完整**：所有邊界情況和錯誤路徑都有測試覆蓋
3. **行為等價性**：修改後的 Hook 與原實作行為一致
4. **向後相容性**：現有 Hook 和 API 保持不變

### 1.2 測試策略

採用 **Sociable Unit Tests** 原則（來自 behavior-first-tdd-methodology.md）：

- **不 Mock Domain Entity**：使用真實的 input_data dict、logger、Path 物件
- **只 Mock 外部依賴**：
  - 檔案系統：使用 `pytest.fixture` 的 `tmp_path`
  - 時間：使用 `unittest.mock.patch` 固定時間戳（如必要）
  - 環境變數：使用 `monkeypatch`
  - Hook 底層 API（如 `find_ticket_file`）：必要時 mock
- **測試行為而非實作**：測試 public API 的行為契約，不依賴內部實作細節

### 1.3 測試覆蓋率目標

| 面向 | 目標 | 說明 |
|------|------|------|
| 函式覆蓋 | 100% | 兩個新函式及修改的函式都要測試 |
| 代碼路徑 | 100% | 所有 if/else 分支都要覆蓋 |
| 邊界條件 | 100% | Phase 1 第五節識別的所有邊界都要測試 |
| 場景覆蓋 | 100% | Phase 1 第六節的 9 個 GWT 場景都要實現 |

## 二、測試檔案結構

### 2.1 新增測試檔案清單

| 檔案位置 | 用途 | 測試類型 |
|---------|------|---------|
| `.claude/hooks/tests/test_hook_io_validate_tool_input.py` | `validate_tool_input()` 單元測試 | 單元測試（Sociable） |
| `.claude/hooks/tests/test_hook_ticket_validate_ticket_unified.py` | `validate_ticket_unified()` 單元測試 | 單元測試（Sociable） |
| `.claude/hooks/tests/test_behavior_equivalence.py` | 修改前後行為等價測試 | 整合測試 |
| `.claude/hooks/tests/test_imports.py` | `__init__.py` re-export 測試 | 靜態測試 |

### 2.2 現有測試檔案影響

| 檔案 | 修改 | 說明 |
|------|------|------|

| `conftest.py` | 無修改 | 複用既有 fixture（`project_root`, `mock_env_var`, `reset_loggers`） |

### 3.1 `validate_tool_input()` 測試（12 個案例）

#### 文件：`.claude/hooks/tests/test_hook_io_validate_tool_input.py`

**測試類別**：`TestValidateToolInput`

**前置條件**：
- 每個測試都使用 `@pytest.fixture` 提供的 `logger`（真實 logging.Logger）
- 使用真實 dict 物件模擬 `input_data`，不 mock dict 本身

#### 3.1.1 正常路徑測試（Case A1-A3）

**Case A1: 場景 1 — 正常流程（子欄位完整）**

```
Test ID: test_validate_tool_input_normal_all_fields_present
Given:
  - input_data = {"tool_name": "Write", "tool_input": {"file_path": "test.md", "content": "hello"}}
  - required_subfields = ("file_path", "content")
  - logger: 真實 Logger
When:
  - validate_tool_input(input_data, logger, required_subfields) 被呼叫
Then:
  - 回傳 True
  - logger 記錄 debug level "tool_input 子欄位驗證通過"
  - 無 error 級別訊息
```

驗證點：
- [ ] 回傳值型別為 bool
- [ ] 回傳值為 True
- [ ] logger.debug 被呼叫（檢查 logger 記錄）
- [ ] logger.error 未被呼叫

**Case A2: 場景 1 變體 — 多於 2 個子欄位**

```
Test ID: test_validate_tool_input_normal_extra_fields
Given:
  - input_data = {"tool_name": "Write", "tool_input": {"file_path": "x.md", "content": "x", "extra": "y"}}
  - required_subfields = ("file_path", "content")
When:
  - validate_tool_input(input_data, logger, required_subfields) 被呼叫
Then:
  - 回傳 True（額外欄位不影響）
  - logger 記錄 debug "tool_input 子欄位驗證通過"
```

驗證點：
- [ ] 不要求 tool_input 只有指定欄位

**Case A3: 場景 1 變體 — 空字串值（非 None）**

```
Test ID: test_validate_tool_input_normal_empty_string_value
Given:
  - input_data = {"tool_name": "Write", "tool_input": {"file_path": "", "content": "text"}}
  - required_subfields = ("file_path", "content")
When:
  - validate_tool_input(input_data, logger, required_subfields) 被呼叫
Then:
  - 回傳 True（空字串通過，不視為 None）
  - logger 記錄 debug "tool_input 子欄位驗證通過"
```

驗證點：
- [ ] 空字串（falsy but not None）應通過
- [ ] 只檢查 None，不檢查 falsy 值

#### 3.1.2 異常路徑測試（Case B1-B8）

**Case B1: 場景 2 — 子欄位缺失**


驗證點：

**Case B2: 場景 2 變體 — 多個子欄位缺失（回傳第一個發現的）**

```
Then:
  - 回傳 False

```

驗證點：
- [ ] 偵測多欄位缺失
- [ ] 早期返回（發現第一個缺失即返回）

**Case B3: 場景 3 — tool_input 整體缺失**

```
Test ID: test_validate_tool_input_missing_tool_input_field
```

### 3.3 行為等價測試（2 個整合測試）

#### 文件：`.claude/hooks/tests/test_behavior_equivalence.py`

**測試類別**：`TestBehaviorEquivalence`

目標：確認修改後的 Hook 與原實作行為等價。

#### 3.3.1 場景 8 — ticket-quality-gate-hook 修改前後等價

```
Test ID: test_ticket_quality_gate_hook_equivalence
Given:
  - input_data = {
      "tool_name": "Write",
      "tool_input": {"file_path": "test.md", "content": "content"}
    }
  - logger: 真實 Logger
When:
  - 修改前 validate_input(): 呼叫 validate_hook_input() + 手動子欄位驗證
  - 修改後 validate_input(): 呼叫 validate_hook_input() + validate_tool_input()
Then:
  - 兩種實作對相同輸入回傳相同 bool 值
  - 兩種實作的 logger 行為相同
```

驗證方式：
- [ ] 測試 10 個不同的 input_data（正常、子欄位缺失、tool_input 缺失等）
- [ ] 對每個 input_data，檢查回傳值相同
- [ ] 檢查 logger 記錄的錯誤訊息相同

實作細節：
- 在測試中同時定義修改前後的 validate_input() 版本
- 對相同輸入呼叫兩個版本，比對結果

#### 3.3.2 場景 9 — agent-ticket-validation-hook 修改前後等價

```
Test ID: test_agent_ticket_validation_hook_equivalence
Given:
  - ticket_id = "0.1.0-W1-001"
  - Ticket 檔案存在且有效
  - logger: 真實 Logger
When:
  - 修改前 validate_ticket(): find_ticket_file + read + validate_decision_tree
  - 修改後 validate_ticket(): validate_ticket_unified(ticket_id, None, logger)
Then:
  - 兩種實作對相同 ticket_id 回傳相同 (bool, Optional[str]) tuple
  - 兩種實作的 logger 行為相同
```

驗證方式：
- [ ] 測試 6 個不同的 ticket_id（有效、不存在、缺少決策樹等）
- [ ] 對每個 ticket_id，檢查回傳 tuple 相同

## 四、Mock 和 Fixture 策略

### 4.1 Sociable Unit Tests 原則應用

| 元件 | Mock | 理由 |
|------|------|------|
| `dict` / `input_data` | 否 | Domain 內部邏輯 |
| `logging.Logger` | 否 | 真實 Logger，測試日誌行為 |
| `Path` / 檔案系統 | 是（用 tmp_path） | 外部依賴，使用 pytest fixture |
| 時間戳 | 是（patch datetime） | 外部依賴（非必要，除非驗證檔名） |
| 環境變數 | 是（monkeypatch） | 外部依賴 |
| `find_ticket_file()` | 視情況 mock | 行為等價測試中需要 mock 原版本 |
