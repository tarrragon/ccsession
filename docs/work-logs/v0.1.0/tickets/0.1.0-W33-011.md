### 3. 共用 Utilities 放置策略

分析後，`get_project_root` 和 `run_git` 是最常被跨模組引用的底層 utility：

- `get_project_root`：被 hook_logging（setup_hook_logging、save_check_log）和 hook_ticket（find_ticket_file、find_ticket_files）同時需要。
- `run_git`：僅被外部 hook 直接使用（4 個），不涉及子模組間依賴。

**決策**：`get_project_root` 放入 `hook_logging.py`，hook_ticket 透過 `from .hook_logging import get_project_root` 引用。

**理由**：hook_logging 是最底層的模組（setup_hook_logging 是所有 hook 的第一個呼叫），放在此處可確保依賴方向單向（hook_ticket 依賴 hook_logging，不反向）。

### 4. 子模組間依賴圖

```
hook_logging.py
  └── 無外部 hook_utils 子模組依賴（純標準庫）

hook_ticket.py
  └── from .hook_logging import get_project_root
  （hook_ticket 中的 find_ticket_file 需要呼叫 get_project_root）

hook_io.py
  └── 無 hook_utils 子模組依賴（純標準庫 + sys/json）
```

依賴方向：hook_ticket → hook_logging（單向，無循環）

**循環依賴風險**：無。hook_logging 不引用 hook_ticket 或 hook_io，hook_io 不引用任何子模組。

## Solution

### 實作策略

基於 Phase 1 功能設計，拆分方式如下：

#### 目標目錄結構

```
.claude/hooks/hook_utils/
├── __init__.py       # re-export 所有公開 API，向後相容
├── hook_logging.py   # 日誌相關（9 個內部 + 4 個公開）
├── hook_ticket.py    # Ticket 操作相關（11 個公開）
└── hook_io.py        # I/O 操作相關（4 個公開）
```

#### 各子模組職責

**hook_logging.py（日誌與基礎設施）**：
- 包含：get_project_root, setup_hook_logging, save_check_log, run_hook_safely
- 包含所有內部輔助函式（_find_project_root 等 8 個）
- 包含所有日誌常數（TIMESTAMP_FORMAT、FILE_FORMAT 等）

**hook_ticket.py（Ticket 操作）**：
- 包含：parse_ticket_frontmatter, parse_ticket_date, check_error_patterns_changed, get_current_version_from_todolist, scan_ticket_files_by_version, find_ticket_files, find_ticket_file, extract_version_from_ticket_id, extract_wave_from_ticket_id, validate_ticket_has_decision_tree, _parse_version_from_ticket_id
- 依賴：from .hook_logging import get_project_root
- 包含：DECISION_TREE_MARKERS 常數

**hook_io.py（I/O 操作）**：
- 包含：run_git, read_json_from_stdin, extract_tool_input, extract_tool_response
- 無子模組依賴（純標準庫）

#### 向後相容保證

__init__.py 重新匯出全部 19 個公開 API，使現有 54 個 hook 的 `from hook_utils import ...` 語句**無需任何修改**即可繼續運作。

### 4. 風險管理和回滾策略

#### 主要風險點

**風險 1：循環 import**
- 發生場景：hook_logging 誤引用 hook_ticket，導致循環
- 防護方式：每步完成後驗證 `from hook_utils.submodule import X` 能否獨立成功
- 回滾：不搬移該函式，留在原檔；或調整依賴關係

**風險 2：__init__.py 不完整**
- 發生場景：漏掉某個公開 API，54 個 hook 中某個找不到函式
- 防護方式：__init__.py 中的 `__all__` 必須與「待搬移清單」一致（見 Problem Analysis）
- 回滾：補入漏掉的 import，再驗證

**風險 3：函式簽名變更**
- 發生場景：搬移時不小心改了函式參數或返回值
- 防護方式：搬移後立即執行對應單元測試，確保行為不變
- 回滾：對比舊檔案，逐行檢查，恢復原簽名

**風險 4：內部函式位置錯誤**
- 發生場景：hook_ticket 中的 find_ticket_file 呼叫 _find_project_root，但 _find_project_root 搬到 hook_logging 了
- 防護方式：搬移前掃描所有內部函式呼叫，確認依賴位置
- 處理方式：hook_ticket 加 `from .hook_logging import _find_project_root` 或改呼叫公開 API `get_project_root`

---

#### 回滾步驟

若任意步驟失敗：

1. 停止當前步驟
2. 刪除已建立的子模組檔案（保留舊 hook_utils.py）
3. 執行現有全量測試驗證舊檔案仍可用：`pytest .claude/hooks/tests/test_hook_utils.py`
4. 預期：全部通過，系統恢復

**回滾範圍**：
- 若第 1 步失敗（hook_logging）：刪除 hook_utils/ 目錄，恢復 hook_utils.py
- 若第 2 步失敗（hook_io）：保留 hook_logging，刪除 hook_io，重試
- 若第 4 步失敗（__init__.py）：保留 3 個子模組，修正 __init__.py 後重試
