### 2. last_stale_count 介面評估與改善

#### 替代方案評估

##### 方案 A：維持現狀（函式屬性）

```python
# 現行做法
list_pending_handoffs.last_stale_count = stale_count  # 寫入
stale_count = list_pending_handoffs.last_stale_count   # 讀取
```

**評估結果**：

| 維度 | 評分 |
|------|------|
| 可讀性 | ☆☆☆ 低 |
| Python 慣例 | ☆☆☆ 非標準 |
| 類型安全 | ☆☆☆ 無 |
| 呼叫端數量 | 1 處（低風險） |
| 向後相容性 | ★★★ 完全相容 |

**缺點**：
- 函式屬性是 Python 的隱藏機制，讀者看到 `list_pending_handoffs()` 不會預期有額外屬性
- 無法被 IDE 自動補全、類型檢查工具識別
- 作為「狀態儲存」的副作用，違反函式單一責任原則

##### 方案 B：改為 namedtuple 返回值（**選定方案**）

```python
from collections import namedtuple

HandoffListResult = namedtuple(
    "HandoffListResult",
    ["handoffs", "stale_count", "schema_error_count"]
)

# 改善後
result = list_pending_handoffs()
stale_count = result.stale_count  # 明確的欄位訪問
```

**評估結果**：

| 維度 | 評分 |
|------|------|
| 可讀性 | ★★★ 高 |
| Python 慣例 | ★★★ 標準做法 |
| 類型安全 | ★★★ 具名欄位 |
| 呼叫端數量 | 1 處（低風險） |
| 向後相容性 | ☆☆☆ 破壞相容 |

**優點**：
- 明確的返回型別，讀者能清晰了解函式返回的資料結構
- IDE 能自動補全 `result.` 欄位名稱
- 消除了函式屬性的隱藏副作用，提高程式碼清晰度
- 標準 Python 做法（類似 `os.stat_result`）

##### 方案 C：改為 dataclass

```python
from dataclasses import dataclass

@dataclass
class HandoffListResult:
    handoffs: List[Dict[str, Any]]
    stale_count: int
    schema_error_count: int
```

**評估結果**：與方案 B 類似優缺點，但更冗長。namedtuple 足以應付需求。

#### 選定方案理由

**選擇方案 B（namedtuple）**，原因：

1. **低風險**：整個系統只有 1 個核心呼叫端（`_execute_list()` 函式），修改影響有限。
2. **高收益**：消除設計缺陷（函式屬性隱含副作用），提升程式碼品質。
3. **標準化**：namedtuple 是 Python 標準做法，易於理解和維護。
4. **測試覆蓋**：現有測試已覆蓋此功能，改善後測試更新簡單（6 處呼叫點）。
