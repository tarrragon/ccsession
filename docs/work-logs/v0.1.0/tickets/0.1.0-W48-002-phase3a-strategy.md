### 2.1 DECISION_KEYWORDS 字典擴展

**現有結構**：
```
DECISION_KEYWORDS = {
    (keyword1, keyword2): "scenario_name",  # 組合關鍵字（2 個詞）
    (keyword,): "scenario_name",             # 單一關鍵字（1 個詞）
}

**新增條目**（3 組組合關鍵字）：
```
("順序", "任務"): "執行方向確認",
("接下來", "做"): "執行方向確認",
("先後", "執行"): "執行方向確認",

**為何選擇這個結構**：
- 與現有 ACTION_KEYWORDS 和 HANDOFF_KEYWORDS 結構一致（複用 check_decision_keywords 函式邏輯）
- 雙關鍵字組合降低誤觸發率（相比單一關鍵字）
- 字典查詢複雜度 O(1)，不影響性能

### 2.2 否定詞處理機制（複用現有）

**現有實作**：`_is_keyword_negated(prompt: str, keyword: str) -> bool`
- 檢查單個關鍵字是否被否定詞修飾
- 支援遠距否定詞（15 字符窗口內）
- 無需修改，直接複用於場景 #8 關鍵字檢查

**呼叫位置**：`check_decision_keywords()` 函式內部

### 7.2 已知限制

1. **否定詞窗口固定 15 字符**
   - 限制：遠距否定詞超過 15 字符時無法識別
   - 影響：極少數複雜句子會誤觸發提醒
   - 接受：此限制符合設計意圖（合理的窗口大小）

2. **字典迭代順序依賴 Python 3.7+ 保證**
   - 限制：若運行 Python 3.6 或更早版本，關鍵字匹配順序不確定
   - 影響：極低（現代 Python 版本普遍 3.7+，專案使用 3.11）
   - 接受：無需特殊處理，符合項目約束

3. **組合關鍵字無優先級控制**
   - 限制：若同時匹配多個場景，回傳第一個匹配的（字典迭代順序）
   - 影響：Phase 1 設計確認新增關鍵字與現有無重疊，故無此問題
   - 驗證：邊界測試 D1-D2 驗證此點

### 7.3 技術債務標記

**優先級：無重大技術債務**

實作中可能發現的後續改善機會（留給 Phase 4）：

1. **關鍵字管理集中化**
   - 現狀：關鍵字硬編碼在 Hook 檔案中
   - 建議：考慮在未來版本提取到 `constants.py` 或配置檔案
   - 優先級：低（當前規模小，無需提取）

2. **否定詞窗口大小參數化**
   - 現狀：NEGATION_WINDOW_SIZE 為硬編碼常數
   - 建議：考慮根據不同場景調整窗口大小
   - 優先級：低（15 字符窗口已足夠）

3. **測試案例複雜度**
   - 現狀：35 個測試案例採用簡單單元測試方法
   - 建議：考慮在 Phase 4 重構時引入參數化測試（pytest.mark.parametrize）
   - 優先級：低（當前案例清晰，無需參數化）

### 7.4 Phase 4 建議方向

1. **測試案例參數化**：使用 `pytest.mark.parametrize` 減少測試代碼重複
2. **性能優化**：若關鍵字數量增多（>100 個），考慮使用 Trie 樹等高效資料結構
3. **關鍵字管理**：若新增更多場景，考慮統一管理關鍵字配置
