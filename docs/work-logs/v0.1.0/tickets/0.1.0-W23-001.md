## Solution

### 問題 1 修復：check_acceptance_status 四步驟全部執行不提前 return

**位置**：710-714 行（原版本）

**修復方式**：
- 移除 `if warning_msg: return ...` 的提前 return 邏輯
- 改為繼續執行步驟 3（error-pattern 檢查）和步驟 4（sibling tickets 掃描）
- 將 `warning_msg` 帶入最終 AcceptanceCheckResult 回傳值

**關鍵改動**：
```python
# 修復前（714 行）
if warning_msg:
    return AcceptanceCheckResult(False, False, warning_msg, False, [])

# 修復後（733 行）
return AcceptanceCheckResult(
    False,
    has_acceptance,
    warning_msg,  # 將 warning_msg 帶入最終結果（不提前 return）
    has_new_error_patterns,
    new_error_pattern_files,
    pending_siblings,
    task_type,
    priority,
)
```

**影響**：場景 #17（error-pattern）和場景 #9（sibling tickets）現在全部執行，不會被提前 return 跳過。

### 問題 2 修復：generate_hook_output 場景 #2 在場景 #9 路徑也附加

**位置**：839-841 行（原版本）

**修復方式**：
- 場景 #2 從 `if len(...) < 2` 的條件塊移出
- 改為無條件觸發（僅判斷 `not check_result.message`）
- 即使有 2+ sibling（場景 #9），也會附加場景 #2 下一步提醒

**關鍵改動**：
```python
# 修復前（839-841 行，在 if len(...) < 2 條件塊內）
if (not check_result.message and len(...) < 2):
    # ...場景 #1 邏輯...
    context_parts.append(AskUserQuestionMessages.COMPLETE_NEXT_STEP_REMINDER)

# 修復後（840-842 行，在 if len(...) < 2 條件塊外）
if not check_result.message:
    context_parts.append(AskUserQuestionMessages.COMPLETE_NEXT_STEP_REMINDER)
```

**影響**：場景 #9（2+ sibling 時）現在也會收到場景 #2 的下一步提醒。

### 問題 3 修復：frontmatter 每次 complete 只解析一次

**修復方式**：
- `get_ticket_status()` 和 `get_ticket_type()` 改為接收已解析的 frontmatter，而非 content
- `get_ticket_created_time()` 改為接收已解析的 frontmatter，避免重複讀取檔案
- `_check_error_patterns()` 改為接收已解析的 frontmatter，而非 ticket_file

**關鍵改動**：
- 698-701 行：在 check_acceptance_status 中一次讀取和解析
- 721 行：傳遞 frontmatter 而非 ticket_file 給 _check_error_patterns
- 函式簽名更新：所有函式接收 `frontmatter: Dict[str, str]` 而非檔案路徑

**影響**：
- 單次 complete 只讀取一次 ticket 檔案
- 只解析一次 frontmatter
- `has_acceptance_record()` 仍呼叫 2 次，但基於已讀取的 content，未造成額外磁碟 I/O

### 附加修復：stderr 輸出

在 except 區塊新增 stderr 輸出，符合規範要求：
- 742-743 行：check_acceptance_status 異常時
- 523 行：get_ticket_created_time 異常時
