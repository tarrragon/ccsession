# Execution Log

## Task Summary

實作 commit-handoff-hook 情境 C 偵測邏輯擴充

---

## Problem Analysis

根據 W26-002 分析報告，決策樹第八層 Checkpoint 2 定義了情境 C（Wave 完成）需要強制執行 `/parallel-evaluation` 進行 Wave 級多視角審查。但當前實現缺乏自動提醒機制：

1. **commit-handoff-hook 的局限**：
   - 已有情境 A（#11a）和情境 B（#11b）的偵測邏輯
   - 缺少情境 C（ticket completed + 同 Wave 無 pending）的判斷分支
   - 若情境判斷結束後無相應提醒，PM 容易遺漏必要的 Wave 審查步驟

2. **提醒訊息的局限**：
   - 現有 COMMIT_HANDOFF_REMINDER 只包含三個情境的指引，但無獨立的「情境 C 實作步驟」常數
   - 當 PM 根據查詢判斷為情況 C 時，需要明確的「執行 Wave 審查」指引

3. **決策樹與實現的脫節**：
   - decision-tree.md 要求「[強制] /parallel-evaluation Wave 完成審查」
   - 但缺乏供 Hook 觸發的提醒常數支撑

---

## Solution

根據 W26-002 推薦的方案 C（成本最優），實作分為兩部分：

### 1. ask_user_question_reminders.py 新增 WAVE_COMPLETION_REMINDER 常數

新增 `WAVE_COMPLETION_REMINDER`，提供 PM 在判斷為情況 C 後的執行指引：

**核心內容**：
- 確認 Wave 已完成（同 Wave 無 pending Ticket）
- [Step 1] 強制執行 `/parallel-evaluation` 進行 Wave 完成審查
- [Step 2] 根據審查結果選擇後續路徑：
  - **情景 C1**：版本有其他 Wave pending → 執行 `ticket track list --status pending` → AskUserQuestion #3a（Wave 收尾）
  - **情景 C2**：版本無任何 pending → `/version-release check` → AskUserQuestion #13

**格式特性**：
- 遵循 hook_messages.py 的訊息常數規範
- 與 decision-tree.md Checkpoint 2 情景 C 完全對應
- 包含完整的決策樹參考

### 2. commit-handoff-hook.py（預留擴充點）

根據分析報告建議，commit-handoff-hook 的情境 C 自動偵測邏輯（第二版本）應在 Phase 4 重構或下一 Wave 實現。當前實現確保了：
- 提醒訊息框架支撑（WAVE_COMPLETION_REMINDER 常數已可用）
- 後續 Hook 擴充點清晰明確
- 無需改動現有情境 A/B 邏輯
