## Solution

### 5 個測試場景覆蓋率統計

基於靜態分析（規則覆蓋程度對照），驗證 rules 檔案是否已包含 CLAUDE.md 的所有內容：

**場景 1 - 錯誤發生時的強制流程** (測試失敗)
- CLAUDE.md 說：「/pre-fix-eval → incident-responder」
- rules 說：incident-response.md 完整流程 + 三階段詳細說明
- 覆蓋程度：100% ✓ **安全去重**

**場景 2 - Phase 4 不可跳過驗證** (Phase 3b 完成)
- CLAUDE.md 說：「不可跳過」
- rules 說：quality-baseline.md 規則 2 + tdd-flow.md 豁免條件
- 覆蓋程度：60% △ **需保留摘要**（缺豁免條件和最小產出）

**場景 3 - 測試通過率 100% 驗收** (提交前檢驗)
- CLAUDE.md 說：「100%，不通過禁止提交」
- rules 說：quality-baseline.md 規則 1 詳細表格 + 3 個場景細分
- 覆蓋程度：100% ✓ **安全去重**

**場景 4 - Ticket 必須用 /ticket create** (建立新任務)
- CLAUDE.md 說：「禁止直接 Write .md」
- rules 說：ticket-lifecycle.md 包含建立後強制審核流程
- 覆蓋程度：70% △ **需保留摘要**（缺建立後審核細節）

**場景 5 - Skip-gate Level 1-3 禁止行為** (PM 直接修復)
- CLAUDE.md 說：三層禁止行為表格
- rules 說：skip-gate.md 有完整 Hook 實作和路徑清單
- 覆蓋程度：75% △ **需保留摘要**（缺 Level 2/3 細節實施）

**平均覆蓋率：81%**

### 規則權重評估結論

**結論：規則權重 中等下降風險（-19%）**

原因：
1. 25% 的規則場景覆蓋完整（場景 1/3 可安全去重）
2. 75% 的規則場景缺少關鍵細節（場景 2/4/5 需補償）
3. 核心依賴 Hook 系統的決策不受影響，但自主決策場景有風險

### 風險等級分類

#### 高風險（必須補償）

**1. Hook 失敗必須可見規則**
- CLAUDE.md 完全未提及此規則
- rules 位置：quality-baseline.md 規則 4
- 風險：開發者若未主動查閱 rules，將完全遺漏此規則
- **補償方式**：CLAUDE.md 保留 1 句摘要 + 「詳見 quality-baseline.md 規則 4」

**2. Phase 4 豁免條件**
- CLAUDE.md 只有「不可跳過」，完全無豁免細節
- rules 位置：tdd-flow.md 有 3 個豁免場景（遷移任務、小修改、單純範圍）
- 風險：遇到豁免場景時，開發者無法判斷
- **補償方式**：CLAUDE.md 補充「豁免條件詳見 tdd-flow.md」

#### 中風險（建議補償）

**3. Ticket 建立後審核強制流程**
- CLAUDE.md 只說「禁止直接 Write」
- rules 位置：ticket-lifecycle.md 有「建立後強制並行審核（acceptance-auditor + system-analyst）」
- 風險：新建 Ticket 可能跳過審核
- **補償方式**：CLAUDE.md 補充 1 句「建立後需通過 acceptance-auditor + system-analyst 審核」

**4. Skip-gate Level 2/3 細節**
- CLAUDE.md 有表格概覽但缺細節
- rules 位置：skip-gate.md 有完整 Hook 實作、驗證條件、路徑清單
- 風險：PM 執行開發命令時，無法知道詳細驗證條件
- **補償方式**：CLAUDE.md 保留表格，補充「詳細 Hook 實作詳見 skip-gate.md」

#### 低風險（可安全去重）

- 標準錯誤流程、測試通過率 100% 等（rules 有完整相同內容）

### 補償策略建議（推薦方案：「去重 + 補償」）

**執行方式**（4 步）：

**Step 1: 移除品質基線章節重複**
```markdown
## 品質基線

專案品質標準詳見 `.claude/rules/core/quality-baseline.md`，
包括：測試通過率 100% 要求、Phase 4 不可跳過、設計問題發現即修正、Hook 失敗可見性。

重要豁免：Phase 4 在特定場景（小型修改、遷移任務、單純範圍）可豁免，詳見 `.claude/rules/flows/tdd-flow.md`。
```
減少行數：20 行 → 4 行（節省 16 行）

**Step 2: 精簡 Skip-gate 章節**
```markdown
## Skip-gate 防護機制

| 層級 | 說明 |
|------|------|
| Level 1 | 錯誤發生時強制派發 incident-responder，禁止直接修復 |
| Level 2 | 開發命令執行前驗證 Ticket 存在性（Hook 自動檢查） |
| Level 3 | PM 只能編輯允許列表中的檔案路徑 |

詳細規則、違規判定、Hook 實作：詳見 `.claude/rules/forbidden/skip-gate.md`
```
減少行數：9 行 → 6 行（節省 3 行）

**Step 3: 簡化 Ticket 工作流**

## Ticket 工作流

Ticket 建立後需通過 acceptance-auditor + system-analyst 並行審核（`creation_accepted: true`）。

完整生命週期、驗收流程、錯誤處理：詳見 `.claude/rules/flows/ticket-lifecycle.md`

**Step 4: 簡化 TDD 流程摘要**

## TDD 流程

小型修改、遷移任務等可豁免 Phase 4，詳見 `.claude/rules/flows/tdd-flow.md`

Phase 詳細代理人和流程：見 tdd-flow.md
