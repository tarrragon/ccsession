## 孤立 handoff 檔案的成因

### Root Cause 1（已在 W2-018 修復）
_execute_resume() 缺少 archive_handoff_file() 呼叫。
- 問題：ticket resume X 執行後，X.json 仍殘留在 pending/，不會被移至 archive/
- 影響：已恢復的 handoff 在每次 session 重複出現在 resume --list
- 修復：commit 14efc49，W2-018 新增 archive_handoff_file() 呼叫

### Root Cause 2（現仍存在的缺口）
list_pending_handoffs() 對任務鏈 handoff 的過濾邏輯不完整。

當前邏輯：to-sibling/to-parent/to-child 類型，只要 source ticket completed → 一律保留在列表
缺失邏輯：若 direction 指定了目標 ticket（如 to-sibling:0.1.0-W3-009），且目標 ticket
已 in_progress 或 completed，則該 handoff 應視為 stale 並過濾。

實際案例：
- W3-008.json：direction=to-sibling:0.1.0-W3-009，from_status=completed，resumed_at=null
- W3-008 ticket：已 completed
- W3-009 ticket：已 in_progress（本 session 已認領）
- 現象：resume --list 仍顯示 W3-008 為待恢復任務 → 誤導

### Root Cause 3（使用者行為）
使用者跳過 ticket resume，直接 ticket track claim 目標 ticket，
導致 source handoff 的 resumed_at 永遠是 null 且不被歸檔。

## 修復方案：增強 list_pending_handoffs() 目標狀態檢查

### 核心邏輯修改

在 list_pending_handoffs() 的任務鏈過濾區塊，新增目標 ticket 狀態檢查：

現有程式碼（有缺口）：
  if _is_task_chain_direction(direction):
      handoffs.append(data)  # 一律保留，不管目標是否已啟動
      continue

建議修改：
  if _is_task_chain_direction(direction):
      # 新增：若有指定目標 ticket，且目標已 in_progress/completed → stale
      target_id = direction.split(':')[1] if ':' in direction else None
      if target_id and _is_ticket_in_progress_or_completed(target_id):
          continue  # 目標已啟動，source handoff 已無效
      handoffs.append(data)
      continue

### 新增輔助函式 _is_ticket_in_progress_or_completed(ticket_id)
類似現有的 _is_ticket_completed()，但也匹配 in_progress 狀態。

### 效果
- W3-008 handoff（to-sibling:W3-009）：W3-009 已 in_progress → 自動過濾，不再出現
- 無目標指定（舊格式 to-sibling）：保持現有行為（不過濾）
