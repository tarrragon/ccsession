### 視角 1：CLI 邏輯分析

**分析結論**

1. **CLI 是否有 bug**：否

   CLI 的狀態驗證邏輯（`_verify_handoff_status()`）正確，允許的狀態規則為：
   - 有 children 的父任務：in_progress 或 completed 都允許
   - 無 children 的獨立任務：
     - --context-refresh：僅允許 in_progress
     - 普通 handoff：僅允許 completed

2. **根本問題**：PM 誤用 --context-refresh 旗標

   `--context-refresh` 語義是「在同一 session 內刷新 context 並返回同一 ticket」，用於：
   - ticket 仍 in_progress
   - 代理人切換或較長工作後需要重載 ticket 資訊
   - 典型場景：Agent A 完成部分工作 → handoff context → Agent B 接手時用 --context-refresh
   
   PM 試圖在 completed ticket 上用 --context-refresh，違背此旗標語義。

3. **正確做法**：completed ticket 應用 auto 或明確方向

   對於 0.1.0-W11-001（status=completed，無 children）：

   **選項 A：自動判斷（推薦）**
   ```bash
   ticket handoff 0.1.0-W11-001
   ```
   CLI 自動分析 ticket 的 chain 資訊，判斷應 to-parent、to-sibling、to-child 還是待機

   **選項 B：明確指定方向（若 PM 知道下一步）**
   ```bash
   ticket handoff 0.1.0-W11-001 --to-parent
   ticket handoff 0.1.0-W11-001 --to-sibling <id>
   ```

4. **需要修正的地方**：決策樹/規則文件增補（非 CLI 修正）

   - .claude/rules/core/decision-tree.md 需增補「Checkpoint 2 completed ticket handoff 方向選擇指南」
   - 明確說明：
     * context-refresh 專用於 in_progress
     * completed ticket 只能用 --to-parent/--to-sibling/--to-child 或 auto
     * 決策規則：有 children → to-child；無但有 parent → to-parent；有 sibling → to-sibling；否則 auto

**驗證基礎**（根據 handoff.py 第 82-144 行 _verify_handoff_status）

ticket 0.1.0-W11-001 特徵：
- status: completed
- children: [] （無子任務）
- 屬於「無 children 的獨立任務」

因此允許的命令：
- ✓ `ticket handoff 0.1.0-W11-001` （auto）
- ✓ `ticket handoff 0.1.0-W11-001 --to-parent` （若有 parent）
- ✓ `ticket handoff 0.1.0-W11-001 --to-sibling <id>` （若有 sibling）
- ✗ `ticket handoff 0.1.0-W11-001 --context-refresh` （錯誤，completed 狀態不支援）

**結論**：PM 決策需修正，而非 CLI 有 bug。核心缺陷是決策樹文件未清楚說明 context-refresh 的適用場景。

視角 2：PM 決策根因分析

## 根本原因

PM 在 completed ticket (W11-001) 執行 handoff 時，誤選 `--context-refresh` 旗標，應該使用 `--to-sibling` 或無旗標（自動偵測）。

根本原因是**規則邊界與文件交叉引用的缺陷**：

1. **context-refresh 應用域名不明確**
   - AskUserQuestion #11a（Checkpoint 2 情境 A）提到「Context 刷新 Handoff」並說「在新 session 以乾淨 context 繼續同一 ticket」
   - 但 #11a 只適用於 **in_progress** ticket（line 208, askuserquestion-rules.md）
   - 當 PM 在 **completed** ticket 上進行 handoff 時（情境 B/C），決策樹沒有明確禁止或警告 context-refresh 不適用

2. **AskUserQuestion #11b 選項描述過於抽象**
   - #11b（情境 B：ticket completed + 同 Wave 有 pending）的選項只寫「Handoff 到 {next_ticket_id}」
   - 沒有列舉具體可用的 handoff 旗標（--to-sibling / --to-parent / --to-child）
   - 導致 PM 不知道有哪些選擇，被迫自行猜測或誤用 in_progress 的選項

3. **Handoff 命令文件缺乏警告**
   - handoff-command.md 的「五種情境」表格（line 43-52）沒有明確指出哪些旗標適用哪些 ticket 狀態
   - 沒有「禁止行為」章節警告 completed ticket 不能使用 context-refresh
   - 缺乏「常見錯誤」章節指導

4. **決策樹與 handoff 命令文件的銜接缺陷**
   - Checkpoint 2「Handoff 強制動作」（decision-tree.md line 424）說「PM 必須執行 /ticket handoff」
   - 但沒有連結或引用 handoff 命令的旗標規則、五種情境
   - PM 必須自己去 handoff-command.md 查詢，但該文件的信息不足以指導決策
