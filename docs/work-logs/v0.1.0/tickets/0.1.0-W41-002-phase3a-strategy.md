## 1. 實作順序規劃（依賴關係）

### 模組拆分設計

Hook 實作涉及 5 個邏輯模組，建議實作順序如下：

```
模組 A （工具函式層）
├─ normalize_path()
├─ find_missing_files()
└─ extract_ticket_id()

模組 B （輸入驗證層）
├─ is_ticket_complete_success()
└─ read_json_from_stdin()  [依賴 hook_utils]

模組 C （資料讀取層）
├─ read_ticket_where_files()
├─ get_git_changed_files()
└─ 均依賴模組 A（路徑規範化）

模組 D （比對和警告生成層）
├─ find_missing_files() [已在模組 A]
└─ format_warning_message()
    └─ 依賴模組 C（where.files 和 git_changed 資料）

模組 E （Hook 主入口層）
├─ main()
└─ 依賴模組 A-D 的所有函式
```

### 建議實作順序

**第 1 階段（核心工具層）：優先完成**

1. **normalize_path()**
   - 目的：路徑標準化，是所有比對的基礎
   - 複雜度：低（純字串轉換）
   - 依賴：無
   - 測試：17+ 個 normalize_path 測試用例（Phase 2）

2. **extract_ticket_id()**
   - 目的：從 stdin JSON 或命令輸出中提取 Ticket ID
   - 複雜度：低（正則表達式提取）
   - 依賴：無
   - 測試：3+ 個 extract_ticket_id 測試用例

3. **find_missing_files()**
   - 目的：集合比對邏輯
   - 複雜度：低（集合運算）
   - 依賴：normalize_path()
   - 測試：6 個 find_missing_files 測試用例

**第 2 階段（輸入驗證層）：依賴第 1 階段**

4. **is_ticket_complete_success()**
   - 目的：判斷 ticket complete 是否成功執行
   - 複雜度：低（字串匹配）
   - 依賴：無
   - 測試：5+ 個 is_ticket_complete_success 測試用例

**第 3 階段（資料讀取層）：依賴第 1-2 階段**

5. **read_ticket_where_files()**
   - 目的：讀取 Ticket 的 where.files 欄位
   - 複雜度：中（YAML 解析，依賴 hook_utils）
   - 依賴：normalize_path()、hook_utils.parse_ticket_frontmatter()
   - 技術選擇：使用 hook_utils.parse_ticket_frontmatter() 而非手寫 YAML 解析
   - 測試：6+ 個測試（正常、空、不存在、YAML 錯誤等）

6. **get_git_changed_files()**
   - 目的：執行 git diff 並取得變更檔案清單
   - 複雜度：中（subprocess 呼叫、錯誤處理）
   - 依賴：normalize_path()
   - 技術選擇：使用 subprocess.run()，timeout 設定為 10 秒（對應 settings.json 的 timeout）
   - 技術風險：git 命令不存在、非 git repo、超時
   - 測試：5+ 個測試（成功、失敗、超時、格式差異）

**第 4 階段（比對和輸出層）：依賴第 1-3 階段**

7. **format_warning_message()**
   - 目的：生成警告訊息
   - 複雜度：低（字串格式化）
   - 依賴：無特殊依賴
   - 技術選擇：多行字串模板，按 Phase 1 6.3 規格生成
   - 測試：6+ 個測試（不同缺失數、邊界值）

**第 5 階段（Hook 主入口層）：依賴第 1-4 階段**

8. **main()**
   - 目的：Hook 主邏輯協調
   - 複雜度：中（多個條件判斷、流程控制）
   - 依賴：所有上述模組
   - 技術選擇：使用 run_hook_safely() 包裝，永遠返回 exit 0
   - 測試：18+ 個測試（場景 1-6、邊界 A-E、NAC、異常 E1-E4）

## 2. 虛擬碼和邏輯流程

### 2.1 normalize_path() 虛擬碼

```pseudocode
函式 normalize_path(path: 字串) -> 字串
    輸入：原始路徑字串（可能來自 where.files 或 git diff）

    步驟 1：轉換反斜線為正斜線
        path = path.replace("\\", "/")

    步驟 2：移除前綴 "./"
        當 path 以 "./" 開頭時：
            path = path[2:]

    步驟 3：移除尾斜線
        當 path 以 "/" 結尾時：
            path = path[:-1]

    步驟 4：轉為小寫
        path = path.lower()

    步驟 5：簡化多重斜線
        當 path 包含 "//" 時：
            path = path.replace("//", "/")

    步驟 6：邊界條件
        若 path 為空或只含 "./"：
            返回 ""

    返回 path
```

**特殊處理**：
- 相對路徑 `"."` 和 `".."` 本版本不展開，視為字面路徑
- 絕對路徑（以 `/` 開頭）保持原樣，用於比對

### 2.2 find_missing_files() 虛擬碼

```pseudocode
函式 find_missing_files(
    where_files: 字串列表（已規範化）,
    git_changed: 字串列表（已規範化）
) -> 字串列表

    輸入：兩個規範化後的路徑清單

    步驟 1：檢查特殊情況
        若 where_files 為空：
            返回 []  // 無預期修改，無缺失

    步驟 2：轉換為集合（便於集合運算）
        where_set = 集合(where_files)
        git_set = 集合(git_changed)

    步驟 3：計算缺失檔案
        missing = where_set - git_set  // 集合差

    步驟 4：處理目錄路徑前綴匹配
        若某 where_files 項以 "/" 結尾（表示目錄）：
            對每個 git_changed 項：
                若 git_changed 項 以 where_files 項開頭：
                    從 missing 中移除此 where_files 項

    步驟 5：返回結果（轉回列表，排序便於展示）
        返回 sorted(missing)
```

**核心邏輯**：
- 集合差：只計算出現在 where_files 但不在 git_changed 的檔案
- 不計算 extra_files（git_changed 中多出的），因為代理人可能修改輔助檔案

### 2.3 is_ticket_complete_success() 虛擬碼

```pseudocode
函式 is_ticket_complete_success(input_data: 字典) -> 布林值

    輸入：PostToolUse Hook 格式的 stdin JSON

    步驟 1：驗證工具類型
        tool_name = input_data.get("tool_name")
        若 tool_name != "Bash"：
            返回 False

    步驟 2：提取命令
        tool_input = input_data.get("tool_input", {})
        command = tool_input.get("command", "")
        若 command 為空：
            返回 False

    步驟 3：檢查命令是否為 ticket complete
        若 "ticket track complete" 不在 command：
            返回 False

    步驟 4：檢查執行結果
        tool_response = input_data.get("tool_response", {})
        exit_code = tool_response.get("exit_code", -1)
        stdout = tool_response.get("stdout", "")

        若 exit_code != 0：
            返回 False

    步驟 5：檢查成功標記
        成功標記清單 = ["已完成", "成功"]
        對每個標記：
            若 stdout 包含此標記：
                返回 True

        返回 False
```

### 2.4 read_ticket_where_files() 虛擬碼

```pseudocode
函式 read_ticket_where_files(
    ticket_id: 字串,
    project_root: 路徑
) -> 字串列表

    輸入：Ticket ID（如 "0.1.0-W41-002"）、專案根目錄

    步驟 1：定位 Ticket 檔案
        Ticket 路徑 = project_root / "docs/work-logs/v{version}/tickets/{ticket_id}.md"
        若 檔案不存在：
            記錄警告到日誌
            返回 []

    步驟 2：讀取檔案內容
        嘗試：
            內容 = 讀取檔案(Ticket 路徑)
        捕獲 異常：
            記錄錯誤到日誌（stderr + 日誌檔，雙通道）
            返回 []

    步驟 3：解析 YAML frontmatter
        使用 hook_utils.parse_ticket_frontmatter(內容)
        若解析失敗：
            記錄警告
            返回 []

    步驟 4：提取 where.files
        frontmatter = 解析結果
        where_dict = frontmatter.get("where", {})
        where_files_raw = where_dict.get("files", [])

        若 where_files_raw 為空或不存在：
            記錄（debug 層級）
            返回 []

    步驟 5：規範化每個路徑
        where_files_normalized = []
        對每個 file_path 在 where_files_raw：
            normalized = normalize_path(file_path)
            若 normalized 非空：
                加入 where_files_normalized

        返回 where_files_normalized
```

**技術決策**：
- 使用 hook_utils.parse_ticket_frontmatter() 而非自寫 YAML 解析
- 錯誤時返回空列表，不拋異常（安全降級）
- 所有讀取失敗都記錄到雙通道（stderr + 日誌）

### 2.5 get_git_changed_files() 虛擬碼

```pseudocode
函式 get_git_changed_files(project_root: 路徑) -> 字串列表

    輸入：專案根目錄

    步驟 1：構建 git diff 命令
        命令 = "git diff HEAD --name-only"

    步驟 2：執行 git 命令
        嘗試：
            結果 = 執行命令(
                command = 命令,
                cwd = project_root,
                timeout = 10 秒,
                capture_output = True
            )
        捕獲 超時異常：
            記錄錯誤到 stderr 和日誌
            返回 []
        捕獲 其他異常（非 git repo、git 不存在等）：
            記錄錯誤到 stderr 和日誌
            返回 []

    步驟 3：檢查返回碼
        若 exit_code != 0：
            記錄錯誤（stderr + 日誌）
            返回 []

    步驟 4：解析輸出
        stdout = 結果.stdout
        若 stdout 為空：
            返回 []

        行清單 = stdout.split("\n")

    步驟 5：規範化每行
        git_changed = []
        對每個行 在 行清單：
            行 = 移除前後空格(行)
            若 行 非空：
                normalized = normalize_path(行)
                加入 git_changed

        返回 git_changed
```

**技術風險**：
- git 命令失敗：repo 可能不是 git repo、git 不可用、權限問題
- 超時：repo 可能很大，訂 10 秒超時（同 settings.json 的 timeout）
- 編碼問題：可能出現非 UTF-8 檔案名，需要容錯

### 2.6 format_warning_message() 虛擬碼

```pseudocode
函式 format_warning_message(
    ticket_id: 字串,
    missing_files: 字串列表,
    where_files: 字串列表,
    git_changed_relevant: 字串列表
) -> 字串

    輸入：Ticket ID、缺失檔案清單、預期修改清單、實際變更清單

    步驟 1：計算統計數據
        missing_count = len(missing_files)
        where_count = len(where_files)
        git_count = len(git_changed_relevant)

    步驟 2：構建訊息頭
        訊息 = "============================================================\n"
        訊息 += "[並行派發驗證警告]\n"
        訊息 += "============================================================\n\n"

    步驟 3：添加主要內容
        訊息 += f"Ticket {ticket_id} 完成後，偵測到以下預期修改的檔案\n"
        訊息 += f"未出現在 git diff 中：\n\n"

    步驟 4：列出缺失檔案
        訊息 += f"缺失檔案（{missing_count} 個）：\n"
        對每個 file 在 missing_files（排序後）：
            訊息 += f"  - {file}\n"

    步驟 5：列出預期修改清單
        訊息 += f"\n預期修改（where.files 清單，共 {where_count} 個）：\n"
        對每個 file 在 where_files（排序後）：
            訊息 += f"  - {file}\n"

    步驟 6：列出實際變更清單
        訊息 += f"\n實際變更（git diff HEAD --name-only，共 {git_count} 個相關檔案）：\n"
        對每個 file 在 git_changed_relevant（排序後）：
            訊息 += f"  - {file}\n"

    步驟 7：添加建議和注意事項
        訊息 += "\n建議動作：\n"
        訊息 += "1. 確認代理人是否確實完成了這些檔案的修改\n"
        訊息 += "2. 若代理人遺漏了修改，手動補完或重新派發\n"
        訊息 += "3. 補完後執行 git diff 確認\n\n"

        訊息 += "注意：此驗證使用 git diff HEAD，包含所有未提交的變更。\n"
        訊息 += "若有其他任務的未提交變更，可能影響比對結果。\n\n"

        訊息 += "============================================================\n"

    返回 訊息
```

### 2.8 日誌檔格式和位置

**位置**：`.claude/hook-logs/parallel-dispatch-verification/`

**檔案命名**：`parallel-dispatch-verification-{YYYY-MM-DD}.log`（按日期輪轉）

**日誌內容格式**：

```
[{timestamp ISO8601}]
  TicketID: {ticket_id}
  WhereFiles: {N} 個
    - {file_1}
    - {file_2}
  GitChanged: {K} 個
    - {file_changed_1}
    - {file_changed_2}
  MissingFiles: {M} 個
    - {missing_1}
    - {missing_2}
  Result: {PASS | WARN}
  Message: {msg or empty}
```

**技術決策**：
- 日誌追加模式（不覆蓋）
- 使用 ISO8601 時間戳（便於機器解析）
- 結果簡碼：PASS（全部匹配）/ WARN（有缺失）/ SKIP（where.files 為空）/ ERROR（git 或讀取失敗）

## 3. 資料結構選擇

### 3.1 選擇的資料結構及理由

| 資料 | 選擇的結構 | 理由 | 替代方案 |
|------|----------|------|---------|
| where.files | Python list[str] | 與 Ticket frontmatter 格式一致，順序保存 | set（失去順序，便於比對） |
| git_changed | Python set[str] | 快速查找，比對時使用集合運算 | list（需要遍歷查找） |
| missing_files | Python set[str]（轉回 list 排序後輸出） | 集合差運算，再排序輸出便於展示 | list 遍歷比對 |
| 路徑映射 | dict[str, bool]（normalize 後是否匹配） | 可選，目前未用；若後續需要追蹤每個檔案的匹配狀態 | 不用（目前不需要） |

### 3.2 記憶體和效能考量

**預期規模**（Phase 1 邊界條件 A）：
- where.files：100+ 個檔案
- git diff：500+ 個檔案
- 集合運算複雜度：O(n + m)，其中 n = len(where_files)，m = len(git_changed)

**性能目標**：執行時間 < 5 秒（對應設計選擇「git timeout 10s」的安全裕量）

**記憶體估計**：
- 平均 1000 個檔案，平均路徑 50 字元 → ~50 KB
- 日誌輸出最多 10 KB
- 總體記憶體需求 < 1 MB（無問題）

## 4. 技術債務和實作風險

### 4.1 已識別的技術風險

| 風險項 | 影響級別 | 緩解策略 | Phase 4 重構建議 |
|--------|---------|---------|-----------------|
| **git diff 基準選擇** | 中 | 使用 `git diff HEAD`（無狀態），文件說明限制 | Phase 1 已討論，未來可考慮 commit hash 儲存（複雜） |
| **目錄路徑前綴匹配** | 低 | 支援以 `/` 結尾的目錄前綴匹配 | Phase 1 設計 4.3，目前不支援 glob 展開 |
| **路徑編碼問題** | 低 | 假設 UTF-8 編碼，git diff 通常使用 UTF-8 | 若出現非 UTF-8 檔案名，可能失敗，建議後續改進 |
| **超時設定** | 低 | 訂 10 秒超時（同 settings.json）| 若 repo 超大，可考慮動態調整 |
| **YAML 解析依賴** | 低 | 依賴 hook_utils.parse_ticket_frontmatter()，不自寫 | 風險轉移至 hook_utils，已由他人維護 |

### 4.2 邊界條件和權宜方案

| 邊界條件 | 處理方式 | Phase 4 改進 |
|---------|---------|------------|
| 空 where.files | 靜默跳過，不警告 | 保持（符合設計哲學） |
| git 命令失敗 | 安全降級，返回空清單 | 考慮重試機制或更詳細的錯誤報告 |
| glob 模式 | 不展開，視為字面路徑 | 若未來 Ticket 使用 glob，可擴展 find_missing_files() |
| Unicode 路徑 | 假設 UTF-8，不額外處理 | 後續改進字符編碼容錯 |
| 目錄路徑 | 支援前綴匹配（以 `/` 結尾） | 已在設計中實現 |

### 4.3 已知限制（Phase 1 設計邊界）

- **glob 模式**：Ticket where.files 應為具體路徑，不支援 `lib/**/*.dart` 展開
- **相對路徑**：不展開 `..` 等，視為字面路徑
- **絕對路徑**：保持原樣，用於比對

## 5. 架構決策記錄

### 決策 D1：為何不將所有邏輯寫成一個函式？

**決策**：拆分為 8 個函式（normalize_path, extract_ticket_id, ...）

**理由**：
- 每個函式單一責任，便於測試和維護
- Phase 2 的 18+ 個測試案例要求能獨立測試每個邏輯單元
- 易於重用（例如 normalize_path 用於 where.files 和 git_changed）
- 便於 Phase 4 的單元重構

---

### 決策 D2：為何使用 hook_utils.parse_ticket_frontmatter 而非自寫 YAML 解析？

**決策**：依賴 hook_utils 的 parse_ticket_frontmatter()

**理由**：
- 避免重複造輪子，hook_utils 已提供
- 降低本 Hook 的複雜度
- parallel-suggestion-hook 的同類函式已經使用，驗證可行
- 風險轉移至 hook_utils（已由他人維護）

**替代方案排除**：
- 自寫 YAML 解析：複雜、易出錯、難以維護
- 正則表達式解析：脆弱，可能誤判

---

### 決策 D3：為何使用 git diff HEAD 而非其他基準？

**決策**：使用 `git diff HEAD --name-only`

**理由**：
- 無狀態設計，不需要記住派發前的 commit hash
- 比對的是「工作區 vs 已提交狀態」，涵蓋所有未提交變更
- 與並行派發場景相符（多個並行代理人的變更都未提交）

**替代方案比較**：
| 方案 | 優點 | 缺點 |
|------|------|------|
| `git diff HEAD` | 無狀態、簡單 | 若有其他未提交變更會誤判 |
| `git diff HEAD~1` | 比對上次提交後的變更 | 假設上次提交是派發前（不總是成立） |
| 記錄派發前 commit hash | 精確 | 需要跨 Hook 執行的持久化狀態（複雜） |

**結論**：選擇無狀態的 git diff HEAD，並在警告訊息中說明限制

---

### 決策 D4：為何 missing_files 使用集合而非列表比對？

**決策**：使用 Python set 執行集合差運算

**理由**：
- 集合運算 O(1) lookup，快於列表 O(n) 遍歷
- 符合「預期集合 - 實際集合」的數學直觀
- 自動去重（若 where.files 有重複路徑）

**替代方案排除**：
- 逐行遍歷 where.files 查找 git_changed：時間複雜度 O(n²)

## 9. 語言特定實作考量（Python）

### 9.1 Python 特有設計

| 項目 | 決策 |
|------|------|
| 路徑操作 | 使用 pathlib.Path（而非 os.path） |
| 子進程 | 使用 subprocess.run（而非 os.system） |
| 字串 | 完整使用 f-string，不用 % 格式化 |
| 型別標註 | 使用 PEP 484 型別提示，便於靜態檢查 |
| 異常處理 | 具體捕獲異常類型，不用 bare except |

### 9.2 效能考量

**字串連接**：
- 警告訊息可能很大（1-10 KB），使用列表 + join() 而非 +=

**檔案讀取**：
- Ticket 檔案通常 < 10 KB，直接 read_text() 無問題

**集合運算**：
- where.files 最多 100+，git_changed 最多 500+，集合運算 O(n) 無問題

**subprocess 超時**：
- 訂 10 秒超時，git diff 通常 < 1 秒

### 9.3 相容性

**Windows 路徑**：
- normalize_path() 將反斜線轉為正斜線
- pathlib.Path 自動處理平台差異

**macOS 大小寫不敏感檔案系統**：
- normalize_path() 轉為小寫，便於比對
- git diff 輸出也會統一格式

## 12. 附錄：常見問題和回答

### Q1: 為什麼有時 where.files 為空也要觸發驗證？

**A**: Phase 1 的保守策略是「對所有 ticket complete 都觸發驗證」，這能減少漏報。若 where.files 為空，則靜默通過（AC-4）。

### Q2: 若 git diff 包含未提交的其他任務變更怎麼辦？

**A**: 這是 git diff HEAD 的已知限制（設計決策 D3）。警告訊息中已說明此限制。長期解決方案是實現狀態記錄（Phase 4），目前接受此誤差。

### Q3: 為什麼不支援 glob 模式？

**A**: Phase 1 設計邊界（4.3）決定不展開 glob。Ticket where.files 慣例是具體路徑。若未來需要支援，可擴展 find_missing_files() 使用 fnmatch 或 pathlib.glob()。

### Q4: 日誌寫入失敗時怎麼辦？

**A**: Hook 永遠 exit 0，日誌失敗記錄到 stderr（雙通道）。不會阻擋工作流。

### Q5: 效能是否足夠？

**A**: 預期場景（100+ 個檔案）在 < 5 秒內完成。git diff 通常 < 1 秒，集合運算 O(n)，路徑規範化 O(n)。無性能風險。
