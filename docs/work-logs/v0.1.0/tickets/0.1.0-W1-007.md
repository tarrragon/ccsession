# Execution Log

## Task Summary

補充 spec.md SessionEvent 欄位定義和訊息協議（H1+M1+M8+M9+M10）

---

## Problem Analysis

W1-006 四視角審查發現 spec.md 有 5 項缺漏，影響 Phase 1 資料結構設計和前後端協議一致性：

1. **H1**（高）：SessionEvent struct 缺少 messageId 和 contentIndex 欄位，無法支援 assistant content array 展開後的事件去重和定位
2. **M1**（中）：WebSocket 協議中的時間戳格式未統一定義，前後端可能使用不同精度/格式
3. **M8**（中）：get_session_history limit 預設值在 spec（100）和 UC-002（1000）之間不一致
4. **M9**（中）：SessionEvent.Content 各 Type 對應的值結構未說明，實作時需猜測
5. **M10**（中）：get_session_history before=null 的語義未文件化，前端不知道不帶 before 時的行為

透過檢查實際 Claude Code JSONL 檔案（~/.claude/projects/ 下的 .jsonl），確認了：
- 時間戳格式為 ISO 8601 UTC 毫秒（如 `2026-03-05T04:44:24.361Z`）
- 每行有 `uuid` 欄位作為唯一識別符
- assistant 的 content array 元素有 `type`/`text`/`id`/`name`/`input` 等欄位
- tool_result 是獨立的頂層 type（非僅存在於 content array 中）
- JSONL 還存在 `progress`、`queue-operation`、`file-history-snapshot` 等非對話類型

---

## Solution

### H1: SessionEvent 補充 messageId 和 contentIndex

在 SessionEvent struct 新增兩個欄位：
- `MessageID`（string）：對應 JSONL 行的 `uuid`，用於事件去重
- `ContentIndex`（int）：assistant content array 展開時的索引位置，非 assistant 類型為 -1

同時將 Content 型別從 `json.RawMessage` 改為具名的 `EventContent`，並新增完整的 EventContent 定義表格。

### M1: 時間戳格式統一

新增 2.3 節「時間戳格式定義」，明確所有位置統一使用 ISO 8601 UTC 毫秒格式，並說明選擇理由（沿用 Claude Code 原生格式）。

### M8: limit 預設值統一

- spec.md `get_session_history` 的 `limit` 預設值為 100（每次請求）
- `max_history_lines`（配置值 1000）改為「Backend 記憶體保留上限」而非「首次載入最大行數」
- 同步修正 UC-002 的描述，消除歧義

### M9: EventContent 各 Type 值說明

新增 EventContent 定義表格，詳細說明 5 種 Type（user/assistant/tool_use/tool_result/thinking）各自的 Content 結構和欄位含義。

### M10: before=null 語義

在新增的 get_session_history 參數說明中明確定義：before 為 null 或未提供時，回傳該 session 最新的 limit 筆事件。

### 額外改善

- 新增 2.4 節 JSONL 頂層常見欄位表格（uuid、sessionId、parentUuid、version、gitBranch、cwd）
- 補充頂層事件類型表格，加入 tool_result 和非對話類型說明
- 修正 Assistant content array 元素的關鍵欄位（tool_use 的 `id`/`name`/`input`）
