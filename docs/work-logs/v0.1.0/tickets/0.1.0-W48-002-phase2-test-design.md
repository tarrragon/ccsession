# Phase 2 測試設計：場景 #8 Hook 覆蓋

## 1. 測試策略

### 1.1 測試層級與目標

| 層級 | 目標 | 測試代理人 |
|------|------|-----------|
| 單元測試（Unit） | `check_decision_keywords()` 函式邏輯 | pytest |
| 整合測試（Integration） | Hook 完整流程（輸入→檢測→輸出） | pytest |
| 邊界測試（Boundary） | 場景 #8 與其他場景的邊界互動 | pytest |

### 1.2 測試分類

**行為鏈式推演方向**：

```
用戶輸入含「執行順序/方向」類詢問
    ↓ [前置驗證：輸入已小寫化]
    ↓ [When] check_decision_keywords() 掃描 DECISION_KEYWORDS
    ↓ [Then] 匹配場景 #8 關鍵字
    ↓ [中間狀態] 回傳 "執行方向確認"
    ↓ [Then] check_skill_suggestion() 包裝成 AskUserQuestion 提醒訊息
```

### 1.3 測試關鍵字對應表

Phase 1 新增的 3 組場景 #8 關鍵字：

| 關鍵字組合 | 場景標籤 | 典型觸發句 |
|------------|---------|------------|
| `("順序", "任務")` | 任務排序詢問 | 「任務順序怎麼安排」 |
| `("接下來", "做")` | 下一步方向 | 「接下來要做什麼」 |
| `("先後", "執行")` | 執行先後安排 | 「先後執行的順序」 |

### 2.2 否定語境測試（不應觸發）

**原則**：任一組合關鍵字若被 `_is_keyword_negated()` 過濾（被否定詞修飾），則 `check_decision_keywords()` 應跳過此匹配，回傳 None。

#### 場景 B1：「順序」被否定詞修飾
```
Given: 輸入包含 "不" + "順序" + "任務"
When: 呼叫 check_decision_keywords("不需要確認任務順序")
Then: _is_keyword_negated(prompt, "順序") 回傳 True，函式跳過此匹配，回傳 None
```

測試案例清單：
- 否定 01a：明確否定
  - 輸入：`"不需要確認任務順序"`
  - 預期：`None`
  - 否定詞：`"不需要"`

- 否定 01b：使用「無需」否定詞
  - 輸入：`"無需考慮任務順序的問題"`
  - 預期：`None`
  - 否定詞：`"無需"`

- 否定 01c：遠距否定詞（中間隔詞彙）
  - 輸入：`"完全不需要去關心任務順序這個事"`
  - 預期：`None`（"不需要" 和 "順序" 在 15 字符窗口內）

#### 場景 B2：「接下來」被否定詞修飾
```
Given: 輸入包含 "不" + "接下來" + "做"
When: 呼叫 check_decision_keywords("不用接下來要做什麼")
Then: _is_keyword_negated(prompt, "接下來") 回傳 True，函式回傳 None
```

測試案例清單：
- 否定 02a：明確否定
  - 輸入：`"無須考慮接下來要做什麼"`
  - 預期：`None`
  - 否定詞：`"無須"`

- 否定 02b：「不用」否定詞
  - 輸入：`"不用擔心接下來做什麼"`
  - 預期：`None`

#### 場景 B3：「執行」被否定詞修飾

### 2.4 與現有場景的優先級和邊界測試

#### 場景 D1：場景 #8 不與場景 #7（派發方式選擇）重疊

```
Given: 輸入同時包含 #7 關鍵字「並行」「派發」
When: 呼叫 check_decision_keywords("要用並行派發嗎")
Then: 因 #7 關鍵字先在字典中出現，回傳 #7 的場景 "派發方式選擇"（非 #8）
```

測試案例清單：
- 邊界 04a：#7 關鍵字觸發
  - 輸入：`"要並行派發這些任務嗎"`
  - 預期：`"派發方式選擇"`（#7，非 #8）
  - 原因：`("並行", "派發")` 在 DECISION_KEYWORDS 中，且不與 #8 關鍵字重疊

- 邊界 04b：確保 #8 關鍵字不會誤觸發為 #7
  - 輸入：`"任務順序和派發方式分開考慮"`
  - 預期：看是否同時觸發兩個，預期應該是 #7（先匹配）或 #8（按字典順序）
  - 注：此測試驗證字典迭代順序

#### 場景 D2：場景 #8 與場景 #5（優先級確認）的邊界

```
Given: 輸入包含 #5 關鍵字「先做」「哪個」
```

測試案例清單：
- 邊界 05a：#5 關鍵字仍保留原場景
  - 輸入：`"先做哪個任務比較好"`
  - 預期：`"優先級確認"`（#5，非 #8）
  - 原因：`("先做", "哪個")` 已歸類為 #5，#8 的 `("先後", "執行")` 與其無重疊

## 4. 邊界條件與誤觸發防護測試

### 4.1 大小寫處理

**原則**：Hook 在呼叫 `check_decision_keywords()` 前已執行 `prompt.lower()`，所以測試輸入皆為小寫。

#### 場景 G1：已小寫化輸入
```
Given: 輸入已小寫化
When: 呼叫 check_decision_keywords("TASK 順序怎麼樣")（符合實際 .lower() 後的格式）
Then: 因 "TASK" 小寫後為 "task"，不含「任務」關鍵字，回傳 None
```

測試案例清單：
- 邊界 06a：英文和中文混合（已小寫）
  - 輸入：`"task 順序如何"`
  - 預期：`None`（缺少「任務」）

- 邊界 06b：純英文（已小寫）
  - 輸入：`"execution order"`
  - 預期：`None`（無中文關鍵字）

### 4.2 長文本和嵌套語境

#### 場景 G2：長文本仍正確偵測
```
Given: 輸入為長文本
When: 呼叫 check_decision_keywords("我想了解一下這幾個不同的任務執行順序應該如何安排比較好，有沒有什麼建議")
Then: 關鍵字「順序」和「任務」都在長文本中找到，回傳 "執行方向確認"
```

測試案例清單：
- 邊界 07a：長度超過 100 字符
  - 輸入：長文本，包含「順序」和「任務」
  - 預期：`"執行方向確認"`

- 邊界 07b：多句組合
  - 輸入：`"現在我們有幾個任務，先後執行的順序要怎麼決定？"\n"你有什麼建議"`
  - 預期：`"執行方向確認"`（因為「先後」「執行」都在）

### 4.3 否定詞窗口邊界

**原則**：`NEGATION_WINDOW_SIZE = 15` 字符，關鍵字必須在否定詞之後 15 字符內才被認為被否定。

#### 場景 G3：關鍵字在否定窗口邊界
```
Given: 否定詞位置 + 15 字符窗口
When: 呼叫 check_decision_keywords("不用一二三四五六七八九十十一十二任務順序")
Then: 「任務」位置 ~18+ 字符，超出窗口，_is_keyword_negated 回傳 False，函式回傳 "執行方向確認"
```

測試案例清單：
- 邊界 08a：關鍵字恰好在窗口內
  - 輸入：`"不用一二三四五六順序"`（不用後 ~8 字符）
  - 預期：`None`（被否定）

- 邊界 08b：關鍵字剛好超出窗口
  - 輸入：`"不用一二三四五六七八九十十一十二三順序"`（不用後 ~18 字符）
  - 預期：`"執行方向確認"`（不被否定，因超出 15 字符窗口）

## 6. 測試覆蓋率分析

### 6.1 覆蓋範圍

|---------|--------|--------------|

| 否定語境測試（B1-B3） | 7 | `_is_keyword_negated()` 的否定過濾邏輯 |
| 邊界條件測試（C1-C3） | 6 | 雙關鍵字要求檢查 |

| **合計** | **35** | **100%** |

### 6.2 路徑覆蓋

**核心路徑**（`check_decision_keywords` 函式）：

```
prompt 輸入
    ↓
遍歷 DECISION_KEYWORDS 字典
    ├─ [路徑 1] 雙關鍵字，都出現，都未被否定 → 回傳場景名稱 ✓ 正向測試覆蓋
    ├─ [路徑 2] 雙關鍵字，都出現，任一被否定 → continue ✓ 否定語境測試覆蓋
    ├─ [路徑 3] 雙關鍵字，只有一個出現 → continue ✓ 邊界條件測試覆蓋
    ├─ [路徑 4] 單一關鍵字，出現且未被否定 → 回傳場景名稱 ✓ 現有測試已覆蓋
    ├─ [路徑 5] 單一關鍵字，出現但被否定 → continue ✓ 現有測試已覆蓋
    └─ [路徑 6] 無匹配 → 回傳 None ✓ 無匹配測試覆蓋
```

## 8. 測試設計決策記錄

### 決策 1：使用現有 `_is_keyword_negated()` 函式

### 決策 2：覆蓋三層測試（單元→邊界→整合）

**理由**：
- 單元測試（正向 + 否定 + 邊界）驗證 `check_decision_keywords()` 邏輯
- 邊界測試（與其他場景）驗證設計無衝突
- 整合測試驗證 Hook 完整流程
- 三層結合提供 100% 邏輯覆蓋

### 決策 3：採用 GWT（Given-When-Then）格式

**理由**：
- 符合 BDD 測試設計方法論
- 使測試意圖和預期結果清晰明確
- 便於後續 Phase 3b 實作測試程式碼時參考

### 決策 4：35 個測試案例規模

**理由**：
- 3 個新關鍵字組合 × 3 個正向案例 = 9 個正向
- 3 個新關鍵字組合 × 2-3 個否定案例 = 7 個否定
- 3 個組合 × 2 個邊界案例 = 6 個邊界
- 加上場景邊界、無匹配、整合、防護 = 合計 35 個
- 規模適中，覆蓋完整不過度
