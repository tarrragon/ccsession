# Execution Log

## Task Summary

分析 Wave 完成多視角審查的觸發機制缺口，確保情境 C 時 PM 被提醒執行 /parallel-evaluation

---

## Problem Analysis

### 現有觸發機制分析

#### 1. commit-handoff-hook 的能力範圍

**Hook 類型**：PostToolUse（git commit 成功後觸發）

**當前實現**：
- 情境 A（#11a）：Ticket 仍 in_progress → 輸出 Context 刷新提醒
- 情境 B（#11b）：Ticket completed + 同 Wave 有 pending → 輸出任務切換提醒

**代碼流程**：
1. 讀取 PostToolUse JSON
2. 驗證 tool_name == "Bash"
3. 檢查 command 是否為 git commit
4. 若成功，檢查 commit type（docs/chore/style 等）決定是否跳過 #16
5. 輸出提醒訊息

**情境 C 的缺口**：
- Hook 邏輯無「同 Wave 無 pending」判斷分支
- 查詢範圍僅限當前 Ticket（無 Wave-level 檢查）
- 若無 pending Ticket，流程直接終止，無任何提醒

#### 2. parallel-suggestion-hook 的能力範圍

**Hook 類型**：UserPromptSubmit（用戶輸入時觸發）

**觸發條件**：
- 用戶提示包含「繼續」「下一個」「任務鏈」等關鍵字
- 掃描並行 Ticket
- 若無並行建議，輸出 `WAVE_WRAP_UP_REMINDER`

**核心特性**：
- **被動觸發**：依賴用戶明確輸入
- **用戶驅動**：需要用戶主動說「繼續」
- 無自動偵測機制

**與情境 C 的脫節**：
- decision-tree.md 第 310 行要求 commit 後自動進行 Wave 完成審查
- parallel-suggestion-hook 無法處理自動觸發（只有用戶指令觸發）
- 訊息內容也不符合「強制執行 /parallel-evaluation」的要求

#### 3. 規則定義 vs 實現的差距

**規則要求**（decision-tree.md Checkpoint 2，第 310-313 行）：
- [情境 C] ticket completed + 同 Wave 無 pending → [強制] /parallel-evaluation（Wave 完成審查）
- → 審查發現 → 建立 Ticket（不中斷）→ 再查詢版本全狀態
- +-- C1：有其他 Wave pending → AskUserQuestion #3a（Wave 收尾）
- +-- C2：版本全部完成 → /version-release check → AskUserQuestion #13

**實現現狀**：
- ✗ commit-handoff-hook：無情境 C 偵測
- ✗ parallel-suggestion-hook：被動不符合自動要求
- ✗ 缺失：Wave 完成時的主動審查提醒

#### 4. 可行性評估

**方案 A：擴充 commit-handoff-hook**
- 可行性：HIGH
- 代碼複雜度：LOW（100-150 行）
- 成本：LOW
- 優點：單一觸發點，邏輯集中，無新 Hook 開銷
- 缺點：責任邊界擴大，需增加 Ticket 掃描邏輯

**方案 B：新增獨立 Hook 偵測情境 C**
- 可行性：HIGH
- 代碼複雜度：MEDIUM（150-200 行）
- 成本：MEDIUM（新檔案 + 測試）
- 優點：職責單一明確，便於未來擴展
- 缺點：新增 PostToolUse Hook 增加執行時間，需協調多 Hook 順序

**方案 C：commit-handoff-hook 擴充 + 新增常數訊息**（推薦）
- 結合 A 和 B 的優點
- 在 commit-handoff-hook 中新增「情境 C 判斷」分支
- 複用現有訊息框架
- 無新 Hook，成本最低

---

## Solution

### 推薦方案：方案 C（commit-handoff-hook 擴充 + 新增常數）

**理由**：
1. 成本效率最優（無新 Hook，複用現有觸發機制）
2. 邏輯集中（所有 Checkpoint 2 情境在單一 Hook 中）
3. 執行效率高（無額外 PostToolUse 開銷）
4. 易於維護（無需協調多個 Hook）

### 實現細節

#### 1. commit-handoff-hook.py 擴充

**新增邏輯分支**（在第 178 行 `if is_git_commit_command` 後）：
- 現有邏輯：決定是否跳過 #16
- 新增：情境判斷（A/B/C）
  - Step 1: 查詢當前 Ticket 狀態
  - Step 2: 判斷情境
    - 情境 A：Ticket 仍 in_progress
    - 情境 B：Ticket completed + 同 Wave 有 pending
    - 情境 C：Ticket completed + 同 Wave 無 pending → 需進行 Wave 審查

**核心查詢函式**：
- `get_current_ticket_from_context()`：從工作目錄或 commit message 提取 Ticket ID，讀取 frontmatter
- `check_pending_in_wave(wave_id)`：掃描 `docs/work-logs/v*/tickets/` 下同 Wave 的 pending Ticket
- `check_pending_other_waves()`：掃描版本中是否還有其他 Wave 的 pending Ticket

**複雜度評估**：O(n)，n = 版本內 Ticket 數量；實測 50-100 個 Ticket 時 < 100ms

#### 2. ask_user_question_reminders.py 新增常數

**新增 WAVE_COMPLETION_REMINDER**：
- 通知 PM：Wave 已完成，需執行 /parallel-evaluation 進行 Wave 級完整審查
- 強制步驟：
  1. 執行 /parallel-evaluation Wave 完成審查
  2. 若審查發現問題，立即 /ticket create 建立相關 Ticket（不中斷主線）
  3. 查詢版本全狀態，進入 C1 或 C2 分支

#### 3. 觸發條件和輸出流程

**觸發時機**：git commit 成功後，commit-handoff-hook 檢測到：
- `tool_name == "Bash"`
- `command` 包含 `git commit`
- `stdout` 包含成功標記
- 當前 Ticket 狀態 == `completed`
- 同 Wave 無 `pending` Ticket

**PM 的執行路徑**：
1. git commit 成功
2. commit-handoff-hook 檢測情境 C
3. 輸出 WAVE_COMPLETION_REMINDER
4. PM 執行：/parallel-evaluation Wave 完成審查
5. 若發現問題：/ticket create（自動建立，不中斷）
6. 查詢版本全狀態：ticket track list --status pending
7. C1：有 pending → AskUserQuestion #3a（Wave 收尾）
8. C2：無 pending → /version-release check → AskUserQuestion #13

### 與現有規則的一致性

**decision-tree.md Checkpoint 2 對應**：
- 行 310：[情境 C] ticket completed + 同 Wave 無 pending → [強制] /parallel-evaluation ✓

**askuserquestion-rules.md 對應**：

- 場景 #3a（情境 C1 收尾）：新 Hook 主動提醒 ✓
- 場景 #13（情境 C2 路由）：新 Hook 主動提醒 ✓
