#### 測試設計概述

3. Hook 配置檔完整性檢查 - 9 個案例
4. .claude/config 目錄檢查 - 4 個案例
5. README.md 檢查 - 3 個案例
6. 開發規範文件驗證 - 6 個案例

#### 核心設計決策

| 決策 | 內容 | 理由 |
|-----|------|------|
| **Mock 策略** | 不 Mock 檔案系統，使用 tmp_path fixture 建立真實檔案 | Sociable Unit Tests 原則：檔案操作是 module 外部世界邊界 |
| **規則匹配** | .gitignore 規則使用模糊匹配（容許變體如 /coverage/, coverage, coverage/*） | 驗證意圖而非精確文字，提升使用者體驗 |
| **責任分離** | 配置檔檢查和目錄檢查分開 | 不同責任（檔案+格式 vs 目錄+權限），允許獨立迭代 |
| **優先級區分** | MUST 項（.gitignore、.claude、Hook、config），SHOULD 項（README、規範檔） | 框架啟動的必要條件 vs 推薦項 |

#### 測試案例設計模式

所有測試均採用 GWT (Given-When-Then) 格式：

```
Given [前置條件]
When [執行檢查]
Then [預期結果]
```

涵蓋路徑類型：
- **正常路徑**: 所有項目都存在、格式正確
- **缺失路徑**: 部分或全部項目缺失
- **邊界情況**: 空檔案、檔案與目錄混淆、權限問題
- **異常路徑**: 編碼錯誤、格式無效（JSON/YAML）

#### 資料結構設計

為每個檢查項定義專用 Dataclass，遵循現有模式（FrameworkFileInfo、DocsStructureInfo）：

1. **GitignoreCheckInfo** - 檢查規則完整性
2. **ClaudeDirectoryCheckInfo** - 檢查目錄存在性
3. **HookConfigurationCheckInfo** - 檢查配置檔和格式
4. **ClaudeConfigCheckInfo** - 檢查配置目錄狀態
5. **ReadmeCheckInfo** - 檢查 README.md 存在性
6. **LanguageStandardCheckInfo** - 檢查語言規範檔

#### 測試環境規劃

**Fixture 需求**：
- tmp_project_path - 臨時專案結構
- tmp_gitignore - 各種 .gitignore 變體
- tmp_claude_structure - 完整 .claude 結構
- invalid_json、invalid_yaml - 格式錯誤測試檔案
- various_encodings - 多種編碼檔案

**Mock 策略**：
- 所有檔案系統操作使用真實 tmp_path（不 Mock）
- 字串解析使用真實檔案內容
- 編碼、權限等通過 tmp_path 建立實際條件
