## 重構發現與分析

### 並行評估（情境 A）發現

**發現 1**：`_execute_set_relation_field()` 函式過長

```python
# 原始情況
位置：track_relations.py L54-147
長度：93 行（含文件字串 16 行，邏輯 77 行）
認知負擔指數：13（超過 10 的閾值）
混合邏輯：
  - Ticket 驗證（4 步驟）
  - ID 清單標準化（4 種格式）
  - 三種模式處理邏輯（replace/add/remove）
  - Ticket 更新和保存
  - 結果輸出
```

**發現 2**：值標準化邏輯重複

```python
# 重複出現的邏輯（L105-111）
if isinstance(current_value, str):
    current_list = [id_str.strip() for id_str in current_value.split(",") if id_str.strip()]
elif isinstance(current_value, list):
    current_list = current_value
else:
    current_list = []

# 維護問題：
# - 若要修改標準化邏輯，需在多個地方更新
# - 可複用性差
# - 代碼重複率高
```

### 重構方案

**方案選擇**：函式分解 + 邏輯提取

**具體實施**：

1. **提取 `_normalize_ticket_id_list()`**
   - 統一所有 ID 清單的標準化邏輯
   - 支援字符串（逗號或空格分隔）和列表
   - 行數：21 行（含文件）
   - 認知負擔：2

2. **提取三個模式處理函式**
   - `_execute_set_relation_field_replace()`: 2 行
   - `_execute_set_relation_field_add()`: 8 行
   - `_execute_set_relation_field_remove()`: 4 行
   - 每個函式單一職責

3. **簡化主協調函式**
   - 從 93 行簡化為 75 行邏輯
   - 職責清晰：驗證 → 解析 → 選擇模式 → 更新 → 輸出
   - 認知負擔：6-7（51% 改善）
