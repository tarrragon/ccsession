## 1. Phase 4a 多視角分析結果

### 1.1 Redundancy 視角（代碼重複）

**P1 級發現**：Hook 常數/正則重複（DRY 違反）
- Hook 第 54-66 行複製了 `constants.py` 的 `TICKET_ID_PATTERN` 和 `KNOWN_TICKET_SUFFIXES`
- 同一份知識有兩個 Source of Truth，違反 DRY 原則
- 未來新增後綴時必須同步修改兩處，易產生遺漏

**P2 級發現**：Hook has_description_suffix() 邏輯重複
- Hook 第 185-199 行與 id_parser.py 第 68-100 行幾乎相同
- Hook 版本缺少 None 輸入防守（輸入衛生問題）

**P2 級發現**：ticket_loader.py 未使用導入
- Line 45 導入 `TICKET_ID_PATTERN`，實際未使用
- 應移除（單行修復）

### 1.2 Coupling 視角（耦合度）

**發現**：Hook 與 constants.py 隱性耦合
- 常數複製導致版本同步困難
- 修改規則時需要多個檔案同步變更
- Hook 設計上是 standalone，但內容依然耦合

### 1.3 Complexity 視角（複雜度）

**P3 級發現**：Hook main() 函式複雜度過高（CLi=12）
- main() 第 427-538 行承擔 7 個步驟
- 重複的「日誌格式化 + save_check_log + print」模式出現兩次
- 應提取輔助函式降低複雜度

**P3 級發現**：validate_ticket_id_format() 回傳語義模糊
- 回傳 `Tuple[bool, str, bool]`，呼叫端需理解非對稱語義
- 應改為 dataclass 提升可讀性（風險較高，留作後續技術債務）

**P4 級發現**：list_tickets() docstring 不完整
- 缺少「.md 優先於 .yaml」的排序說明

## 6. 設計決策說明

### 決策 1：為什麼不動態導入 Hook 常數？

**背景**：Phase 4a 分析指出常數複製違反 DRY，為什麼不從 ticket_system 導入？

**理由**：
- Hook 設計為 standalone 腳本（`dependencies = []`）
- Hook 由框架直接執行，不經過 Python 模組系統初始化
- 動態導入需要設置 sys.path，增加依賴複雜度，違反 Hook 原意
- 當前 Python 版本支援 zipimport，但 Hook 需保持輕量級

**決策**：接受複製，用註釋和測試防護同步問題

### 決策 2：為什麼 _emit_result 不內聯？

**背景**：提取輔助函式會增加函式呼叫開銷

**理由**：
- DRY 原則：相同邏輯出現 2 次
- 可讀性：main() 從 111 行簡化為 ~70 行
- 可維護性：日誌格式集中，未來修改只需改一處
- 效能影響：可忽略（Hook 執行頻率低）

**決策**：提取 _emit_result 輔助函式，改善程式碼組織

### 決策 3：為什麼不改 validate_ticket_id_format 回傳類型？

**背景**：P3 分析指出回傳 Tuple[bool, str, bool] 語義模糊

**風險評估**：
- 改為 dataclass 需修改約 30 行程式碼
- 呼叫端 3+ 處需同步改動
- Hook 層和 ticket_system 層都需改
- 當前功能運作正常，無 bug

**決策**：留作技術債務，當前優先級不足以承擔改動風險
