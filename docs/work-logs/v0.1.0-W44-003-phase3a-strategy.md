## 1. 實作策略設計

### 1.1 核心需求概述

Ticket 系統需要擴充以支援描述性後綴（如 `-phase1-design`），允許 22 個現存非標準命名檔案被正確識別和載入。

**修改 5 個檔案**：
1. `constants.py` — 改寫 TICKET_ID_PATTERN + 新增 KNOWN_TICKET_SUFFIXES
2. `id_parser.py` — 新增 extract_core_ticket_id() + has_description_suffix()
3. `ticket_loader.py` — 更新 list_tickets() 去重邏輯
4. `ticket-id-validator-hook.py` — 動態導入 + 寬鬆驗證
5. `.claude/rules/core/ticket-id-conventions.md` — 新建命名規範文件

### 1.2 實作順序和依賴關係

```
Step 1: constants.py 改寫正則 + 新增常數
        ↓ (影響下游所有模組)
Step 2: id_parser.py 新增純邏輯函式（無外部依賴）
        ↓ (提供核心 ID 解析能力)
Step 3: ticket_loader.py 更新掃描邏輯（使用 Step 2 函式）
        ↓ (整合到核心載入流程)
Step 4: ticket-id-validator-hook.py 更新（可與 Step 3 並行）
        ↓ (驗證層更新)
Step 5: ticket-id-conventions.md 編寫（獨立，最後執行）
```

**複雜度分布**：
- Step 1: 低（改寫 1 行正則 + 新增 1 個常數列表）
- Step 2: 中（2 個新函式，邏輯複雜但完全隔離）
- Step 3: 高（修改核心掃描邏輯，涉及去重機制）
- Step 4: 中（動態導入 + 條件分支驗證）
- Step 5: 低（純文件編寫）

**依賴分析**：
- Step 3 必須在 Step 2 完成後開始（使用 extract_core_ticket_id()）
- Step 1 是所有其他步驟的前置（定義正則和常數）
- Step 4 可與 Step 3 並行（只需使用 Step 1 的常數）
- Step 5 完全獨立（文件編寫）

## 2. 資料結構選擇

### 2.1 TICKET_ID_PATTERN 正則表達式

**新正則**：
```python
r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)(-[a-z0-9][a-z0-9-]{0,59})?$"
```

**設計理由**：
- Group(1,2,3) 保持不變（向後相容性）
- Group(4) 新增可選後綴（`(...)?` 使其可選）
- 後綴格式：以 `-` 開頭，首字元為小寫字母或數字，長度限制 60 字元
- 限制小寫保持命名一致性（全系統採用小寫後綴規範）

**邊界條件設計**：
- 標準格式（無後綴）：仍完全有效，group(4) = None
- 後綴只有連字號：不匹配（`[a-z0-9]` 要求至少一字元）
- 後綴超過 60 字元：不匹配（實務限制，避免過長識別符）
- 後綴含大寫：不匹配（規範強制小寫）

### 2.2 KNOWN_TICKET_SUFFIXES 常數集合

**資料結構**：
```python
KNOWN_TICKET_SUFFIXES: List[str] = [
    "-phase1-design",
    "-phase2-test-design",
    "-phase3a-strategy",
    "-phase3b-execution-report",
    "-analysis",
    "-test-cases",
    "-test-cases-quick-reference",
    "-feature-spec",
]
```

**用途**：
1. Hook 驗證時區分「已知」vs「未知」後綴
2. 後續擴充時的參考清單
3. 文件 ticket-id-conventions.md 中的規範來源

### 2.3 extract_core_ticket_id() 函式設計

**輸入/輸出**：
- 輸入：原始 ID 字串（可能含後綴或無效）
- 輸出：核心 ID（去掉後綴），或 None（無效）

**邏輯流程**：
```
1. 型別檢查（None、空字串）→ 返回 None
2. 正則匹配 → 不匹配返回 None
3. 提取 group(1,2,3) → 重組為 "{v}-W{w}-{s}"
4. 返回核心 ID
```

**設計特點**：
- 純邏輯函式（無副作用）
- 單一責任（只負責去掉後綴）
- 容易測試（對所有邊界案例都有預期行為）

### 2.4 has_description_suffix() 輔助函式

**輸入/輸出**：
- 輸入：原始 ID 字串
- 輸出：布林值（是否含有有效後綴）

**邏輯流程**：
```
1. 正則匹配
2. 匹配且 group(4) != None → True
3. 否則 → False
```

**用途**：
- Hook 驗證時快速判斷是否含後綴
- 輔助信息提示（「這是帶後綴的 ID」）

## 5. 關鍵實作指引

### 5.1 Step 1: constants.py 改寫（複雜度低）

**關鍵點**：
1. 正則字串以原始字串形式定義（`r"..."`），確保反斜線正確
2. 四個群組順序重要：group(1) = version, group(2) = wave, group(3) = sequence, group(4) = suffix
3. 新增 KNOWN_TICKET_SUFFIXES 時，確保列表順序（按字母或用途分類）
4. 驗證方法：快速命令列測試
   ```python
   import re
   pattern = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)(-[a-z0-9][a-z0-9-]{0,59})?$"
   test_cases = [
       "0.31.0-W3-001",           # standard ✓
       "0.1.0-W11-004-phase1",    # with suffix ✓
       "0.1.0-W11-004-Phase1",    # uppercase ✗
   ]
   for tc in test_cases:
       assert (re.match(pattern, tc) is not None) # verify
   ```

### 5.2 Step 2: id_parser.py 新增函式（複雜度中）

**邏輯實作細節**：

extract_core_ticket_id():
```
1. 防守式編程：None 檢查
   if raw_id is None:
       return None

2. 正則匹配
   match = re.match(TICKET_ID_PATTERN, raw_id)
   if not match:
       return None

3. 群組提取和重組
   version = match.group(1)
   wave = match.group(2)
   sequence = match.group(3)
   # 注意：不取 group(4)，直接拼接前三個
   core_id = f"{version}-W{wave}-{sequence}"
   return core_id
```

has_description_suffix():
```
1. 正則匹配
   match = re.match(TICKET_ID_PATTERN, raw_id)
   if not match:
       return False

2. 檢查後綴群組
   suffix = match.group(4)
   return suffix is not None
```

**測試驅動點**：
- 9 個邊界案例確認 extract_core_ticket_id() 正確性
- 3 個案例確認 has_description_suffix() 布林邏輯
- 特別注意：None 輸入、空字串、無效格式的處理

### 5.3 Step 3: ticket_loader.py 更新（複雜度高）

**去重機制核心邏輯**：

```
Initial state:
  loaded_core_ids = set()  # 追蹤已載入的核心 ID

For each file in sorted order:
  filename = "0.1.0-W11-004-phase1-design.md"
  stem = "0.1.0-W11-004-phase1-design"  # 去副檔名

  core_id = extract_core_ticket_id(stem)
  # 返回 "0.1.0-W11-004"

  if core_id in loaded_core_ids:
      # 已載入此 core_id，跳過此檔案
      continue

  ticket = load_ticket(version, core_id)
  if ticket:
      tickets.append(ticket)
      loaded_core_ids.add(core_id)
```

**重要設計決策**：
- 使用 `extract_core_ticket_id()` 而非直接正則匹配（分離關注點）
- 集合去重 O(1) 查詢效率
- 標準檔案優先載入（檔案掃描順序，通常標準格式檔名在後綴檔之前）

### 5.4 Step 4: ticket-id-validator-hook.py 更新（複雜度中）

**動態導入策略**：

```python
# 嘗試導入
try:
    sys.path.insert(0, str(Path(__file__).parent.parent / "skills/ticket"))
    from ticket_system.lib.constants import TICKET_ID_PATTERN, KNOWN_TICKET_SUFFIXES
    _using_shared_constant = True
except ImportError:
    # Fallback：維護本地副本
    TICKET_ID_PATTERN = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)(-[a-z0-9][a-z0-9-]{0,59})?$"
    KNOWN_TICKET_SUFFIXES = ["-phase1-design", ...]
    _using_shared_constant = False
```

**寬鬆驗證邏輯**：

```python
match = re.match(TICKET_ID_PATTERN, ticket_id)

if not match:
    # 格式錯誤
    print("WARNING: 無效 Ticket ID 格式")
    return  # exit 0

if match.group(4) is None:
    # 標準格式，執行完整驗證（版本一致性等）
    perform_full_validation()
else:
    # 帶後綴格式，寬鬆驗證
    suffix = match.group(4)
    if suffix in KNOWN_TICKET_SUFFIXES:
        print(f"INFO: 識別到已知後綴 {suffix}")
    else:
        print(f"INFO: 後綴 {suffix} 不在規範中，建議參考 ticket-id-conventions.md")
    # 無論如何都不阻止（exit 0）
```

**輸出訊息設計**：
- 標準 ID 驗證失敗 → WARNING 級別
- 帶後綴 ID → INFO 級別（只提示，不阻止）
- 所有情況 → exit code = 0（寬鬆驗證）

## 6. 權宜方案與技術債務

### 6.1 最小可行實作

實作聚焦於「解析現存 22 個檔案並去重」，暫不實現：
- 複雜的後綴推薦引擎（Hook 只是簡單提示）

### 6.2 已知限制

| 限制 | 影響 | Phase 4 建議 |
|------|------|------------|
| 後綴需手動維護 KNOWN_TICKET_SUFFIXES | 新增後綴需同步常數 | 建立後綴發現機制，自動掃描 |
| Hook 動態導入失敗時使用 fallback | 需確保 fallback 與主正則同步 | 定期檢查機制，防止版本漂移 |
| 後綴限制小寫 + 數字 + 連字號 | 無法支援混合大小寫或其他符號 | 重新評估命名規範，可能擴充字元集 |
| 後綴最長 60 字元 | 過長描述無法表達 | 評估是否提升限制或使用別名機制 |

### 6.3 技術債務標記

**TD-001: Hook 正則同步**
- 位置：ticket-id-validator-hook.py fallback 正則副本
- 建議：建立同步檢查機制（Hook 啟動時驗證版本一致）
- 優先級：中（目前 try-except 已防護，但長期風險）

**TD-002: 後綴動態註冊**
- 位置：KNOWN_TICKET_SUFFIXES 硬編碼列表
- 建議：掃描 .claude/rules/core/ticket-id-conventions.md 自動提取後綴列表
- 優先級：低（當前手動維護可接受，未來擴展性優化）

**TD-003: 後綴規範化**
- 位置：後綴限制為小寫字母+數字+連字號
- 建議：Phase 4 評估是否擴充字元集（考慮下劃線、大小寫混用等）
- 優先級：低（當前規範足以支援現有需求）

## 7. 3b 拆分評估

### 拆分判斷

**評估標準**（tdd-flow.md 規範）：
- 預期修改檔案數：5 個 ✓ (閾值 > 5 個)
- 認知負擔指數：~7-8（3 層邏輯：常數→解析→載入）✗ (閾值 > 10)
- 跨架構層級：2 層（Constants/Parser → Loader）✗ (閾值 > 2 層)
- 依賴關係：Step 3 依賴 Step 2（線性依賴）

### 拆分方案評估

**方案 A：不拆分（推薦）**
- 一個 Ticket，thyme-python-developer 順序執行 5 步
- basil-hook-architect 修改 Hook（可與 Step 3 並行，但時間短）
- 依賴關係簡單，單人上下文連貫
- 修改都圍繞「後綴解析」單一目標，認知負擔低

**方案 B：拆分為 2 個子任務**
- 子任務 1 (3b.1)：Step 1-2 (Constants + Parser 新函式)
- 子任務 2 (3b.2)：Step 3-5 (Loader + Hook + 文件)
- 优点：並行派發
- 缺點：3b.2 必須等 3b.1 完成（仍有依賴），反而增加協調成本

**方案 C：拆分為 3 個子任務**
- 子任務 1：Step 1-2 (Constants + Parser)
- 子任務 2：Step 3 (Loader)
- 子任務 3：Step 4-5 (Hook + 文件)
- 缺點：依賴關係複雜，子任務 2 等 1，子任務 3 等 1，無真正並行

### 結論

**建議：不拆分（方案 A）**

**理由**：
1. **認知負擔適中**（指數 ~8，未超過 10）：單一代理人可負責全部 5 步
2. **依賴關係線性**：拆分後仍需順序執行，並行優勢不明顯
3. **上下文連貫**：從常數到解析到載入，邏輯流暢，修改一致
4. **修改小而精**：除 Step 3（Loader）複雜度稍高外，其餘都是相對簡單的修改
5. **測試獨立**：35+ 個測試可並行執行，驗證效率不受影響

**方案 A 的具體流程**：
```
Phase 3b (不拆分)
├─ thyme-python-developer
│  ├─ Step 1: constants.py 改寫（~30 分鐘）
│  ├─ Step 2: id_parser.py 新函式（~1 小時）
│  ├─ Step 3: ticket_loader.py 更新（~1.5 小時）
│  ├─ Step 5: ticket-id-conventions.md（~30 分鐘）
```
