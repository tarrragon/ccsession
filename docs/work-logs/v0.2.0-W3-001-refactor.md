## Phase 1: 重構計劃與工作日誌建立

### 重構動機與目標

**當前架構的具體問題**：

1. `_print_onboard_result()` 函式過長（84 行），認知負擔指數 11-12
   - 包含語言偵測、Hook 分類、框架檔案檢查、待辦清單等多個邏輯段落
   - 嵌套深度 3 層，變數 5+ 個，分支 10+ 個
   - 違反單一責任原則和認知負擔閾值（<= 10）

2. `parse_hook_classification()` 中的異常處理過度寬泛
   - 捕捉 `Exception`，無法區分具體錯誤類型
   - 降低除錯可靠性和錯誤診斷精度

**重構後期望達成的狀態**：

- 將單一的 84 行輸出函式拆分為 6 個專責函式（各 10-20 行）
- 每個函式認知負擔指數 <= 8，符合「可接受」標準
- 異常處理精確到具體類型 (`OSError`, `UnicodeDecodeError`)
- 保持測試 100% 通過，功能完全不變

**重構如何解決核心問題**：

- 拆分遵循單一責任原則，每個函式只負責一個輸出段落
- 提升程式碼可讀性，降低維護成本
- 精確的異常捕捉提升程式碼健壯性
