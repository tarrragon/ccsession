# Phase 1 功能規格設計：ID 解析包容性擴充和命名字典

## 1. 功能需求分析

### 1.1 問題背景

W43-005 分析發現 309 個 Ticket 檔案中有 22 個（6.7%）使用非標準命名，主要模式為在標準 ID 後附加描述性後綴：

- TDD Phase 標記：`0.1.0-W11-004-phase1-design.md`
- 測試設計產出：`0.1.0-W1-005-phase2-test-design.md`
- 分析產出：`0.1.0-W25-005-analysis.md`
- 策略文件：`0.1.0-W39-001-phase3a-strategy.md`
- 快速參考：`0.1.0-W1-005-test-cases-quick-reference.md`
- 執行報告：`0.1.0-W41-001-phase3b-execution-report.md`

### 1.2 問題根源

目前 `TICKET_ID_PATTERN` 為嚴格正則：

```python
TICKET_ID_PATTERN = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)$"
```

此正則要求 ID 結尾為 `$`（行末），導致帶後綴的檔名無法通過驗證，在 `ticket_loader.py` 中被跳過（不載入）。

### 1.3 影響範圍

| 模組 | 受影響行為 |
|------|-----------|
| `ticket_loader.py:list_tickets()` | 第 97 行使用 TICKET_ID_PATTERN 過濾，非標準檔名被跳過不載入 |
| `id_parser.py:extract_id_components()` | 傳入帶後綴 ID 時返回 None，鏈式計算失敗 |
| `ticket-id-validator-hook.py` | 獨立維護 TICKET_ID_REGEX，與 constants.py 脫鉤，需同步更新 |
| Handoff/chain 追蹤 | 依賴 extract_id_components() 正確解析，後綴 ID 會導致鏈式資訊計算失敗 |

### 1.4 核心使用者場景

場景一：lavender/sage/pepper 代理人在 TDD Phase 執行後，產出 TDD 階段文件（phase1-design.md、phase2-test-design.md 等），這些檔案應能被 ticket CLI 的 list_tickets() 正確識別並與主 Ticket 關聯。

場景二：parsley 執行 3b 拆分後，建立帶有模組描述的子任務檔案，系統應能正確解析其父子關係。

場景三：工程師在不熟悉命名規範時，產出了帶後綴的 Ticket 檔案；validator hook 應提示而非靜默或阻止。

## 2. 功能規格設計

### 2.1 TICKET_ID_PATTERN 擴充規格

#### 目標
修改正則表達式，允許描述性後綴，同時保持向後相容（標準格式仍完全有效）。

#### 新正則規格

```python
# 擴充後的 TICKET_ID_PATTERN（支援描述後綴）
# 格式: {version}-W{wave}-{seq[.seq...]}[-description]
# 描述後綴規則：
#   - 以 "-" 開頭
#   - 只包含小寫字母、數字、連字號
#   - 不以 "-W" 開頭（避免與 wave 格式混淆）
#   - 長度 1-60 字元
TICKET_ID_PATTERN = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)(-[a-z0-9][a-z0-9-]{0,59})?$"
```

#### 正則設計說明

| 群組 | 內容 | 說明 |
|------|------|------|
| `group(1)` | `\d+\.\d+\.\d+` | 版本號（不變） |
| `group(2)` | `\d+` | Wave 號（不變） |
| `group(3)` | `\d+(?:\.\d+)*` | 序號，支援子任務（不變） |
| `group(4)` | `(-[a-z0-9][a-z0-9-]{0,59})?` | 描述後綴（新增，可選） |

#### 向後相容性驗證

| 測試輸入 | 預期結果 | 說明 |
|---------|---------|------|
| `0.31.0-W3-001` | 匹配（group 4 = None） | 標準格式，仍有效 |
| `0.31.0-W3-001.1` | 匹配（group 4 = None） | 子任務格式，仍有效 |
| `0.31.0-W3-001.1.2` | 匹配（group 4 = None） | 深層子任務，仍有效 |
| `0.1.0-W11-004-phase1-design` | 匹配（group 4 = `-phase1-design`） | TDD Phase 標記 |
| `0.1.0-W25-005-analysis` | 匹配（group 4 = `-analysis`） | 分析後綴 |
| `0.1.0-W39-001-phase3a-strategy` | 匹配（group 4 = `-phase3a-strategy`） | 策略後綴 |
| `invalid` | 不匹配 | 無效格式 |
| `0.1.0-W11-004-Phase1` | 不匹配 | 大寫字母不允許 |

#### 邊界條件

| 邊界情況 | 行為規格 |
|---------|---------|
| 描述後綴含大寫 | 不匹配（強制小寫，保持一致性） |
| 描述後綴以 `-W` 開頭 | 需驗證不會誤判為 Wave 格式；正則中 `\d+` 要求數字開頭，自然避免 |
| 描述後綴為純數字（如 `-123`）| 不匹配，避免與序號混淆（要求首字元非純數字：`[a-z0-9]` 後接 `[a-z0-9-]`，實際此處可接受，但需測試確認） |
| 描述後綴超過 60 字元 | 不匹配 |
| 描述後綴為空字串 | 視同無後綴（`?` 使其可選） |

### 2.2 extract_core_ticket_id() 函式規格

#### 目標
新增函式，從可能帶後綴的檔名或 ID 字串中提取核心 Ticket ID。

#### 函式簽名

```python
def extract_core_ticket_id(raw_id: str) -> Optional[str]:
    """
    從可能帶描述後綴的 ID 字串中提取核心 Ticket ID。

    核心 ID 定義：符合 {version}-W{wave}-{sequence} 格式的部分，
    不含描述後綴。

    使用場景：
    - 從帶後綴的檔名提取可比較的 ID（用於鏈式計算、關聯查詢）
    - 驗證帶後綴 ID 是否對應有效的核心 ID

    Args:
        raw_id: 原始 ID 字串，可能帶描述後綴
                範例：
                - "0.1.0-W11-004-phase1-design"（帶後綴）
                - "0.1.0-W11-004"（標準格式）
                - "invalid"（無效格式）

    Returns:
        str: 核心 Ticket ID（不含後綴），如無法解析則返回 None
             範例：
             - "0.1.0-W11-004-phase1-design" -> "0.1.0-W11-004"
             - "0.1.0-W11-004" -> "0.1.0-W11-004"
             - "invalid" -> None

    邊界行為：
    - 輸入為空字串或 None：返回 None
    - 描述後綴含非法字元（大寫等）：返回 None
    - 標準格式（無後綴）：返回原始 ID（group 4 = None）
    - group 4 存在：返回去掉 group 4 後的字串
    """
```

#### 實作邏輯規格

```
輸入: raw_id
  |
  v
使用 TICKET_ID_PATTERN 進行 re.match
  |
  +-- 不匹配 -> 返回 None
  |
  +-- 匹配 ->
        提取 group(1), group(2), group(3)
        重組為 "{group1}-W{group2}-{group3}"
        返回重組的核心 ID
```

#### 邊界測試規格

| 輸入 | 預期輸出 | 說明 |
|------|---------|------|
| `"0.1.0-W11-004-phase1-design"` | `"0.1.0-W11-004"` | 去掉 phase 後綴 |
| `"0.1.0-W11-004.1-phase1-design"` | `"0.1.0-W11-004.1"` | 子任務 + 後綴 |
| `"0.31.0-W3-001"` | `"0.31.0-W3-001"` | 標準格式，原樣返回 |
| `"0.31.0-W3-001.1.2"` | `"0.31.0-W3-001.1.2"` | 深層子任務，原樣返回 |
| `""` | `None` | 空字串 |
| `"invalid"` | `None` | 無效格式 |
| `"0.1.0-W11-004-Phase1"` | `None` | 大寫字元不匹配正則 |
| `"0.1.0-W11-004-"` | `None` | 後綴只有連字號，不匹配正則（`[a-z0-9]` 最少一字元）|

### 2.3 ticket_loader.py 掃描邏輯更新規格

#### 目標
更新 `list_tickets()` 的過濾邏輯，使用 `extract_core_ticket_id()` 正確識別帶後綴的 Ticket 檔案。

#### 現行邏輯（第 94-103 行）

```python
ticket_id = ticket_file.stem           # 從檔名提取（不含副檔名）
if not re.match(TICKET_ID_PATTERN, ticket_id):  # 嚴格格式檢查
    continue
ticket = load_ticket(version, ticket_id)
```

#### 更新後邏輯規格

```
ticket_id_raw = ticket_file.stem       # 從檔名提取原始字串
core_id = extract_core_ticket_id(ticket_id_raw)
if core_id is None:
    continue                            # 無法解析的非 Ticket 檔案（如驗收報告）
ticket = load_ticket(version, core_id)  # 使用核心 ID 載入（YAML frontmatter 中的 id 欄位）
if ticket:
    tickets.append(ticket)
```

#### 行為變更說明

| 場景 | 舊行為 | 新行為 |
|------|--------|--------|
| `0.1.0-W11-004.md` | 匹配，載入 | 匹配，載入（不變） |
| `0.1.0-W11-004-phase1-design.md` | 不匹配，跳過 | 匹配，以 `0.1.0-W11-004` 載入 |
| `acceptance-report.md` | 不匹配，跳過 | 不匹配，跳過（不變） |
| `README.md` | 不匹配，跳過 | 不匹配，跳過（不變） |

#### 重要設計決策：不重複載入防護

當目錄中同時存在 `0.1.0-W11-004.md` 和 `0.1.0-W11-004-phase1-design.md` 時，兩個檔案的核心 ID 都是 `0.1.0-W11-004`，需防止重複載入同一 Ticket。

規格：對 `core_id` 去重，已載入過的 core_id 跳過後續檔案。

```
core_id = extract_core_ticket_id(ticket_id_raw)
if core_id is None or core_id in loaded_core_ids:
    continue
ticket = load_ticket(version, core_id)
if ticket:
    tickets.append(ticket)
    loaded_core_ids.add(core_id)
```

### 2.5 ticket-id-validator-hook.py 寬鬆驗證規格

#### 目標
更新 Hook 的驗證邏輯：對標準 ID 執行完整驗證；對帶後綴 ID 執行寬鬆驗證（提示但不阻止）。

#### 現行驗證邏輯

```python
# 使用 TICKET_ID_REGEX（獨立維護，與 constants.py 不同步）
TICKET_ID_REGEX = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)$"
```

#### 問題：Hook 維護獨立正則副本

SA 建議指出，修改 `constants.py` 後需同步 Hook 中的獨立正則。設計決策：**Hook 應引用 constants.py 的 TICKET_ID_PATTERN**，避免副本不同步問題。

然而 Hook 使用 `#!/usr/bin/env -S uv run` 腳本模式，直接 `import` ticket_system 模組可能有路徑問題。設計方案：在 Hook 中動態引入 ticket_system 模組，或在 Hook 的常數區塊中使用與 constants.py 相同的正則字串，並加入明確的同步提醒注釋。

#### 推薦方案：動態導入（確保同步）

```python
# Hook 中動態導入 ticket_system constants
try:
    sys.path.insert(0, str(Path(__file__).parent.parent / "skills/ticket"))
    from ticket_system.lib.constants import TICKET_ID_PATTERN as TICKET_ID_REGEX
    _using_shared_constant = True
except ImportError:
    # Fallback：維護本地副本（需手動同步）
    TICKET_ID_REGEX = r"^(\d+\.\d+\.\d+)-W(\d+)-(\d+(?:\.\d+)*)(-[a-z0-9][a-z0-9-]{0,59})?$"
    _using_shared_constant = False
```

#### 新驗證行為規格

| 輸入場景 | 驗證行為 | 輸出 |
|---------|---------|------|
| 標準 ID（`0.31.0-W5-001`） | 完整驗證（格式 + 版本一致性） | 通過或警告 |
| 帶後綴 ID（`0.1.0-W11-004-phase1-design`） | 寬鬆驗證（提示識別出後綴，核心 ID 有效） | 提示訊息，不阻止 |
| 無效 ID（`invalid`） | 格式錯誤警告 | 警告訊息 |
| 可識別後綴（如 `-phase1-design`） | 提示：「使用已知後綴模式，核心 ID: X」| 資訊提示 |
| 不可識別後綴（如 `-my-notes`） | 提示：「後綴不在命名規範中，建議使用標準後綴」| 建議提示 |

## 3. 邊界條件分析

### 3.1 正則擴充的邊界條件

| 邊界條件 | 規格行為 | 測試需求 |
|---------|---------|---------|
| 描述後綴為空字串（`0.1.0-W3-001-`） | 不匹配（`[a-z0-9]` 後至少一字元）| 需測試 |
| 描述含多個連續連字號（`0.1.0-W3-001--test`）| 不匹配（`[a-z0-9-]` 中連字號合法，但 `[a-z0-9]` 開頭規則避免 `--`）| 需測試 |
| 描述後綴以數字開頭（`0.1.0-W3-001-3b`）| 匹配，`3b` 符合 `[a-z0-9]` 開頭規則 | 需測試 |
| 非常深的子任務帶後綴（`0.1.0-W3-001.1.2.3-phase1`）| 匹配 | 需測試 |
| 只有後綴無序號（`0.1.0-W3--phase1`）| 不匹配（序號必填）| 需測試 |

### 3.2 extract_core_ticket_id() 的邊界條件

| 邊界條件 | 規格行為 | 測試需求 |
|---------|---------|---------|
| None 輸入 | 返回 None（TypeError 防護）| 需測試 |
| 空字串輸入 | 返回 None | 需測試 |
| 描述後綴含 `-W`（如 `0.1.0-W3-001-W5`） | 不匹配（`W` 為大寫，不符合 `[a-z0-9]`）| 需測試 |
| 描述部分包含數字 `-W5`（如果 `W` 是小寫 `w5`）| `0.1.0-W3-001-w5` 匹配，core_id = `0.1.0-W3-001` | 需澄清：是否允許 `w` 小寫？ 規格：允許，因為是後綴，不是 Wave 格式 |
| 已存在的 22 個非標準檔名 | 全部應能成功提取 core_id | 必須測試全部 22 個 |

### 3.3 ticket_loader.py 不重複載入邊界條件

| 場景 | 規格行為 |
|------|---------|
| 同時存在 `X.md` 和 `X-phase1-design.md` | 優先載入 `X.md`（先排序），`X-phase1-design.md` 被跳過 |
| 只存在 `X-phase1-design.md`，無 `X.md` | 以 `X-phase1-design.md` 作為 core_id `X` 的來源載入 |
| 同一 core_id 有 3 個後綴檔案 | 只載入第一個，後續跳過 |

## 5. 行為場景提取（Given-When-Then）

### 場景 1：標準 ID 解析（正常流程）

```
Given: 版本目錄中存在標準命名的 Ticket 檔案 "0.31.0-W3-001.md"
When: 呼叫 list_tickets("0.31.0")
Then: 成功載入該 Ticket
And: Ticket 的 id 欄位為 "0.31.0-W3-001"
```

### 場景 2：帶後綴 TDD Phase 檔案載入（正常流程 — 擴充後）

```
Given: 版本目錄中存在 "0.1.0-W11-004-phase1-design.md"，無同名標準檔案
When: 呼叫 list_tickets("0.1.0")
Then: 成功載入該 Ticket（以 core_id "0.1.0-W11-004" 載入）
And: Ticket 的 id 欄位來自檔案內部 YAML frontmatter
```
