## Problem Analysis

**分析流程**

1. 讀取 version-release 工具原始碼和文件
   - 版本檔偵測機制：支援 pubspec.yaml（根目錄優先）、ui/pubspec.yaml、package.json、pyproject.toml
   - 版本同步檢查：detect_version_files() → extract_version_from_file() → 對比檢查
   - 當前處理：警告模式（不阻塞）+ "monorepo 場景下此為預期行為" 提示

2. 盤點當前版本現況
   - monorepo 版本：0.1.0（定義在 docs/todolist.yaml current_version）
   - UI 版本：1.0.0+1（ui/pubspec.yaml）
   - Server 版本：無版本定義（server/go.mod 無版本欄位）

3. 分析 Ticket 0.1.0-W1-019 的 handoff 機制
   - 任務鏈 handoff（to-sibling/to-parent/to-child）中，ticket completed 是預期狀態
   - stale 判斷需考慮 direction 上下文
   - 與 monorepo 版本策略相容（同版本 handoff 完全支援）

**發現的問題**

| 問題 | 表現 | 根本原因 |

| 策略不清晰 | 警告輸出無決策標準 | 三層版本的獨立性未定義 |

| 用戶指引不足 | 警告提示無法判斷是否應修復 | 缺乏同步規則和期望值定義 |

## Solution

### S1：三層版本明確定義

#### L1 Monorepo 版本（專案管理層）
- **來源**：docs/todolist.yaml → current_version
- **含義**：整個專案的發布版本（用於 Ticket、Wave、技術債務）
- **語義**：Semantic Versioning，代表整體系統演進
- **更新時機**：功能里程碑、主要架構變更

#### L2 Flutter UI 版本（應用層）
- **來源**：ui/pubspec.yaml → version
- **當前值**：1.0.0+1
- **含義**：Flutter 應用的獨立版本（用於 App Store）
- **獨立性**：YES - 可獨立於 monorepo 升級（特別是小版本）
- **更新時機**：App 功能新增、重大重構、商店發布

#### L3 Go Server 版本（後端層）
- **來源**：server/go.mod（無版本欄位）
- **含義**：Go module 無自身版本，由 monorepo 決定
- **獨立性**：NO - 隱含同步於 monorepo 版本
- **語義**：由 Go module 依賴機制決定

### S2：版本關係矩陣

| 版本層 | 來源 | 獨立性 | 同步規則 | 決定時機 |
|--------|------|--------|---------|---------|
| L1 Monorepo | todolist.yaml | 完全獨立 | N/A（權威） | Wave 完成、功能里程碑 |
| L2 UI | pubspec.yaml | 獨立 | 建議同步（非強制） | App 發布、功能新增 |
| L3 Server | go.mod | 隱含同步 | 同 monorepo | 整體系統發布 |

**核心規則**
1. Monorepo 版本是唯一的「發布版本」定義
2. UI 版本可獨立於 monorepo 升級
3. Server 版本由 monorepo 版本隱含決定

### S3：version-release check 適配建議

#### 改進建議 A：顯式版本同步規則

在 check_version_sync() 中定義：

```python
VERSION_SYNC_POLICY = {
    "monorepo": {
        "source": "todolist.yaml",
        "mandatory": True,
        "description": "整體專案版本"
    },
    "ui": {
        "source": "ui/pubspec.yaml",
        "mandatory": False,
        "recommendation": "建議主版本號一致（如都是 v0.x.0）",
        "description": "Flutter 應用版本，獨立於 monorepo"
    },
    "server": {
        "source": "server/go.mod",
        "mandatory": None,
        "note": "Go module 無自身版本欄位，版本由 monorepo 決定"
    }
}
```

改進 CLI 輸出邏輯：
- 詳細說明每個版本的作用和獨立性
- 提供具體的修復建議
- 區分強制檢查和建議檢查

#### 改進建議 B：新增 .version-release.yaml 配置檔（推薦）

**位置**：`.version-release.yaml`（根目錄）

**關鍵內容**
- 版本定義（來源、語義、獨立性）
- 版本同步規則（發布時、開發中、衝突偵測）
- 版本偵測策略和優先順序
- 發布前檢查項

**優點**
- 中央化配置，文件化策略
- 工具可自動讀取並一致執行
- 易於維護和調整

**預期工作量**
- 設計和文件：2-3 小時
- 工具實現：3-4 小時
- 測試和驗證：1-2 小時
