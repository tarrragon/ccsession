## 1. Go Server 版本管理評估

### 1.1 現狀分析

**server/go.mod 現況**：
```
module github.com/anthropics/ccsession-monitor
go 1.25.1
```

| 項目 | 現狀 |
|------|------|
| 版本欄位 | 無（採 Go module 標準做法） |
| 版本定義來源 | 隱含於 monorepo L1 版本（todolist.yaml） |
| 版本更新需求 | 不需主動更新 go.mod |

### 1.2 Go Module 版本管理最佳實踐

**Go 生態系統中的版本管理原則**：

1. **Go Module 無自身版本定義**（符合現狀）
   - Go module 不在 `go.mod` 中定義版本
   - 版本由 git tag 和 semantic versioning（SemVer）管理
   - 導入該 module 的外部專案透過 `go.mod` 的 `require` 指令指定版本

2. **模組發布版本標記方式**（go.mod 的典型用法）
   ```
   require github.com/anthropics/ccsession-monitor v0.1.1
   ```
   - 版本透過 git tag（`v0.1.1`）標記，不是在 go.mod 內部定義

3. **Monorepo 場景的特例**（當前情況）
   - Server 和 UI 同屬一個 monorepo
   - Server 不是獨立發布的 Go module（內部模組）
   - Monorepo L1 版本應作為 Server 版本的權威來源

### 1.3 改進建議

**選項 A（推薦）：在 `server/version.go` 中定義版本常數**

優點：
- 與 monorepo 版本同步，便於程式碼引用
- 成為運行時可查詢的版本資訊
- 符合 Go 專案通常的版本管理做法

實現方式：
```go
// server/version.go
package main

const Version = "0.1.1"

// 在 main.go 中可使用
fmt.Printf("Server version: %s\n", Version)
```

**選項 B（最小改動）：保持當前狀態**

- 繼續依賴 monorepo L1 版本
- Go module 無自身版本定義符合標準做法
- 適用於內部整合專案

**推薦執行**：在下個 Wave 實施選項 A，便於運行時版本查詢和診斷。

### 2.3 版本提取邏輯驗證

**extract_version_from_ticket_id() 的可靠性**（ticket_validator.py:14-34）：

```python
def extract_version_from_ticket_id(ticket_id: str) -> Optional[str]:
    """從 Ticket ID 提取版本號。格式：0.1.0-W9-002 → "0.1.0" """
    if not ticket_id or "-W" not in ticket_id:
        return None
    return ticket_id.split("-W")[0]
```

| 測試案例 | 輸入 | 輸出 | 評估 |
|---------|------|------|------|
| 標準格式 | `"0.1.0-W1-001"` | `"0.1.0"` | ✓ 正確 |
| 多位數版本 | `"0.31.1-W42-003"` | `"0.31.1"` | ✓ 正確 |
| 跨版本 ID | `"0.2.0-W1-001"` | `"0.2.0"` | ✓ 正確 |
| 無版本 ID | `"invalid"` | `None` | ✓ 正確（安全） |
| 子任務 ID | `"0.1.0-W1-001.1"` | `"0.1.0"` | ✓ 正確 |
| 深層子任務 | `"0.1.0-W1-001.1.2"` | `"0.1.0"` | ✓ 正確 |

**結論**：版本提取邏輯堅實，支援所有 Ticket ID 格式。

### 3.2 Handoff 與版本邊界的相容性

**版本邊界時的 Handoff 流程**：

```
W49 完成（版本 0.1.1）
    ↓
ticket track complete 0.1.1-W49-005
    ↓
/ticket handoff 0.1.1-W49-005
    ↓
direction: to-sibling or to-parent（同版本內）
    ↓
版本 0.1.1 內部完成轉移
    ↓

```

```

    ↓

    ↓

    ↓
前一版本最後的 Ticket：0.1.1-W50-010
    ↓
0.1.1-W50-010 → handoff --to-sibling 0.2.0-W1-001
    ↓
version 提取：0.2.0（自動）
    ↓
load_ticket("0.2.0", "0.2.0-W1-001")  ✓ 找到
```

**結論**：完全相容，版本邊界轉移無任何障礙。

## 6. 技術細節補充

### 6.1 Handoff 目標驗證流程圖

```
handoff 命令
    ↓
_execute_handoff()
    ↓
[P1] _validate_target_ticket_exists(direction)
    ├─ 分解 direction ("to-sibling:0.1.1-W1-001")
    ├─ 提取 target_id "0.1.1-W1-001"
    ├─ 提取 target_version "0.1.1" ← 版本自動偵測
    ├─ load_ticket("0.1.1", "0.1.1-W1-001") ← 載入目標版本
    └─ 驗證結果 → 通過或拋出 HandoffTargetNotFoundError
    ↓
[P2] _validate_no_duplicate_handoff(ticket_id)
    ├─ 檢查 pending/ 目錄
    └─ 驗證無重複 handoff
    ↓
_create_handoff_file(ticket, direction)
```

### 6.2 版本偵測的優先級

handoff.py 中的版本偵測遵循優先級：

1. **顯式指定**：`--version` 參數（如果有）
2. **自動提取**：從 Ticket ID 或 target_id 提取
3. **自動偵測**：呼叫 `resolve_version()` 獲取當前版本

**跨版本 handoff 場景**：
- 自動提取機制（優先級 2）確保版本正確
- 無需使用者手動指定版本
- 最大化使用便利性和容錯能力
