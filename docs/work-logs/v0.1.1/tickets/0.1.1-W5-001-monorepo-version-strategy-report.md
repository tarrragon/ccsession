# Monorepo 版本管理策略分析報告

## 執行摘要

本分析針對專案的 monorepo 版本管理進行全面評估。發現：
1. **現狀**：version-release 工具已具備 monorepo 基礎支援，但缺乏正式的版本策略定義
2. **核心問題**：三層版本（monorepo + UI + Server）的關係未明確定義，導致版本同步檢查設計模糊
3. **建議**：建立正式的 monorepo 版本策略，明確各層版本的獨立性和同步規則，補充配置檔

---

## 第一部分：現況盤點

### 1.1 Version-Release 工具現有能力

**版本檔偵測機制（動態、語言感知）**

```python
VERSION_FILE_CANDIDATES = [
    ("pubspec.yaml", "yaml"),           # Flutter（根目錄優先）
    ("ui/pubspec.yaml", "yaml"),        # Flutter（子目錄備選）
    ("package.json", "json"),           # Node.js
    ("pyproject.toml", "toml"),         # Python
]
```

工具採用**優先順序偵測**，首先查找根目錄 pubspec.yaml，若不存在則查找 ui/pubspec.yaml，依此類推。

**現有行為分析**

| 檢查項目 | 實現方式 | 當前處理 |
|---------|---------|---------|
| 版本號偵測 | 多文件、多語言支援 | 支援 pubspec.yaml（0.1.0）、go.mod（無版本行，故不偵測） |
| 版本同步檢查 | 對所有偵測到的版本檔進行對比 | 警告模式（不阻塞）+ monorepo 說明 |
| 版本不匹配提示 | 輸出 WARNING，附註「monorepo 場景下此為預期行為」 | 已有相關註釋，但策略未正式定義 |
| 分支檢查 | 檢查 git 分支名稱 | 檢查 feature/v{major_minor} 命名慣例 |
| 工作目錄檢查 | 檢查 git status --porcelain | 非致命警告 |

### 1.2 當前版本現況

**monorepo 結構**
```
ccsession/
├── server/
│   ├── go.mod                  # Go module 版本（無版本欄位）
│   └── main.go
├── ui/
│   ├── pubspec.yaml            # Flutter 版本：1.0.0+1
│   └── lib/
├── docs/
│   ├── todolist.yaml           # 專案版本：0.1.0（active）
│   └── work-logs/v0.1.0/
└── CHANGELOG.md
```

**版本現狀**
- **monorepo（專案管理版本）**：`0.1.0`（定義在 todolist.yaml 的 current_version）
- **Flutter UI 版本**：`1.0.0+1`（pubspec.yaml）
- **Go Server 版本**：無版本定義（go.mod 無版本欄位）

**版本差異原因**
- Flutter 版本獨立管理（應用程式版本）
- Go module 無自身版本概念（只有 module path）
- monorepo 版本用於整體專案管理、Ticket 版本、發布計劃

## 第二部分：三層版本定義

### 2.1 版本的三個層級

#### L1：Monorepo 版本（專案管理層）
**來源**：`docs/todolist.yaml` → `current_version`  
**含義**：
- 整個專案（Go + Flutter + 文件）的發布版本
- 用於 Ticket 版本號（如 0.1.0-W1-001）
- 用於版本計劃、Wave 管理、技術債務追蹤
- **決定因素**：功能里程碑、主要架構變更、整體發布時機

**示例**
- v0.1.0：初版規格設計完成
- v0.2.0：新功能發布（假設）
- v0.3.0：架構升級（假設）

**語義**
遵循 Semantic Versioning，代表整個系統的演進版本。

---

#### L2：Flutter UI 版本（應用層）
**來源**：`ui/pubspec.yaml` → `version`  
**當前值**：`1.0.0+1`  
**含義**：
- Flutter 應用程式的發布版本
- 用於 App Store / Google Play 版本管理
- build number（+後面的數字）用於內部迭代

**決定因素**：
- UI 功能新增、大版本重構、App 商店發布時機
- **獨立於 monorepo 版本**（可在 monorepo v0.1.0 期間升級到 UI v1.1.0）

**語義**
- 1.0.0：首次 App 商店發布
- build number：每次構建遞增

---

#### L3：Go Server 版本（後端層）
**來源**：`server/go.mod` 無版本欄位  
**含義**：
- Go module 不具有自身版本（module path 為 github.com/anthropics/ccsession-monitor）
- 版本管理由 Go 模組依賴機制（go.mod 依賴版本）處理
- Server 發布版本由 monorepo 版本決定

**決定因素**：
- 與 monorepo 版本同步（Server 內部 API 版本由 Go module 版本系統決定）

**語義**
- Go modules 使用 semantic versioning 標籤（如 v1.0.0）
- 當前 module 無自身發布版本

---

### 2.2 三層版本關係矩陣

| 版本層 | 來源 | 獨立性 | 同步規則 | 更新時機 |
|--------|------|--------|---------|---------|
| **L1 Monorepo** | todolist.yaml | 完全獨立 | N/A | Wave 完成、功能里程碑 |
| **L2 UI** | pubspec.yaml | 獨立 | 建議同步（非強制） | App 發布、功能新增 |
| **L3 Server** | go.mod | 隱含同步 L1 | 同 monorepo | 整體系統發布 |

**核心規則**
1. **Monorepo 版本**是唯一的「發布版本」定義
2. **UI 版本**可獨立於 monorepo 版本升級（特別是小版本更新）
3. **Server 版本**由 monorepo 版本隱含決定

## 第三部分：version-release check 適配建議

### 3.1 現有檢查流程分析

```python
# 檢查版本同步的五個步驟

1. detect_version_files()
   ↓ 找到 ["ui/pubspec.yaml"]
   ↓
2. extract_version_from_file()
   ↓ 提取 UI 版本 "1.0.0+1"
   ↓
3. 版本對比
   ↓ 比較 "1.0.0+1" vs "0.1.0"（monorepo 版本）
   ↓
4. 輸出警告（非致命）
   WARNING: pubspec.yaml 版本不匹配: 1.0.0+1 vs 0.1.0（monorepo 場景下此為預期行為）
   ↓
5. 繼續執行（檢查通過）
```

**問題分析**

| 問題 | 當前表現 | 根本原因 |
|------|---------|---------|
| **不清楚哪些版本差異是預期的** | 輸出「monorepo 場景下此為預期行為」但未定義策略 | 版本策略未正式規範 |
| **無法驗證 Server 版本** | go.mod 無版本欄位，不檢查 | 後端版本管理不明確 |
| **不知道何時應該強制同步** | 所有不匹配都是警告，無上升到錯誤的判斷 | 缺乏同步規則定義 |
| **user feedback 不清楚** | 使用者看到警告但無法判斷是否應該修復 | 缺乏決策標準 |

### 3.2 改進建議

#### 建議 A：顯式版本同步規則（推薦）

**Step 1：定義同步政策**

在工具的版本同步檢查中，明確定義預期行為：

```python
VERSION_SYNC_POLICY = {
    "monorepo": {
        "source": "todolist.yaml",
        "mandatory": True,         # 必須更新
        "description": "整體專案版本"
    },
    "ui": {
        "source": "ui/pubspec.yaml",
        "mandatory": False,        # 可選同步（獨立版本）
        "recommendation": "建議與 monorepo 版本保持主版本號一致（如都是 v0.x.0）",
        "description": "Flutter 應用版本，獨立於 monorepo"
    },
    "server": {
        "source": "server/go.mod",
        "mandatory": None,         # 尚無標準定義
        "note": "Go module 無自身版本欄位，版本由 monorepo 決定"
    }
}
```

**Step 2：改進檢查邏輯**

```python
def check_version_sync(version: str, config: dict = None) -> Tuple[bool, List[str]]:
    """檢查版本號同步，支援 monorepo 策略"""
    
    errors = []
    warnings = []
    
    # 1. Monorepo 版本（必須）
    monorepo_version = version  # 來自 --version 參數或自動偵測
    
    # 2. UI 版本（建議同步，非強制）
    ui_version = extract_version_from_file("ui/pubspec.yaml")
    if ui_version and ui_version != monorepo_version:
        warnings.append({
            "type": "ui_version_mismatch",
            "monorepo": monorepo_version,
            "ui": ui_version,
            "severity": "warning",
            "action": "可選。建議 UI 主版本號與 monorepo 保持一致"
        })
    
    # 3. Server 版本（暫無檢查）
    # go.mod 無版本欄位，故無檢查
    warnings.append({
        "type": "server_no_version",
        "note": "server/go.mod 無版本定義，版本由 monorepo 決定"
    })
    
    return len(errors) == 0, {"errors": errors, "warnings": warnings}
```

**Step 3：改進 CLI 輸出**

```
✓ 版本同步檢查

monorepo 版本: 0.1.0 (todolist.yaml)
├─ ui/pubspec.yaml: 1.0.0+1
│  └─ 不匹配警告（此為預期行為）
│     理由：UI 應用版本獨立於 monorepo 版本
│     建議：若需同步，請修改 ui/pubspec.yaml → version: 0.1.0
│
└─ server/go.mod: 無版本欄位（正常）
   理由：Go module 無自身版本，由 monorepo 決定
   └─ 版本由 monorepo 0.1.0 決定

結論：✅ 通過（版本策略符合 monorepo）
```

#### 建議 B：新增 .version-release.yaml 配置檔

**用途**

明確定義 monorepo 的版本管理策略，使工具和使用者有一致的理解。

**位置**：`.version-release.yaml`（根目錄）

**內容結構**

```yaml
# .version-release.yaml - Monorepo 版本管理配置

# 版本定義
versions:
  monorepo:
    source: docs/todolist.yaml       # 唯一的權威來源
    key: current_version
    semantic_version: true
    description: "整個專案的發布版本（用於 Ticket、Wave、發布計劃）"
    
  ui:
    source: ui/pubspec.yaml
    key: version
    semantic_version: true
    independent: true                 # UI 可獨立於 monorepo 升級
    description: "Flutter 應用版本（用於 App Store）"
    sync_policy: "optional"            # 建議同步，非強制
    sync_recommendation: "保持主版本號一致（如都是 v0.x.0）"
    
  server:
    source: null                       # go.mod 無版本欄位
    semantic_version: false
    independent: false
    description: "Go Server 版本由 monorepo 版本決定"
    sync_policy: "implicit"            # 隱含同步

# 版本同步規則
sync_rules:
  # 規則 1：發布時同步
  on_release:
    monorepo:
      required: true                   # 必須更新
    ui:
      required: false                  # 可選
      recommendation: |
        建議更新 UI 版本的 build number（例如 1.0.0+5 → 1.0.0+6）
        或更新主版本號以匹配 monorepo（例如 1.0.0 → 0.1.0）
    server:
      required: false                  # 無需更新（由 go.mod 決定）
  
  # 規則 2：開發中的版本邊界
  on_development:
    allow_version_mismatch: true
    reason: "開發中 UI 和 Server 可獨立迭代"
  
  # 規則 3：版本衝突偵測
  conflict_detection:
    ui_ahead_of_monorepo:
      severity: "warning"              # 非致命
      message: "UI 版本大於 monorepo，確認是否故意？"
      action: "手動確認或同步"
    
    ui_behind_monorepo:
      severity: "info"                 # 僅通知
      message: "UI 版本低於 monorepo（正常）"
      action: "無需處理"

# 版本偵測策略
detection:
  # 優先順序：按列表順序檢查
  version_files:
    - path: pubspec.yaml
      type: yaml
      key: version
      context: "monorepo 根目錄版本（如存在）"
      
    - path: ui/pubspec.yaml
      type: yaml
      key: version
      context: "Flutter 應用版本"
      
    - path: server/go.mod
      type: toml
      key: null
      context: "Go module（無版本欄位）"
      
    - path: docs/todolist.yaml
      type: yaml
      key: current_version
      context: "權威的 monorepo 版本來源"
  
  # 自動偵測順序（--version 未指定時）
  auto_detect_order:
    - git_branch                       # feature/v0.1
    - monorepo_version_file            # pubspec.yaml (monorepo)
    - ui_version_file                  # ui/pubspec.yaml
    - git_tag                          # v0.1-final

# 發布前檢查項
preflight_checks:
  version_sync:
    enabled: true
    fail_on_error: false               # monorepo 版本不匹配不阻塞
    warn_on_mismatch: true
```

**優點**

| 優點 | 說明 |
|------|------|
| 中央化配置 | 所有版本策略集中定義 |
| 文件化 | 團隊成員清楚瞭解版本規則 |
| 工具友善 | version-release 可讀取此配置 |
| 靈活性 | 可根據專案需求調整策略 |
| 可追蹤性 | 版本策略變更有 git 歷史 |

## 第四部分：與 Handoff 跨版本機制相容性驗證

### 4.1 Ticket 0.1.0-W1-019 背景

**問題**：ticket resume --list 錯誤判斷任務鏈 handoff 為 stale

**根本原因**：stale 過濾邏輯只檢查 ticket status==completed，未考慮 handoff direction 上下文

**修復**：新增任務鏈 direction 判斷，允許 completed 票的任務鏈 handoff 顯示

### 4.2 與 monorepo 版本策略的相容性

**檢查點 1：Handoff 跨版本時的版本判斷**

```yaml
# 假設的 handoff 場景
handoff:
  from_ticket: 0.1.0-W1-019
  to_ticket: 0.1.0-W2-001
  direction: to-sibling
  from_status: completed
```

**問題**：若從 0.1.0 版本 handoff 到 0.2.0 版本，resume --list 如何判斷版本邊界？

**相容性評估**

| 檢查項 | 當前狀態 | 相容性 | 說明 |
|--------|---------|--------|------|
| 同版本 handoff | 支援 | ✅ 完全相容 | 0.1.0-W1→0.1.0-W2 正常 |
| 跨版本 handoff | 未測試 | ⚠️ 需驗證 | 0.1.0 → 0.2.0 時版本過濾邏輯 |
| 版本號格式 | 標準格式 | ✅ 相容 | {version}-W{wave}-{seq} |
| Direction 判斷 | 字符串匹配 | ✅ 相容 | 不涉及版本號解析 |

**具體風險**

假設實現跨版本 handoff：
```python
# 潛在問題：stale 判斷中的版本檢查
def list_pending_handoffs():
    for handoff in handoffs:
        from_ticket = handoff['from_ticket']  # "0.1.0-W1-019"
        current_version = "0.2.0"  # 新版本
        
        # ❌ 問題：若以版本號判斷是否過濾，可能誤判
        if extract_version(from_ticket) != current_version:
            # 不應該自動過濾！跨版本 handoff 是合法的
            continue
```

**建議**

1. **當前（同版本 handoff）**：完全相容，無需修改
2. **未來（跨版本 handoff）**：在實現時補充版本邊界檢查，不將版本差異視為 stale 判斷因素

### 4.3 版本策略對 Handoff 的影響

**場景**：如果允許 UI 版本獨立於 monorepo 版本

```yaml
# 假設 UI 版本單獨發布
monorepo:  0.1.0
ui:        1.0.0

# Ticket 版本號是否應調整？
Ticket:    0.1.0-W1-001  # 仍基於 monorepo 版本
```

**結論**：
- Ticket 版本號始終基於 monorepo 版本（唯一權威）
- UI 版本獨立性不影響 Ticket 版本策略
- Handoff 機制無需改動

- 初期設計 + 工具改進（< 1 週）

### 結論 3：當前設計與 handoff 機制相容

**評估**：
- 同版本 handoff：完全支援
- 跨版本 handoff：設計相容，但需驗證實現

**建議**：在實現跨版本 handoff 時補充相容性測試
