# Phase 2 測試設計

## 概述

本測試設計為 `.version-release.yaml` 配置檔和 `version_release.py` 工具改進的完整測試戰略。根據 Phase 1 的四個新增函式和改進設計，設計單元測試和整合測試的分層測試案例。

---

## 測試策略

### 分層決策樹

#### 層級分析

- **Layer: 工具模組層（utility functions）**
  - 四個新增函式：`load_version_release_config()`、`get_monorepo_version()`、`check_monorepo_version_sync()`、`print_version_sync_report()`
  - 決策：**單元測試**（Sociable Unit Tests）
  - 理由：Pure functions / I/O boundary functions，適合單元測試隔離

- **Layer: 整合層（version_release.py check 子命令）**
  - 端對端行為：讀取配置 → 讀取版本 → 執行檢查 → 輸出報告
  - 決策：**整合測試**
  - 理由：多個函式組合，含文件 I/O，檢驗完整流程

- **Layer: 文件系統層**
  - 配置檔、版本檔讀取
  - 決策：**單元測試 + 夾具**
  - 理由：Mock 文件系統或使用 tmp 目錄夾具

### Sociable Unit Tests 原則應用

| 依賴類型 | Mock 決策 | 理由 |
|---------|---------|------|
| YAML 檔案讀取 (`yaml.safe_load`) | Mock | 外部依賴（檔案系統） |
| 版本比對邏輯 | 不 Mock | 內層邏輯，直接測試 |
| 版本字串解析 | 不 Mock | 內層邏輯 |
| 報告輸出 (`print()`) | Mock 或 Capture stdout | 外部依賴（I/O） |

### 測試覆蓋範圍

| 維度 | 目標 |
|------|------|
| **函式級覆蓋** | 4 個新函式 100% 行覆蓋 |
| **路徑覆蓋** | 所有分支（正常、邊界、例外）100% 決策覆蓋 |
| **整合流程** | check 子命令 3 個主要場景 100% 覆蓋 |
| **邊界條件** | Phase 1 定義的 11 個邊界情況 100% 覆蓋 |

## 函式層級測試設計

### 測試 1：`load_version_release_config(root: Path) -> dict`

#### 功能驗證：讀取 .version-release.yaml 配置檔

#### 測試案例

**場景 1.1：配置檔存在且格式正確**
```gherkin
Given: .version-release.yaml 存在於專案根目錄
       文件格式為有效 YAML，包含 versions/ui/sync_policy: "optional"
When: 執行 load_version_release_config(root)
Then: 回傳字典，包含 config["versions"]["ui"]["sync_policy"] == "optional"
      字典非空，鍵值結構與 DEFAULT_VERSION_RELEASE_CONFIG 一致
```

**場景 1.2：配置檔不存在**
```gherkin
Given: .version-release.yaml 不存在於專案根目錄
When: 執行 load_version_release_config(root)
Then: 回傳 DEFAULT_VERSION_RELEASE_CONFIG
      無輸出警告（靜默 fallback）
```

**場景 1.3：配置檔 YAML 格式錯誤**
```gherkin
Given: .version-release.yaml 存在但包含無效 YAML 語法
       例如：version: [invalid: yaml: structure}
When: 執行 load_version_release_config(root)
Then: 捕獲 yaml.YAMLError 異常
      輸出警告訊息至 stderr：含 "[WARNING]" 和 ".version-release.yaml 格式錯誤"
      回傳 DEFAULT_VERSION_RELEASE_CONFIG（fallback）
```

**場景 1.4：配置檔部分欄位缺漏**
```gherkin
Given: .version-release.yaml 存在，但缺少 sync_rules.conflict_detection.ui_behind_monorepo
When: 執行 load_version_release_config(root)
Then: 使用 dict.get(key, default) 補充預設值
      回傳完整字典，缺漏欄位使用預設
      無異常拋出
```

**場景 1.5：配置檔包含未知欄位**
```gherkin
Given: .version-release.yaml 包含額外的未預期欄位
       例如：custom_field: "value"
When: 執行 load_version_release_config(root)
Then: 回傳字典包含額外欄位（保留）
      不拋出異常，視為寬鬆驗證
```

#### 邊界條件測試

| 邊界條件 | 預期行為 |
|---------|---------|
| 空的 YAML 檔案 | 回傳 DEFAULT_VERSION_RELEASE_CONFIG |
| 檔案權限不足 (Permission Denied) | 拋出 PermissionError，由呼叫端捕獲 |
| 路徑為目錄而非檔案 | 拋出 IsADirectoryError |
| 根目錄參數為 None | 拋出 TypeError 或 AttributeError |

### 測試 2：`get_monorepo_version(root: Path) -> Optional[str]`

#### 功能驗證：從 docs/todolist.yaml 讀取 L1 monorepo 版本

#### 測試案例

**場景 2.1：todolist.yaml 存在，current_version 欄位有效**
```gherkin
Given: docs/todolist.yaml 存在
       含有 current_version: "0.1.1" 欄位
When: 執行 get_monorepo_version(root)
Then: 回傳 "0.1.1"
      型別為 str
```

**場景 2.2：todolist.yaml 不存在**
```gherkin
Given: docs/todolist.yaml 不存在於專案根目錄/docs/
When: 執行 get_monorepo_version(root)
Then: 回傳 None
      不拋出異常，靜默處理
```

**場景 2.3：todolist.yaml 存在，但 current_version 欄位不存在**
```gherkin
Given: docs/todolist.yaml 存在
       不包含 current_version 欄位
When: 執行 get_monorepo_version(root)
Then: 回傳 None
      不拋出異常
```

**場景 2.4：current_version 為非字串型別**
```gherkin
Given: docs/todolist.yaml 含 current_version: 0.1 (浮點數)
When: 執行 get_monorepo_version(root)
Then: 回傳 "0.1"（轉換為字串）
      或原樣回傳（不強制正規化）
```

**場景 2.5：current_version 為版本風格變體**
```gherkin
Given: docs/todolist.yaml 含 current_version: "v0.1.1" 或 "0.1" 或 "0.1.1-rc1"
When: 執行 get_monorepo_version(root)
Then: 回傳原樣字串，不強制 X.Y.Z 格式
```

#### 邊界條件測試

| 邊界條件 | 預期行為 |
|---------|---------|
| todolist.yaml 為空檔案 | 回傳 None |
| current_version 欄位值為空字串 "" | 回傳 "" |
| 多個相同欄位 (重複定義) | 回傳第一個值 |

### 測試 3：`check_monorepo_version_sync(version: str, config: dict) -> dict`

#### 功能驗證：三層版本同步檢查邏輯

#### 測試案例

**場景 3.1：L1=0.1.1，L2=1.0.0+1（預期不匹配）**
```gherkin
Given: version = "0.1.1"
       config = DEFAULT_VERSION_RELEASE_CONFIG
       ui/pubspec.yaml 存在，version = "1.0.0+1"
       server/go.mod 無版本欄位
When: 執行 check_monorepo_version_sync(version, config)
Then: 回傳 dict 包含：
      - passed: true（策略符合）
      - l1_version: "0.1.1"
      - l2_version: "1.0.0+1"
      - l3_has_version: false
      - messages 清單含 info 級別訊息：
        {level: "info", text: "UI 應用版本獨立於 monorepo 版本"}
      - summary: "通過（版本策略符合 monorepo 三層架構）"
```

**場景 3.2：L2 版本大於 L1（警告）**
```gherkin
Given: version = "0.1.1"
       ui/pubspec.yaml version = "2.0.0"（大於 L1）
When: 執行 check_monorepo_version_sync(version, config)
Then: 回傳 dict 包含：
      - passed: true（不阻塞）
      - messages 含 warning 級別：
        {level: "warning", text: "UI 版本大於 monorepo，確認是否故意？"}
```

**場景 3.3：L2 版本小於 L1（資訊）**
```gherkin
Given: version = "0.2.0"
       ui/pubspec.yaml version = "0.1.0"（小於 L1）
When: 執行 check_monorepo_version_sync(version, config)
Then: 回傳 dict 包含：
      - messages 含 info 級別：
        {level: "info", text: "UI 版本低於 monorepo（正常）"}
```

**場景 3.4：L2 版本等於 L1（一致）**
```gherkin
Given: version = "0.1.1"
       ui/pubspec.yaml version = "0.1.1"
When: 執行 check_monorepo_version_sync(version, config)
Then: 回傳 dict 包含：
      - messages 含 success 級別或無警告訊息
```

**場景 3.5：ui/pubspec.yaml 不存在**
```gherkin
Given: ui/pubspec.yaml 不存在
When: 執行 check_monorepo_version_sync(version, config)
Then: 回傳 dict 包含：
      - l2_version: None
      - messages 含 info：{level: "info", text: "ui/pubspec.yaml 不存在，跳過 L2 檢查"}
      - passed: true（非阻塞）
```

**場景 3.6：版本字串含 build number 的比對**
```gherkin
Given: version = "0.1.1"
       ui/pubspec.yaml version = "1.0.0+1"
When: 執行版本比對（去除 + 後綴）
Then: 比對的主版本為 "1.0.0" vs "0.1.1"，判定為不同
      l2_version 回傳原始值 "1.0.0+1"（保留 +1）
      判定邏輯：L2 > L1 → warning
```

#### 邊界條件測試

| 邊界條件 | 預期行為 |
|---------|---------|
| version 為空字串 "" | 仍執行比對邏輯，回傳 passed=false（空版本不符合） |
| config["versions"]["ui"]["sync_policy"] = "required" | 檢查邏輯應支援（設定值可變） |
| config["versions"]["ui"]["source"] 指向不存在的檔案 | l2_version=None，跳過比對 |
| messages 清單超過 10 條訊息 | 無限制，全部回傳 |

### 測試 4：`print_version_sync_report(sync_result: dict)`

#### 功能驗證：輸出樹狀格式版本同步報告

#### 測試案例

**場景 4.1：正常輸出三層版本對比**
```gherkin
Given: sync_result 包含：
       - l1_version: "0.1.1"
       - l2_version: "1.0.0+1"
       - l3_has_version: false
       - messages: [{level: "info", ...}]
       - summary: "通過..."
When: 執行 print_version_sync_report(sync_result)
Then: 輸出到 stdout 包含：
      - "版本同步檢查" 或類似標題
      - "L1 monorepo 版本: 0.1.1"
      - "L2 ui/pubspec.yaml: 1.0.0+1"
      - "L3 server/go.mod: 無版本欄位（正常）"
      - 樹狀結構符號："|"、"+--" 等
      - 結論文字
```

**場景 4.2：無 L2 版本的輸出**
```gherkin
Given: sync_result 含 l2_version: None
When: 執行 print_version_sync_report(sync_result)
Then: 輸出顯示：
      - "L2 ui/pubspec.yaml: 未偵測到" 或類似
      - 不顯示版本值
```

**場景 4.3：包含警告訊息的輸出**
```gherkin
Given: sync_result 含 messages: [{level: "warning", text: "UI 版本大於 monorepo..."}]
When: 執行 print_version_sync_report(sync_result)
Then: 輸出包含：
      - "[WARNING]" 標記
      - 警告文字
      - 視覺上區別於 info/success
```

**場景 4.4：空的訊息清單**
```gherkin
Given: sync_result 含 messages: []
When: 執行 print_version_sync_report(sync_result)
Then: 輸出版本資訊，無訊息部分
      結論仍輸出
```

#### 邊界條件測試

| 邊界條件 | 預期行為 |
|---------|---------|
| sync_result 缺少某個鍵（如 l2_version） | 使用 dict.get() 帶預設值或輸出 "未知" |
| summary 含特殊字元或 emoji（若有） | 安全地輸出到 stdout |
| stdout 寬度有限（80 字元終端） | 文字換行或保持簡潔 |

### 測試 5：整合測試 - `check` 子命令完整流程

#### 目的：驗證四個函式的端對端組合行為

#### 測試案例

**場景 5.1：標準 check 流程 - 正常三層版本對比**
```gherkin
Given: 夾具目錄包含：
       - docs/todolist.yaml with current_version: "0.1.1"
       - ui/pubspec.yaml with version: "1.0.0+1"
       - server/go.mod（無版本欄位）
       - .version-release.yaml 存在且格式正確
When: 執行 uv run version_release.py check（或直接呼叫 check_version_sync("0.1.1")）
Then: 命令輸出：
      - 版本同步報告，含三層版本資訊
      - 結論為「通過」
      - exit code: 0
      - 無錯誤訊息
```

**場景 5.2：.version-release.yaml 不存在時的 check**
```gherkin
Given: .version-release.yaml 不存在
       其他檔案同場景 5.1
When: 執行 check（呼叫 check_version_sync）
Then: 執行邏輯同 5.1（使用 DEFAULT_VERSION_RELEASE_CONFIG）
      輸出報告無警告訊息
      exit code: 0
```

**場景 5.3：check 時 L1 todolist.yaml 不存在**
```gherkin
Given: docs/todolist.yaml 不存在
When: 執行 check

      exit code: 1
```

**場景 5.4：L2 版本大於 L1 的 check**
```gherkin
Given: 同場景 5.1，但 ui/pubspec.yaml version: "2.0.0"
When: 執行 check
Then: 輸出含 [WARNING] UI 版本大於 monorepo
      exit code: 0（不阻塞）
```

## Mock 設計與測試資料

### Mock 策略

| 依賴 | Mock 方式 | 夾具/替代 |
|-----|---------|----------|
| 檔案系統（read yaml/toml） | tmpdir pytest 夾具 | 建立臨時檔案目錄 |
| YAML 讀取例外 | monkeypatch.setattr() | 注入 yaml.YAMLError |
| stdout 輸出 | capsys pytest 夾具 | 捕獲並驗證輸出內容 |
| Path 物件 | 直接使用 pathlib.Path | 真實路徑操作 |

### 測試資料檔案清單

#### 有效配置夾具
- `fixtures/valid_version_release.yaml` — 完整、格式正確的配置檔
- `fixtures/minimal_config.yaml` — 最小必填欄位
- `fixtures/version_files/todolist_valid.yaml` — 含有效 current_version
- `fixtures/version_files/pubspec_valid.yaml` — 含有效 version 欄位

#### 無效配置夾具
- `fixtures/invalid_yaml_config.yaml` — 格式錯誤的 YAML
- `fixtures/version_files/todolist_missing.yaml` — 缺少 current_version 欄位
- `fixtures/version_files/pubspec_missing.yaml` — 檔案完全不存在（由測試動態刪除）

### Mock 實例範例

```python
# Mock yaml.safe_load 拋出例外
@pytest.fixture
def mock_yaml_error(monkeypatch):
    def raise_yaml_error(*args, **kwargs):
        raise yaml.YAMLError("Invalid YAML")
    monkeypatch.setattr("yaml.safe_load", raise_yaml_error)

# 捕獲 stdout 輸出
def test_print_version_sync_report(capsys):
    print_version_sync_report(sync_result)
    captured = capsys.readouterr()
    assert "版本同步檢查" in captured.out
```

### 驗證標準

| 驗收項 | 驗證方式 | 成功條件 |
|--------|---------|---------|
| 行覆蓋率 | `pytest --cov` | >= 90% |
| 分支覆蓋率 | 人工檢查 | 所有 if/else 均有測試 |
| 邊界條件 | 指定測試 | 11 個邊界條件 100% 測試 |
| 整合流程 | 3 個場景測試 | 所有場景 exit code 正確 |
| Mock 隔離 | 檢查 import mock 使用 | 外部依賴已隔離 |

### 常見測試失敗原因與排查

| 失敗跡象 | 可能原因 | 排查步驟 |
|---------|--------|--------|
| YAML parse 失敗 | fixtures 檔案格式錯誤 | `python -c "import yaml; yaml.safe_load(open('...'))"` |
| 路徑找不到 | tmpdir 夾具未設定 | 確認 pytest tmpdir 依賴已載入 |
| stdout 驗證失敗 | 預期輸出格式變更 | 檢查 print_version_sync_report() 實作細節 |
| 版本比對邏輯錯誤 | build number 處理不當 | 確認去除 `+` 後綴的邏輯 |
