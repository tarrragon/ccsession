# 0.1.1-W8-001：SA 一致性審查報告 — `/worktree` SKILL 設計

**審查日期**: 2026-03-14  
**審查角色**: System Analyst (Consistency Perspective)  
**審查結果**: ✅ 建議**新建** `/worktree` SKILL（而非擴充 `branch-worktree-guardian`）

---

## 執行摘要

設計方向：整合 Ticket-driven worktree 生命週期為標準化 SKILL，包含建立、查詢、合併、清理四個核心操作。

**一致性審查發現**：
- ✅ 與 `/ticket` 和 `/tdd` 的子命令風格完全一致
- ⚠️ 與 `branch-worktree-guardian` 有功能重疊，但不是完全重複（merge 流程和自動命名是新增特性）
- ❌ 決策樹中未定義 worktree 觸發時機（需補充）
- ❌ Ticket 欄位缺少 `git:` 區塊定義
- ❌ 缺少 Ticket-merge 原子性規則（需澄清時序）

**最終建議**：新建 `/worktree` SKILL，復用 `branch-worktree-guardian` 的底層函式以避免重複實現。

## 重疊分析：`/worktree` vs `branch-worktree-guardian`

### 功能對比

| 操作 | branch-worktree-guardian | /worktree (新) | 重疊程度 |
|------|--------------------------|----------------|--------|
| 建立分支 + worktree | ✅ `create_feature_worktree.py` | ✅ `/worktree setup` | 有重疊（但 /worktree 新增自動命名） |
| 查看 worktree 列表 | ✅ `git worktree list` | ✅ `/worktree status` | 有重疊（但 /worktree 增加 Ticket 視角） |
| 清理已合併 worktree | ✅ 原生 git 命令 | ✅ `/worktree cleanup` | 有重疊（但 /worktree 自動偵測） |
| **合併流程引導** | ❌ 無 | ✅ `/worktree merge` | **全新功能** |
| **Ticket-driven 命名** | ❌ 無 | ✅ `/worktree setup` | **全新功能** |
| **Ticket 欄位同步** | ❌ 無 | ✅ `/worktree setup/merge` | **全新功能** |

**結論**：新增功能（merge、自動命名、Ticket 同步）遠多於重疊部分。

### 使用者心智模型差異

- **`branch-worktree-guardian`**：使用者想的是「git 分支管理」（低層級）
- **`/worktree`**：使用者想的是「Ticket 隔離開發」（高層級）

→ 完全不同的抽象層，新建 SKILL 能減少認知混亂。

## 設計缺口識別

### 1. Worktree 觸發時機（決策樹缺失）

**問題**：決策樹 Checkpoint 0 未明確定義何時應建立獨立 worktree。

**建議規則**：
```
多個並行獨立 Ticket（同 Wave > 1）→ 執行 /worktree setup <ticket-id>
單一 Ticket → 使用主 worktree（無需建立）
PM 明確指定 → 執行 /worktree setup <ticket-id>
```

**涉及文件**：
- `.claude/rules/core/decision-tree.md` → Checkpoint 0（新增子分支）
- `.claude/rules/flows/tdd-flow.md` → 新增「Worktree 策略」小節

### 2. Ticket-Merge 原子性（生命週期缺失）

**問題**：Ticket 完成和 git merge 的同步規則未定義。

**風險**：
- Ticket completed 但分支未合併
- Ticket in_progress 但分支已刪除

**建議時序**：
```
ticket track complete <id>
  ↓
工作日誌更新完成
  ↓
[若使用 worktree] /worktree merge <id>
  ↓
merge 成功 → git commit
  ↓
[Checkpoint 1.5] 錯誤學習 (AskUserQuestion #16)
  ↓
[Checkpoint 2] 情境評估
```

**涉及文件**：
- `.claude/rules/flows/ticket-lifecycle.md` → 新增「Git 分支管理」小節

### 3. Ticket 欄位擴展（Schema 缺失）

**問題**：Ticket 無 git 相關欄位，無法追蹤分支狀態。

**建議新增 Ticket Frontmatter 欄位**：
```yaml
git:
  branch: "feat/0.1.1-W8-001-worktree-automation"
  worktree: "../ccsession-0.1.1-W8-001-worktree"
  status: "setup"  # setup / merged / cleanup
  setup_date: 2026-03-14
  merge_date: null
```

**涉及文件**：
- `.claude/skills/ticket/references/create-command.md` → 新增 git 欄位規格

### 4. Worktree 狀態追蹤（持久化缺失）

**問題**：無持久化的 worktree 狀態記錄，`/worktree status` 需手動掃描。

**建議新增**：
```
.claude/worktrees/
├── .gitkeep
└── 0.1.1-W8-001.json  # Ticket 對應的 worktree 狀態
```

**內容範例**：
```json
{
  "ticket_id": "0.1.1-W8-001",
  "branch": "feat/0.1.1-W8-001-worktree-automation",
  "worktree_path": "../ccsession-0.1.1-W8-001-worktree",
  "status": "merged",
  "setup_timestamp": "2026-03-14T10:30:00Z",
  "merge_timestamp": "2026-03-14T15:45:00Z"
}
```

## 新建 vs 擴充決策

### 決策矩陣

| 維度 | 評估 |
|------|------|
| **功能新增 vs 重疊** | 新增 > 重疊（merge、自動命名、Ticket 同步）→ **新建** |
| **使用者心智** | 完全不同抽象層 → **新建** |
| **命令風格** | 都是子命令式，統一 → **新建** |
| **維護邊界** | 清晰分工（git 工具 vs Ticket 工具）→ **新建** |
| **使用頻率** | PM 經常需要原子操作（setup + complete + merge）→ **新建** |

**決策**：✅ **新建 `/worktree` SKILL**（同時復用 `branch-worktree-guardian` 的底層函式）

### 協作方式

```python
# /worktree 復用 branch-worktree-guardian 的底層實現
from .branch_worktree_guardian.lib.worktree_utils import (
    create_feature_worktree,
    list_worktrees,
    cleanup_worktree
)

class TicketWorktreeManager:
    """在 branch-worktree-guardian 基礎上加入 Ticket 語境"""
    
    def setup(self, ticket_id: str, ticket_title: str):
        branch = auto_branch_name(ticket_id, ticket_title)
        worktree = auto_worktree_dir(ticket_id, ticket_title)
        create_feature_worktree(branch, str(worktree))  # 復用
        self._sync_ticket_git_fields(ticket_id, branch, worktree)
    
    def merge(self, ticket_id: str):
        """merge 流程（branch-worktree-guardian 缺少的部分）"""
        # ... 原生 git merge 邏輯 ...
        self._sync_ticket_merge_status(ticket_id, "merged")
```

## 實施建議

### 分階段 PR 計畫

#### PR 1：核心功能（必須）
- `/worktree setup <ticket-id>` → 自動命名 + Ticket 同步
- `/worktree status` → Ticket 視角的 worktree 列表
- `/worktree cleanup` → 自動偵測已合併

#### PR 2：Merge 引導（重要）
- `/worktree merge <ticket-id>` → 完整 merge 流程
- 衝突偵測和提示
- 自動或手動清理選項

#### PR 3：系統整合（增強）

- 更新決策樹 Checkpoint 0
