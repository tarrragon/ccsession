## 概述

本報告針對 ccsession 專案的 11 個 Use Case 進行逐一審查，評估：
1. 技術可行性（是否能在 Go Backend + Flutter Frontend 架構下實現）

3. 實作邊界（各 UC 的職責範圍）
4. 架構對齊程度（與現有 Go + Flutter 設計的匹配程度）

**目前實作狀態**: 極早期骨架
- `server/main.go`：僅有 `/health` 端點
- `ui/lib/main.dart`：僅有佔位頁面

所有 11 個 UC 均尚未實作，屬於「規格定義完成，實作待開始」的狀態。

### UC-001: Session 列表瀏覽

**優先級**: P0（確認）

**技術可行性**: 高

| 面向 | 評估 |
|------|------|
| Flutter Sidebar 分組顯示 | Flutter ListView + grouped rendering，標準做法，可行 |
| 狀態指示燈（Active/Idle/Completed） | 用顏色圓點 Widget，簡單 |
| 動畫（slide+fade）| AnimatedList 或 Flutter 動畫 API，中等複雜度 |
| 即時更新（session_status_change） | 透過 WebSocket 事件更新 Provider 狀態，然後 rebuild，標準做法 |
| 摘要 fallback 邏輯 | Backend 已在 UC-008 定義優先來源，Frontend 直接顯示 Backend 提供的 summary 欄位即可 |

**實作邊界**:
- Frontend：Sidebar Widget、狀態分組邏輯、動畫效果、相對時間格式化
- Backend：透過 UC-008/UC-009 提供 session metadata
- Frontend 不應自行解析 sessions-index.json，一切 metadata 由 Backend 透過 WebSocket 提供

**潛在問題**:
- 動畫規格（A2/A4 狀態回升 + 高亮）增加實作複雜度。建議 Phase 1 只實作基本 list 更新，Phase 4 再補動畫。
- 批量狀態變更的動畫合併邏輯（避免頻繁跳動）可先設定 debounce，複雜度可控。

**架構對齊**: 完全對齊。純 Frontend UI，依賴 Backend WebSocket 提供資料。

### UC-002: Session 對話檢視

**優先級**: P0（確認）

**技術可行性**: 高

| 面向 | 評估 |
|------|------|
| Chat bubble UI | Flutter Column + 自訂 Widget，可行 |
| Markdown 渲染 | 使用 `flutter_markdown` package，成熟可用 |
| Tool use 可摺疊區塊 | `ExpansionTile` 或自訂 Widget，可行 |
| Timestamp-based cursor 分頁 | 設計合理（`before` 參數），Backend 有完整規格 |
| 自動捲動邏輯 | `ScrollController` + 位置偵測，中等複雜度 |
| 次級排序（同 timestamp 時用 lineIndex） | Backend 已負責排序，Frontend 只需直接渲染 |

**實作邊界**:
- Frontend：所有 UI 渲染、捲動邏輯、Markdown 渲染、摺疊/展開狀態

**潛在問題（需確認）**:
- **分頁連續性保證中的「無遺漏」規則**：Backend 需確保同一 timestamp 的所有事件在同一批次回傳，實作時需特別處理邊界情況（當 `before` timestamp 正好等於某批事件的 timestamp 時，要回傳全部該 timestamp 的事件）。這個邊界邏輯需在 UC-007/UC-009 實作時特別注意。
- 自動捲動與分頁載入同時發生的優先級邏輯（UC-002 A2 中已定義），實作需仔細。

**架構對齊**: 完全對齊。Frontend 渲染，Backend 負責資料查詢和排序。

### UC-003: 即時事件串流

**優先級**: P0（確認）

**技術可行性**: 高

| 面向 | 評估 |
|------|------|
| 端到端延遲 < 500ms | 效能預算分配合理（File Watcher 100ms + Parser 50ms + State 20ms + WS 30ms + Flutter 200ms = 400ms，有 buffer） |
| 事件吞吐量 >= 100 events/sec | Go goroutine 處理，可達到 |
| 不完整 JSON 行容錯 | UC-006/UC-007 已定義 offset-based 讀取，忽略不完整行，可行 |
| 未訂閱 session 事件不佔頻寬 | 訂閱機制（UC-009）設計正確，只推送給已訂閱 Client |

**實作邊界**:
- 這是一個跨層 UC，整合 UC-006（File Watching）、UC-007（Parsing）、UC-009（WebSocket）的能力
- 沒有獨立的新實作，是前述 UC 的整合測試場景

**效能需求評估**:
- Flutter 200ms 預算是最大的風險點。大量連續事件（burst）時可能觸發頻繁 UI rebuild。
- 建議實作 event batching：在 Flutter 側，短時間內的多個 `session_event` 可批次合併後一次 rebuild。

**架構對齊**: 完全對齊。跨層協作的整合 UC。

### UC-004: 多 Session 分割畫面

**優先級**: P1（確認）

**技術可行性**: 中高

| 面向 | 評估 |
|------|------|
| 4 種佈局模式 | Flutter Row/Column + GridView，可行 |
| 面板獨立選擇 session | 每個 Panel 維護自己的 sessionId state，可行 |

| 從 Sidebar 拖拉 session 到面板 | `Draggable` + `DragTarget`，Flutter 原生支援 |
| 持久化（shared_preferences） | `shared_preferences` package，標準做法 |

**實作邊界**:
- 純 Frontend，Backend 不需要變更

**潛在問題（建議降級）**:

- **面板比例可調（A3）**：自訂拖拉分隔線，實作中等複雜。Phase 1 可固定比例（各 50%），Phase 4 再開放自訂。

**Phase 1 建議最小範圍**:
- 4 種佈局模式切換（點擊選擇，不含拖拉）
- 各面板從 Dropdown 選擇 session
- layoutMode 和 sessionId 持久化（scrollPosition 留 Phase 4）

**架構對齊**: 完全對齊。純 Frontend，Backend 透明。

### UC-006: JSONL 檔案監控

**優先級**: P0（確認）

**技術可行性**: 高

| 面向 | 評估 |
|------|------|
| fsnotify 監控 | Go 成熟 library，跨平台支援 macOS/Linux/Windows |
| Offset-based 增量讀取 | 標準 I/O 技巧，可行 |
| 新 Session 動態偵測 | fsnotify Create 事件 + 新 goroutine，可行 |
| 初始載入（最後 N 行，預設 1000） | 讀取整個檔案後取後 N 行，或更高效的 tail 演算法 |
| 同時監控 100+ 個檔案 | Go goroutine + channel 設計可支援 |

**實作邊界**:
- Go Backend 內部，負責所有檔案系統互動
- 不涉及 Frontend

**潛在問題**:
- **Linux inotify 限制**：生產環境需提醒用戶調整 `max_user_watches`，或實作自動檢測和警告。
- **每個 Project 目錄 50 個檔案上限**（A2 A2 多 JSONL 處理）：這個設計決策合理，避免監控過多舊 session。
- **初始載入**：讀取 1000 行（預設）可能讀取很大的 JSONL 文件。建議實作 tail 演算法（從後方讀取，避免讀整個檔案），而非讀取整個檔案後截取。

**架構對齊**: 完全對齊。Go Backend 資料層最底層元件。

### UC-007: JSONL 事件解析

**優先級**: P0（確認）

**技術可行性**: 高

| 面向 | 評估 |
|------|------|
| JSON 行解析 | `encoding/json` 標準庫，可行 |
| 各 content type 處理 | switch/case 分支，可行 |
| Boundary 信號（isLastContent） | 在最後一個子事件設置 flag，設計清晰 |
| 未知欄位 WARN log（UC-011 配合） | 白名單 map + WARN log，可行 |

**實作邊界**:
- Go Backend 解析層
- 輸出：`SessionEvent` struct 到 channel

**潛在問題（需釐清）**:
- **messageId 的構成方式**：文件提到 `sessionID + timestamp + lineNumber`，但 JSONL 的頂層已有 `uuid` 欄位（spec.md 2.4 節）。應直接使用 JSONL 的 `uuid` 欄位作為 `messageId`，而非自行組合。這樣更精確且不重複定義。
- **tool_result 的 parent 指向**：JSONL 中 `tool_result` 是獨立行（type: "tool_result"），不在 assistant 的 content array 中。UC-007 的事件類型矩陣僅定義 assistant content array 的類型，需確認 top-level tool_result 如何處理。建議參考 spec.md 2.4 節重新確認。

**架構對齊**: 完全對齊。Go Backend 資料轉換層。

**需後續確認項目**:
1. `messageId` 使用 JSONL `uuid` 欄位還是自行組合？建議使用 `uuid`。
2. Top-level `tool_result` 行（非在 assistant content array 中）的處理方式。

## 需後續確認的問題（待 lavender-interface-designer 確認）

### 問題 1：UC-007 messageId 定義

**問題**：UC-007 說明 `messageId` 的構成為 `sessionID + timestamp + lineNumber`，但 JSONL 頂層已有 `uuid` 欄位（spec.md 2.4）。

**建議**：直接使用 JSONL 行的 `uuid` 欄位作為 `messageId`，更精確且避免重複定義。

**需要確認**：`uuid` 欄位是否在所有 JSONL 行中都存在？如果某些輔助類型（progress、queue-operation）沒有 `uuid`，需要 fallback 方案。

### 問題 2：UC-007 top-level tool_result 處理

**問題**：spec.md 2.4 說明 JSONL 中有 `type: "tool_result"` 的頂層事件（不在 assistant content array 中）。但 UC-007 的事件矩陣只定義了 assistant content array 的 tool_result。

**建議**：確認 top-level `tool_result` 行如何處理：
- 是否映射為 `SessionEvent` type `tool_result`？
- 與 assistant content array 中的 tool_result 是否使用相同結構？

### 問題 3：UC-004 Phase 1 範圍降級

**問題**：A1（拖拉 session 到面板）和 A3（拖拉調整面板大小）在 Phase 1 範圍中增加不少複雜度。

**建議**：Phase 1 只實作：
- 佈局模式切換（點擊按鈕）
- 各面板從 Dropdown 選擇 session
- layoutMode 和 sessionId 持久化

Phase 3 補充拖拉互動，Phase 4 補充面板大小比例持久化。
