# Phase 1 功能規格設計

## 功能概述

本功能包含兩個緊密相關的交付物：

1. **新建 `.version-release.yaml`**：在專案根目錄建立 monorepo 版本管理策略配置檔，作為三層版本（Monorepo / UI / Server）定義的中央化配置來源。

2. **改進 `version_release.py` check 子命令**：讓 `check` 子命令讀取 `.version-release.yaml`，取代目前硬編碼的版本比對邏輯，輸出更具語意的三層版本對比報告及同步建議。

3. **更新 `SKILL.md`**：補充版本策略說明章節，反映 monorepo 三層版本架構。

---

## 需求分析

### 核心問題

目前 `version_release.py` 的 `check_version_sync()` 函式硬編碼版本比對行為：

- 只知道「版本不匹配」，但無法區分「L2 UI 獨立升級（正常）」和「真正的版本衝突（需警告）」
- 輸出訊息如 `pubspec.yaml 版本不匹配: 1.0.0+1 vs 0.1.0（monorepo 場景下此為預期行為）` 對使用者不夠友善，無法判斷是否需要動作
- 沒有說明三層版本的語義差異

### 使用者情境

使用者在版本發布前執行 `check` 子命令，期望：

1. 看到三層版本現況（L1 Monorepo、L2 UI、L3 Server）
2. 清楚知道哪些不匹配是「預期行為」、哪些是「需要修正」
3. 獲得具體的同步建議（例如：「L2 UI 版本可選同步」）

## 功能規格

### 功能 1：`.version-release.yaml` 配置檔

#### 輸入（使用者操作）

- 不需要使用者輸入，此為靜態配置檔案
- 使用者可手動編輯此檔案調整版本策略

#### 輸出結果

- 專案根目錄存在 `.version-release.yaml` 檔案
- `version_release.py` 可讀取此檔案並依其定義執行版本檢查

#### 正常流程

1. 配置檔建立在專案根目錄（`ccsession/.version-release.yaml`）
2. 包含三層版本定義：`versions.monorepo`、`versions.ui`、`versions.server`
3. 包含同步規則：`sync_rules.on_release`、`sync_rules.on_development`
4. 包含衝突偵測：`sync_rules.conflict_detection`
5. 包含版本偵測策略：`detection.version_files`、`detection.auto_detect_order`
6. 包含發布前檢查設定：`preflight_checks.version_sync`

#### 邊界條件

| 情況 | 處理方式 |
|------|---------|
| 配置檔不存在 | `version_release.py` 使用內建預設值，行為與現有一致 |
| 配置檔格式錯誤 | 捕獲 YAML 解析例外，fallback 至預設值，輸出警告訊息 |
| 部分欄位缺漏 | 使用 `dict.get()` 帶預設值，不強制所有欄位存在 |
| `versions.server.source: null` | 顯示 Server 版本由 monorepo 決定，不嘗試讀取 |

### 功能 2：`check` 子命令讀取配置並輸出三層版本對比

#### 輸入

- 命令列：`uv run version_release.py check [--version X.Y.Z]`
- 配置來源：`.version-release.yaml`（可選，不存在則 fallback）

#### 輸出結果

三層版本對比報告，範例格式：

```
版本同步檢查

monorepo 版本: 0.1.1 (docs/todolist.yaml)
|-- ui/pubspec.yaml: 1.0.0+1
|   +-- 狀態：不匹配（預期行為）
|       理由：UI 應用版本獨立於 monorepo 版本
|       建議：若需同步，可更新 ui/pubspec.yaml -> version: 0.1.1
|
+-- server/go.mod: 無版本欄位（正常）
    理由：Go module 無自身版本，由 monorepo 決定

結論：通過（版本策略符合 monorepo 三層架構）
```

#### 正常流程

1. 讀取 `.version-release.yaml`（或使用預設值）
2. 從 `docs/todolist.yaml` 讀取 L1 monorepo 版本（`current_version` 欄位）
3. 從 `ui/pubspec.yaml` 讀取 L2 UI 版本
4. 偵測 L3 Server 版本（`server/go.mod` 無版本欄位，顯示特殊說明）
5. 依 `sync_policy` 判斷各層版本狀態（`required` / `optional` / `implicit`）
6. 輸出三層版本對比，含語意說明和建議動作
7. 回傳：通過（即使版本不匹配，只要符合策略定義）

#### 異常流程

| 異常情況 | 輸出 | 是否阻塞 |
|---------|------|---------|
| `.version-release.yaml` 不存在 | 使用預設值，不輸出警告 | 否 |
| `.version-release.yaml` 格式錯誤 | `WARNING: 配置檔格式錯誤，使用預設值` | 否 |
| `docs/todolist.yaml` 不存在 | `ERROR: 找不到 monorepo 版本來源` | 是（check 失敗） |
| `ui/pubspec.yaml` 不存在 | `INFO: ui/pubspec.yaml 不存在，跳過 L2 檢查` | 否 |
| L2 版本大於 L1 | `WARNING: UI 版本大於 monorepo，確認是否故意？` | 否 |

### 功能 3：`SKILL.md` 版本策略說明章節

新增章節說明 monorepo 三層版本架構，讓使用者理解版本比對行為的設計意圖。

## API / 函式介面定義

### 新增函式

#### `load_version_release_config(root: Path) -> dict`

```python
def load_version_release_config(root: Path) -> dict:
    """
    讀取 .version-release.yaml 配置檔。

    需求：功能 1 配置檔讀取
    邊界條件：
    - 配置檔不存在 -> 回傳 DEFAULT_VERSION_RELEASE_CONFIG
    - 配置檔格式錯誤 -> 輸出 warning，回傳 DEFAULT_VERSION_RELEASE_CONFIG
    - 部分欄位缺漏 -> dict.get() 帶預設值

    Args:
        root: 專案根目錄（Path 物件）

    Returns:
        配置字典，結構與 .version-release.yaml 一致
        保證回傳值不為 None
    """
```

#### `get_monorepo_version(root: Path) -> Optional[str]`

```python
def get_monorepo_version(root: Path) -> Optional[str]:
    """
    從 docs/todolist.yaml 讀取 L1 monorepo 版本。

    需求：三層版本中 L1 為唯一權威來源
    邊界條件：
    - todolist.yaml 不存在 -> 回傳 None
    - current_version 欄位不存在 -> 回傳 None
    - 版本格式非 X.Y.Z -> 原樣回傳（不強制正規化）

    Args:
        root: 專案根目錄

    Returns:
        版本字串（例如 "0.1.1"）或 None
    """
```

#### `check_monorepo_version_sync(version: str, config: dict) -> dict`

```python
def check_monorepo_version_sync(version: str, config: dict) -> dict:
    """
    執行三層版本同步檢查，取代原有 check_version_sync()。

    需求：功能 2 三層版本對比
    邊界條件：
    - L2 版本不存在 -> 跳過 L2 檢查
    - L3 source 為 null -> 顯示 Server 版本由 monorepo 決定
    - L2 版本大於 L1 -> 輸出 warning（severity: "warning"）
    - L2 版本小於 L1 -> 輸出 info（severity: "info"）

    Args:
        version: L1 monorepo 版本（例如 "0.1.1"）
        config: load_version_release_config() 回傳的配置字典

    Returns:
        {
            "passed": bool,
            "l1_version": str,           # L1 monorepo 版本
            "l2_version": Optional[str], # L2 UI 版本（可能為 None）
            "l3_has_version": bool,      # L3 是否有版本欄位
            "messages": List[dict],      # 每條訊息 {level, text, recommendation}
            "summary": str               # 整體結論文字
        }
    """
```

#### `print_version_sync_report(sync_result: dict)`

```python
def print_version_sync_report(sync_result: dict):
    """
    輸出三層版本對比報告（樹狀結構）。

    需求：功能 2 CLI 輸出格式
    邊界條件：
    - 無 L2 版本 -> 顯示 "ui/pubspec.yaml: 未偵測到"
    - 無 L3 版本 -> 顯示 "server/go.mod: 無版本欄位（正常）"

    Args:
        sync_result: check_monorepo_version_sync() 的回傳值

    Side effects:
        打印到 stdout（使用現有 print_* 函式）
    """
```

### 修改函式

#### `check_version_sync(version: str)` - 改進

現有函式保留但重構內部實作：

- **舊行為**：硬編碼偵測 `ui/pubspec.yaml`，輸出簡單不匹配警告
- **新行為**：呼叫 `load_version_release_config()` + `check_monorepo_version_sync()` + `print_version_sync_report()`
- **介面相容性**：函式簽名不變（`version: str` -> `Tuple[bool, List[str]]`），`passed` 語義調整為「版本策略符合」而非「版本完全匹配」

#### `preflight_check(version: str)` - 調整版本同步說明

- `1.5 檢查版本同步` 輸出改為三層版本對比報告
- 通過標準調整：只要版本策略符合（L2 獨立升級為正常）即通過

## `.version-release.yaml` 完整結構規格

基於 `monorepo-version-strategy.md` 第 199-281 行的建議內容，以下為最終配置檔結構：

```yaml
# .version-release.yaml - Monorepo 版本管理配置
# 來源：.claude/rules/flows/monorepo-version-strategy.md

# 版本定義
versions:
  monorepo:
    source: docs/todolist.yaml       # 唯一的權威來源
    key: current_version
    semantic_version: true
    description: "整個專案的發布版本（用於 Ticket、Wave、發布計劃）"

  ui:
    source: ui/pubspec.yaml
    key: version
    semantic_version: true
    independent: true                 # UI 可獨立於 monorepo 升級
    description: "Flutter 應用版本（用於 App Store）"
    sync_policy: "optional"           # 建議同步，非強制
    sync_recommendation: "保持主版本號一致（如都是 v0.x.0）"

  server:
    source: null                      # go.mod 無版本欄位
    semantic_version: false
    independent: false
    description: "Go Server 版本由 monorepo 版本決定"
    sync_policy: "implicit"           # 隱含同步

# 版本同步規則
sync_rules:
  on_release:
    monorepo:
      required: true                  # 必須更新
    ui:
      required: false                 # 可選
      recommendation: "建議更新 UI 版本的 build number 或更新主版本號以匹配 monorepo"
    server:
      required: false                 # 無需更新（由 go.mod 決定）

  on_development:
    allow_version_mismatch: true
    reason: "開發中 UI 和 Server 可獨立迭代"

  conflict_detection:
    ui_ahead_of_monorepo:
      severity: "warning"
      message: "UI 版本大於 monorepo，確認是否故意？"
    ui_behind_monorepo:
      severity: "info"
      message: "UI 版本低於 monorepo（正常）"

# 版本偵測策略
detection:
  version_files:
    - path: ui/pubspec.yaml
      type: yaml
      key: version
      context: "Flutter 應用版本"

    - path: server/go.mod
      type: toml
      key: null
      context: "Go module（無版本欄位）"

    - path: docs/todolist.yaml
      type: yaml
      key: current_version
      context: "權威的 monorepo 版本來源"

  auto_detect_order:
    - git_branch                      # feature/v0.1
    - monorepo_version_file           # todolist.yaml
    - ui_version_file                 # ui/pubspec.yaml
    - git_tag                         # v0.1-final

# 發布前檢查項
preflight_checks:
  version_sync:
    enabled: true
    fail_on_error: false              # monorepo 版本不匹配不阻塞
    warn_on_mismatch: true
```

### 結構欄位說明

| 欄位路徑 | 類型 | 必填 | 說明 |
|---------|------|------|------|
| `versions.monorepo.source` | string | 是 | L1 版本來源檔案路徑 |
| `versions.monorepo.key` | string | 是 | YAML 欄位名稱 |
| `versions.ui.source` | string | 是 | L2 版本來源檔案路徑 |
| `versions.ui.independent` | bool | 否 | 是否獨立於 monorepo，預設 false |
| `versions.ui.sync_policy` | string | 否 | `optional` / `required` / `implicit` |
| `versions.server.source` | null | 是 | 固定為 null（go.mod 無版本欄位） |
| `versions.server.sync_policy` | string | 否 | 固定為 `implicit` |
| `sync_rules.conflict_detection.*.severity` | string | 是 | `error` / `warning` / `info` |
| `preflight_checks.version_sync.fail_on_error` | bool | 否 | 版本不匹配是否阻塞發布，預設 false |

## 資料流設計

```
check 子命令執行
    |
    v
load_version_release_config(root)
    |
    +-- .version-release.yaml 存在 -> 載入 YAML
    +-- 不存在 -> DEFAULT_VERSION_RELEASE_CONFIG
    |
    v
get_monorepo_version(root)
    -> 讀取 docs/todolist.yaml current_version
    |
    v
check_monorepo_version_sync(version, config)
    |
    +-- 讀取 L2: ui/pubspec.yaml
    +-- 偵測 L3: server/go.mod（無版本欄位）
    +-- 依 sync_policy 判斷狀態
    +-- 依 conflict_detection 生成訊息
    |
    v
print_version_sync_report(sync_result)
    -> 輸出樹狀格式三層版本對比
    |
    v
check_version_sync() 回傳 Tuple[bool, List[str]]
    -> passed = sync_result["passed"]
    -> errors = [] (不匹配不視為 error，只有策略違反才是)
```

## 邊界條件總覽

| 邊界情況 | 預期行為 | 嚴重程度 |
|---------|---------|---------|
| `.version-release.yaml` 不存在 | 使用預設配置，靜默 fallback | 無輸出 |
| `.version-release.yaml` YAML 語法錯誤 | 警告訊息 + fallback | WARNING |
| `docs/todolist.yaml` 不存在 | ERROR，check 失敗 | ERROR（阻塞） |
| `ui/pubspec.yaml` 不存在 | INFO + 跳過 L2 檢查 | INFO |
| `server/go.mod` 無 `module` 行 | 顯示模組名稱不可解析，仍顯示 L3 說明 | INFO |
| L2 版本 > L1 版本 | WARNING 提示確認 | WARNING（不阻塞） |
| L2 版本 < L1 版本 | INFO 說明正常 | INFO |
| L2 版本 = L1 版本 | 顯示版本一致 | SUCCESS |
| 版本字串含 build number（`1.0.0+1`） | 去除 `+` 後部分進行主版本比對 | 無副作用 |
| `preflight_checks.version_sync.fail_on_error: true` | 版本不匹配時 check 失敗 | ERROR（視配置而定） |

## 行為場景（Given-When-Then）

### 場景 1：標準 check - 正常三層版本對比

```
Given: .version-release.yaml 存在且格式正確
       docs/todolist.yaml current_version: 0.1.1
       ui/pubspec.yaml version: 1.0.0+1
       server/go.mod 無版本欄位
When: 使用者執行 uv run version_release.py check
Then: 輸出 L1=0.1.1（來源：docs/todolist.yaml）
```

## 實作注意事項（給 thyme-python-developer）

1. **不破壞現有 check_version_sync() 函式簽名**：現有 `preflight_check()` 呼叫此函式，簽名必須保持相容 `Tuple[bool, List[str]]`。

2. **L2 版本的 build number 處理**：`pubspec.yaml` 版本格式為 `1.0.0+1`，比對時需特殊處理（例如只比對 `+` 前的部分），但 CLI 輸出應顯示完整版本包含 build number。

3. **get_monorepo_version() 與現有 detect_version() 的關係**：現有 `detect_version()` 從 git branch / pubspec.yaml / git tag 偵測版本。新增的 `get_monorepo_version()` 專門讀取 `docs/todolist.yaml` 的 `current_version` 欄位作為 L1 版本。兩者功能不同，不需合併。

4. **YAML 讀取用 yaml.safe_load()**：已有 `import yaml` 相依，直接使用 `yaml.safe_load()`，不需新增相依套件。

5. **常數放在檔案頂部 constants 區段**：依 Python 品質規則，所有常數集中在檔案頂部，禁止硬編碼字串和數值散落在函式中。

6. **except 區塊必須輸出到 stderr 或 logger**：依 quality-baseline.md 規則 4，`load_version_release_config()` 的 YAML 解析例外必須輸出警告，不可靜默吞掉。
